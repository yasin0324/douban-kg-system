# 基于知识图谱的电影推荐系统中期汇报演讲稿

这版讲稿按当前 PPT 的页序组织，适合中期答辩时直接口述，正常语速大约 `6-8` 分钟。整体原则是：少背数据，多讲清楚“我为什么这样设计、现在做到哪一步、下一步准备怎么推进”。

## 1. 封面

各位老师好，我的毕设题目是《基于知识图谱的电影推荐系统设计与实现》，我是郭远信，指导教师是李芳芳老师。下面我从研究目标、系统设计、核心算法、实验评估以及当前进展这几个方面进行中期汇报。

## 2. 研究目标与技术路线

这个课题的目标，不是单纯做一个图数据库展示页面，也不是只做单一推荐算法实验，而是围绕电影推荐场景，构建一个从数据采集、知识图谱建模、推荐算法实现，到前后端系统展示和离线评估都完整打通的系统。

从技术路线来看，我的工作主要分成五部分。第一是基于豆瓣数据构建电影领域知识图谱；第二是实现多种推荐算法，并进行效果对比；第三是在此基础上设计 CFKG 融合推荐框架；第四是让推荐结果具备一定可解释性；第五是把这些能力集成到一个完整的 Web 系统里。

从时间安排来看，一月份主要完成数据采集，二月份完成知识图谱构建，三月上中旬完成系统开发和推荐算法实现，目前，也就是中期这个阶段，重点已经转到离线评估和参数调优上。

## 3. 系统整体架构

整个系统我划分为三层。

最上层是表示层，采用 `Vue 3` 和 `ECharts`，主要负责前端页面展示，包括电影检索、详情查看、个性化推荐、图谱探索和统计可视化等功能。

中间是业务层，采用 `FastAPI + Python 3.11`。这一层负责对外提供 REST API，同时封装推荐算法、推荐解释、图谱查询和用户行为处理等业务逻辑。

最底层是数据层，由 `MySQL` 和 `Neo4j` 共同组成。这里我没有只使用一种数据库，而是做了分工。`MySQL` 更适合存储原始电影数据、用户评分和业务表；`Neo4j` 更适合存储图结构、做关系查询和路径推理。这样设计的好处是，既保留了关系型数据库对业务数据的管理能力，也发挥了图数据库在关联查询上的优势。

## 4. 数据采集与处理

数据源主要来自豆瓣电影页面。采集端我使用的是 `Playwright` 浏览器自动化，而不是普通的请求式爬虫。原因是目标站点存在一定的动态加载和反爬机制，浏览器自动化方式更接近真实用户访问，稳定性更高。

在爬虫实现上，我设计了基于 MySQL 锁机制的任务队列，支持多 worker 并行采集，同时配合代理 IP 池轮换，来降低被限制的风险。当前已经采集了大约 `14` 万部电影和 `19` 万名影人，形成了后续图谱构建和推荐实验的数据基础。

这些原始数据先进入 `MySQL`，之后经过清洗和标准化，再通过 ETL 导入 `Neo4j`。也就是说，我做的不是“抓完直接拿来算”，而是先把类型、地区、语言、导演、演员这些字段统一处理，尽量减少脏数据对后续推荐和图谱构建的影响。

## 5. 知识图谱 Schema 设计

在图谱建模方面，我当前设计了 `7` 类节点和 `7` 类关系。

节点层面，核心实体包括 `Movie`、`Person` 和 `Genre`，另外还扩展了 `Region`、`Language`、`ContentType` 和 `YearBucket` 四类属性节点。关系层面，除了 `DIRECTED`、`ACTED_IN`、`HAS_GENRE` 这些核心关系之外，还补充了地区、语言、内容类型和年代等结构关系。

这样设计有两个目的。第一，是支撑图谱查询和前端可视化展示；第二，是为后续图谱推荐算法提供更丰富的结构语义。特别是 `KG-Embed` 这类方法，不仅依赖电影和影人的直接关系，也依赖这些扩展属性来增强表示能力。

从规模上看，当前图谱已经达到十万级电影节点、近二十万影人节点，以及百万级以上关系边，已经能够支撑本课题的推荐实验和图谱展示需求。

## 6. 推荐算法总览

推荐模块一共实现了五种算法，分别是 `Content-Based`、`Item-CF`、`KG-Path`、`KG-Embed` 和 `CFKG`。

其中，`Content-Based` 和 `Item-CF` 主要作为基线方法，用来对照图谱方法是否真的带来了增益；`KG-Path` 强调显式关系推理和可解释性；`KG-Embed` 强调通过图谱嵌入学习潜在语义；`CFKG` 则是我设计的融合主链路，希望把协同过滤的准确性和图谱方法的语义信息、解释能力结合起来。

另外，这五种算法不是各写各的，而是统一继承推荐基类，输入端共享用户行为信号的处理逻辑。也就是说，用户的评分、喜欢和想看行为，会先被统一整理成不同强度的偏好信号，再交给不同算法使用。这种统一接口的好处，是便于比较算法效果，也便于后续做融合和评估。

## 7. 核心算法一：KG-Embed

第一个重点算法是 `KG-Embed`。它的核心思想是，把知识图谱中的实体和关系映射到低维向量空间中，用 `TransE` 的方式学习“头实体加关系大致接近尾实体”的表示。

在实现过程中，我没有只使用最基础的电影、演员、导演、类型三类关系，而是在图谱中进一步扩展了地区、语言、内容类型和年代桶等关系，让图谱表达更完整。然后在排序阶段，我不是只看单一相似度，而是融合了三部分信息：用户偏好电影的嵌入质心相似度、候选电影与历史种子的实体重叠情况，以及与最强种子电影的最大相似度。

从消融实验可以看到，扩展关系和融合排序都带来了比较明显的提升。这说明知识图谱本身的结构信息是有价值的，不只是“把电影和人连起来”这么简单，而是确实能帮助模型学习更丰富的语义关联。

## 8. 核心算法二：KG-Path

第二个重点算法是 `KG-Path`。和嵌入方法相比，它更强调显式路径推理，也就是一条推荐结果为什么出现，可以沿着图谱路径追溯出来。

这里我主要设计了两类路径。第一类是 `1-hop` 路径，比如“用户喜欢的电影”和候选电影共享导演、演员或者类型；第二类是 `2-hop` 路径，比如通过中间电影和桥接演员，把两个电影关联起来。

为了减少高频实体带来的噪声，我在路径打分时加入了类似 `IDF` 的度惩罚，对非常常见的演员或类型进行抑制，同时只保留前几位主演作为主要证据。这样做的原因是，如果完全不加约束，很多候选会因为“共享一个特别高频的演员”而被错误放大。

虽然 `KG-Path` 在准确率上不是最强的，但它的优势非常明确，就是解释能力最好。对于推荐系统来说，这一点很重要，因为它能把“为什么推荐这部电影”具体落到共享导演、演员或者图谱路径上，而不是只给一个黑盒分数。

## 9. CFKG 融合推荐主链路

在前面几种算法的基础上，我设计了 `CFKG` 作为系统当前的主推荐框架。

它的整体思路不是让某一个算法单独决定最终结果，而是采用多路并行召回，再做加权融合。具体来说，先分别从 `Item-CF`、`KG-Embed` 和 `KG-Path` 拿到各自的候选集合，然后进行去重、归一化和加权合并；如果用户行为比较少，还会用基于内容的方法做兜底，处理冷启动问题。

这里的核心考虑是，`Item-CF` 在数据比较充分时准确率高，`KG-Embed` 能补充潜在语义关联，`KG-Path` 则可以补充显式解释证据。所以 `CFKG` 的目标，不是简单追求某一个单项指标最高，而是在准确性、多样性、解释性和系统可用性之间寻找一个更平衡的方案。

目前来看，这条主链路已经可以稳定工作，但它在准确率和响应时间上还有进一步优化空间，所以这也是我下一阶段的重点。

## 10. 离线评估方法

为了对不同算法做公平比较，我设计了统一的离线评估协议。

评估数据来自公开豆瓣用户，其中验证集 `42` 人，测试集 `168` 人。评估方式采用的是 `sampled leave-one-out`，也就是对每个用户保留 `1` 个正样本，再配合 `99` 个负样本做排序测试。同时，我使用了 `5` 个随机种子重复采样，最后报告均值和标准差，尽量降低偶然性。

评价指标方面，我主要看 `HR@K` 和 `NDCG@K`，分别反映命中率和排序质量；另外也补充了 `Coverage`、`Diversity` 和平均耗时，用来衡量推荐列表的覆盖能力、多样性以及系统响应代价。

我之所以没有只看准确率，是因为这个课题最终落点是一个可运行的推荐系统，而不是单纯做一个算法分数表，所以除了准不准，也要考虑解释性、多样性和实际部署时的效率。

## 11. 评估结果与分析

从当前这页展示的结果来看，`Item-CF` 在准确率指标上整体最好，说明在当前数据规模下，协同过滤依然具有很强的优势。

在图谱方法里，`KG-Embed` 的表现最好，说明通过图谱嵌入学习结构语义是有效的。相比之下，`KG-Path` 的优势不在于绝对分数，而在于每条推荐都能给出较清晰的关系解释，这一点对系统展示和用户理解都很重要。

`CFKG` 的定位需要特别说明一下。它不是当前单独指标最好的算法，但它是我系统中的融合主链路。原因在于，它把协同过滤、图谱嵌入和路径推理整合到了同一个框架里，在准确性和多样性之间取得了相对均衡的结果，同时还能保留一定解释能力。下一阶段，我会继续围绕这条主链路做参数调优、候选融合策略优化以及响应时间压缩。

## 12. 当前进展与后续计划

到中期这个阶段，我已经完成了数据采集、知识图谱构建、前后端系统开发、五种推荐算法实现，以及离线评估框架搭建，系统主链路已经基本打通。

接下来的工作主要有三个方向。第一，继续优化推荐算法，重点是 `CFKG` 的融合策略和运行效率；第二，补充实验分析和论文正文撰写，把方法设计、实验过程和结果分析整理成更完整的论文内容；第三，继续完善系统展示效果，为后续答辩演示做准备。

总体来说，我的中期判断是：这个课题已经从“能不能做出来”的阶段，进入到了“如何把效果和表达做得更扎实”的阶段。后续工作的重点不再是单纯堆功能，而是优化主链路、补足实验分析，并完成论文收尾。

## 口径提醒

- 不要说“`CFKG` 当前效果最好”，更稳妥的说法是：`Item-CF` 当前整体最优，`KG-Embed` 是当前效果最好的图谱方法，`CFKG` 是系统主链路和后续优化重点。
- 强调 `MySQL` 和 `Neo4j` 的分工，不要只说“用了图数据库”。
- 冷启动不要说“还没处理”，更准确的说法是：系统已经有热门内容回退和行为补充引导机制。
- 如果老师追问 AI 参与情况，可以说：AI 提升了实现效率，但需求拆解、模块整合、实验验证和结果取舍是自己主导完成的。
