# 实验记录本文字稿

## 1. 2026-02-01 Schema 设计与课题方案细化

- 实验主题：知识图谱 Schema 设计与课题技术路线细化
- 目的：明确电影知识图谱的实体、关系和属性结构，为后续 ETL、查询接口和推荐实验奠定基础。
- 过程：结合豆瓣电影数据字段，设计 `Movie`、`Person`、`Genre`、`Region`、`Language`、`ContentType`、`YearBucket` 等节点，确定 `DIRECTED`、`ACTED_IN`、`HAS_GENRE` 等关系，并补充唯一约束和索引策略。
- 结果：完成知识图谱 Schema 设计文档，明确主键、属性来源和预估图谱规模，为后续导入 Neo4j 提供统一结构。
- 问题：电影元数据字段存在缺失和命名不统一问题，若直接入图库会影响关系完整性。
- 后续安排：继续推进爬虫与清洗脚本，实现从原始数据到结构化图谱数据的稳定转换。

## 2. 2026-02-03 电影爬虫任务锁与并发机制

- 实验主题：电影爬虫并发抓取与任务锁管理
- 目的：提高电影采集效率，并避免多任务并发时重复抓取同一条记录。
- 过程：为电影爬虫增加即时任务锁定机制，支持分布式抓取和数据库状态管理，同时调整批次拉取与任务队列逻辑。
- 结果：采集流程具备并发执行能力，重复抓取风险降低，为后续大规模数据采集提供基础。
- 问题：并发执行后，若代理失效或任务异常退出，可能造成锁未及时释放。
- 后续安排：继续补充代理切换、失败重试和陈旧锁清理机制，提高抓取稳定性。

## 3. 2026-02-04 代理池切换与稳定性优化

- 实验主题：代理池与任务容错机制优化
- 目的：应对豆瓣反爬策略，提高长时间运行时的抓取稳定性。
- 过程：新增多个代理 API 地址，支持代理失败后自动切换；增加重试机制、超时控制、清理陈旧任务锁和失败任务立即释放锁的逻辑。
- 结果：爬虫在异常网络环境下的容错能力增强，任务堆积和锁死问题明显缓解。
- 问题：代理质量波动仍会影响抓取效率，不同时间段的响应稳定性存在差异。
- 后续安排：继续观察代理成功率，并结合监控信息优化批次大小和请求节奏。

## 4. 2026-02-06 影人爬虫与监控脚本

- 实验主题：影人信息采集与监控能力补齐
- 目的：补充导演、演员等影人信息，为图谱构建和推荐解释提供更完整的数据支撑。
- 过程：新增豆瓣影人爬虫，支持异步 Playwright 和代理池；同时增加监控脚本，实时查看影人抓取进度、统计信息和失败情况。
- 结果：电影数据采集从单一影片信息扩展到影片与影人双主体，图谱关系数据来源更加完整。
- 问题：豆瓣影人页面结构与电影页面不同，字段解析规则需要分别维护。
- 后续安排：继续完善人物数据解析逻辑，并配合后续清洗脚本校验字段完整性。

## 5. 2026-02-23 数据质量分析与 ETL 准备

- 实验主题：数据质量分析与 MySQL 到 Neo4j 的 ETL 准备
- 目的：评估采集结果质量，识别关键缺失字段，并为图谱导入制定清洗策略。
- 过程：统计 Movies 与 Persons 表数据规模、字段缺失率和预估关系量，生成数据质量报告；同步补充 ETL 与分析脚本文档。
- 结果：确认 Movies 表 135,990 条、Persons 表 198,067 条，预估图谱关系总数 1,443,933 条；明确评分、剧情简介、片长等字段存在较高缺失。
- 问题：部分字段缺失率较高，若不处理会影响推荐效果和图谱展示完整性。
- 后续安排：根据质量报告继续补采关键字段，并完成 ETL 脚本以导入 Neo4j。

## 6. 2026-02-24 后端架构与 API 基线设计

- 实验主题：FastAPI 后端架构规划与接口基线确定
- 目的：建立统一的后端服务结构，支撑电影、影人、图谱、统计和推荐模块开发。
- 过程：梳理后端目录结构、服务层职责和路由划分，明确 MySQL 与 Neo4j 双数据源的分工，形成后端开发规划文档。
- 结果：后端开发主线明确，电影查询、影人查询、图谱探索、认证与推荐接口具备统一的设计框架。
- 问题：双库协同会增加接口实现复杂度，需要后续在服务层做好封装和容错。
- 后续安排：按模块逐步实现 FastAPI 接口，并同步准备测试与联调。

- 问题：前后端联调阶段需要不断处理字段命名、空值显示和接口异常情况。
- 后续安排：继续完成图谱可视化、路径查询和用户认证相关页面。

## 8. 2026-03-01 图谱可视化与排序优化

- 实验主题：知识图谱可视化与电影排序逻辑完善

## 12. 2026-03-11 至 2026-03-12 离线评估、参数优化与中期材料整理

- 实验主题：推荐算法离线评估与优化收口
- 目的：验证多种推荐算法的效果，形成中期检查可使用的实验依据。
- 过程：修复评估数据泄露问题，统一采样 leave-one-out 评估流程，对 `content`、`item_cf`、`kg_path`、`kg_embed` 四类算法进行多 seed 验证与参数优化，并将结果归档。
- 结果：形成 2026-03-12 评估报告。在 `Recall@10` 指标上，`item_cf = 0.8335 ± 0.0025`，`kg_embed = 0.5938 ± 0.0145`，说明推荐模块已具备较明确的实验对比基础。
- 问题：知识图谱路径推荐和嵌入推荐仍有进一步优化空间，解释质量也需要结合论文写作继续整理。
- 后续安排：继续完成中期检查表、实验记录、论文正文实验分析和答辩材料收口，不虚构新的技术实验结果。
