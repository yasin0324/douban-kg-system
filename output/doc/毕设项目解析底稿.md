# 基于豆瓣电影知识图谱的电影推荐系统项目解析

## 1. 研究目标

本项目围绕“豆瓣电影知识图谱的构建与应用”展开，核心目标是在真实电影数据基础上完成数据采集、清洗、知识图谱建模、系统实现与推荐验证，形成一个可用于检索、浏览、图谱探索和个性化推荐的电影信息系统。项目不仅关注知识图谱本体构建，还强调图谱在推荐场景中的应用效果和可解释性，服务于毕业论文中的系统设计、实验分析和答辩展示。

## 2. 系统架构

项目整体采用“采集层 + 数据处理层 + 存储层 + 服务层 + 展示层”的分层方案：

- 数据采集层：使用 Playwright 编写电影和影人爬虫，支持并发抓取、重试、锁管理、代理切换和进度监控。
- 数据处理层：使用 Python 和 Pandas 完成数据清洗、缺失值统计、格式统一和质量分析，并为 Neo4j 导入准备结构化数据。
- 存储层：MySQL 用于保存原始电影、影人和用户行为数据，Neo4j 用于保存知识图谱实体与关系。
- 服务层：基于 FastAPI 提供电影、影人、图谱、统计、认证、用户行为和推荐接口。
- 展示层：基于 Vue3、Vite、Element Plus 和 ECharts 实现前端页面，支持检索、详情浏览、图谱可视化、统计展示和推荐结果展示。

## 3. 数据流程

项目的数据主链路如下：

1. 从豆瓣电影页面采集电影基础信息、导演、演员、评分、类型等数据，并同步抓取影人信息。
2. 将采集结果写入 MySQL，执行清洗、去重和缺失值分析，生成数据质量报告。
3. 根据 Schema 设计，将电影、影人、类型及扩展属性转换为图结构数据，再通过 ETL 导入 Neo4j。
4. FastAPI 同时访问 MySQL 和 Neo4j，对外提供搜索、详情、图谱、统计和推荐接口。
5. Vue3 前端调用接口完成系统展示；推荐算法通过离线评估模块输出指标报告，用于验证推荐效果。

当前仓库中的关键数据规模如下：

- Movies 表记录数：135,990
- Persons 表记录数：198,067
- 预估图谱关系总数：1,443,933

## 4. 已完成成果

### 4.1 数据与图谱构建

- 已完成知识图谱 Schema 设计，定义了 `Movie`、`Person`、`Genre` 以及区域、语言、内容类型、年份分桶等扩展节点。
- 已完成 MySQL 到 Neo4j 的 ETL 脚本，能够将电影、导演、演员和类型关系导入图数据库。
- 已完成数据质量分析，明确了评分、剧情简介、片长等字段的缺失情况，为后续补采和数据治理提供依据。

### 4.2 系统实现

- 后端已实现电影、影人、图谱、统计、认证、管理员和用户行为相关接口。
- 前端已实现首页、搜索页、电影详情页、影人详情页、推荐页、图谱页、路径页、统计页、登录注册和个人中心等页面。
- 图谱展示已接入 ECharts，可支持以电影或影人为中心的关系可视化以及路径探索。

### 4.3 推荐算法与实验

当前仓库中已实现 4 类推荐算法：

- `content`：基于内容的推荐
- `item_cf`：基于物品的协同过滤
- `kg_path`：基于知识图谱路径的推荐
- `kg_embed`：基于知识图谱嵌入的推荐

根据 2026-03-12 的离线评估报告，在 `Recall@10` 指标上：

- `item_cf = 0.8335 ± 0.0025`
- `kg_embed = 0.5938 ± 0.0145`

这说明系统已经具备可验证的推荐实验能力，并形成了可写入论文中期材料的实验依据。

## 5. 当前问题与后续任务

当前项目已完成核心系统与主要实验，但仍存在以下问题和后续工作：

- 豆瓣页面结构变动和反爬机制会影响采集稳定性，需要继续优化代理、重试和补采流程。
- 部分原始字段存在缺失或不一致，仍需通过数据清洗和补采提升图谱质量。
- MySQL 与 Neo4j 双库协同增加了接口设计和推荐链路的复杂度，需要继续优化缓存与候选生成效率。
- 推荐实验结果已经具备展示价值，但解释质量、参数调优和实验分析仍可进一步完善。
- 论文正文、实验结果讨论、系统截图整理和答辩材料仍需继续补充和收口。
