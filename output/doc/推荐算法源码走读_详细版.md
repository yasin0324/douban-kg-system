# 推荐算法源码走读（详细版）

这份文档只讲推荐模块，不展开到爬虫、ETL、前端和通用接口。目标不是让你背结论，而是让你真正理解：

- 算法入口在哪里
- 读的是哪类数据
- 中间缓存怎么组织
- 分数是怎么算出来的
- 推荐理由是怎么拼出来的
- 各算法之间为什么会出现效果差异

整套走读顺序固定为：

`Base / GraphCache / Recommend Router -> content -> item_cf -> kg_path -> kg_embed -> cfkg -> evaluator`

---

## 0. 先看公共底座

如果不先理解公共底座，后面五个算法你会反复卡在同样的问题上：

- 用户正反馈到底怎么定义
- 已交互电影怎么排除
- 图谱算法为什么都在用同一个缓存
- 推荐接口是怎么把算法结果包装成前端响应的

### 0.1 `BaseRecommender` 在做什么

文件：

`db-backend/app/algorithms/base.py`

这个类不是“能直接推荐”的算法，而是所有推荐算法共享的行为层。

它主要统一了三件事：

1. 正反馈电影怎么定义  
2. 图谱算法的 seed 怎么截断  
3. 已交互电影怎么排除  

#### 0.1.1 统一接口

所有推荐算法都要实现：

```python
def recommend(
    self,
    user_id: int,
    n: int = 20,
    exclude_mids: set | None = None,
    exclude_from_training: set | None = None,
) -> list[dict]
```

这四个参数里最关键的是后两个：

- `exclude_mids`
  - 从最终候选结果里排除
  - 常见用途：前端“换一批”时，不要把已经展示过的电影再返回
- `exclude_from_training`
  - 只从“训练画像”里排除，不一定从候选里排除
  - 主要用于离线评测，防止测试正样本泄露到训练侧

这两个参数是很多人第一次看推荐代码最容易混淆的点。

#### 0.1.2 正反馈电影怎么构造

核心函数：

```python
get_user_positive_movies()
```

逻辑是：

1. 先查 `user_movie_ratings`
2. 只保留 `rating >= threshold` 的电影，默认阈值是 `3.5`
3. 再查 `user_movie_prefs`
4. 如果某部电影没有达到正向评分，但用户标记了偏好，则补合成评分

映射规则写死在类变量里：

```python
PREF_SYNTHETIC_RATING = {
    "like": 4.5,
    "want_to_watch": 4.0,
}
```

所以这套系统里：

- `like` 会被当成一个强正反馈
- `want_to_watch` 会被当成一个弱一点的正反馈

这一步很重要，因为它意味着这个系统的推荐输入不仅来自评分，还来自显式偏好行为。

#### 0.1.3 为什么要把 `like` 映射成 `4.5`

因为后面的多数算法都在用“评分权重”做计算。

如果偏好行为不映射成数值，那么：

- `content` 无法做加权平均画像
- `kg_path` 无法按 seed 强弱赋权
- `kg_embed` 无法给 seed 向量不同权重

所以作者采用了一个很工程化的方案：

- `rating >= 3.5` 直接用真实分
- `like` 看成 `4.5`
- `want_to_watch` 看成 `4.0`

这样就能把不同类型的用户行为统一成一个“可加权的正反馈集合”。

#### 0.1.4 图谱算法为什么还有一个 `get_user_positive_movies_for_kg()`

因为图谱算法计算开销明显更大。

如果一个重度用户有几百个高分电影，拿全部种子去做路径扩展或嵌入融合，代价会非常高，所以图谱算法会用这个函数来截断 seed：

```python
max_positive_ratings
max_likes
max_wishes
```

例如：

- `kg_path` 默认：
  - 高分种子最多 `30`
  - like 最多 `10`
  - wish 最多 `0`
- `item_cf` 默认：
  - 高分种子最多 `50`
  - like 最多 `10`
  - wish 最多 `0`

也就是说，不同算法对 seed 数量的容忍度不同。

#### 0.1.5 `get_user_all_rated_mids()` 的意义

这个函数名字有点误导，它实际上返回的是：

`评分电影 + 偏好电影`

不是只有评分。

它的作用是构造“已交互电影集合”，后面多数算法都会用它来避免推荐用户已经看过、喜欢过或想看过的电影。

### 0.2 `GraphMetadataCache` 在做什么

文件：

`db-backend/app/algorithms/graph_cache.py`

这是图谱推荐的核心共享缓存，`kg_path`、`kg_embed`、推荐解释接口都依赖它。

它解决的问题是：

- 不要每次推荐都去 MySQL / Neo4j 临时查数据
- 不要每个算法都自己重复构造图谱结构
- 统一维护电影的关系实体集合、反向索引和实体度数

#### 0.2.1 `MovieGraphProfile` 是图谱侧的“电影画像”

每部电影会被组织成一个 `MovieGraphProfile`：

```python
MovieGraphProfile(
    mid,
    name,
    year,
    content_type,
    genres,
    regions,
    languages,
    year_bucket,
    directors,
    actors,
    top_actors,
    actor_orders,
)
```

你可以把它理解成：

“推荐算法并不直接拿 Neo4j 节点对象做计算，而是先把一部电影所关联的结构信息抽成一个内存里的 profile”

这个 profile 能回答几个关键问题：

- 这部电影有哪些导演
- 有哪些演员
- top5 演员是谁
- 属于哪些类型
- 来自哪些地区
- 是什么语言
- 年代桶是什么

#### 0.2.2 缓存是怎么加载的

入口函数：

```python
GraphMetadataCache.ensure_loaded()
```

它内部会做三步：

1. `_load_movies_from_mysql()`
2. `_load_people_and_genres_from_neo4j()`
3. `_build_relation_indexes()`

#### 0.2.3 为什么电影元数据先从 MySQL 读

`_load_movies_from_mysql()` 会先查：

```sql
SELECT douban_id, name, genres, regions, languages, year, type
FROM movies
WHERE douban_id IS NOT NULL
```

也就是说：

- 电影基础字段来自 MySQL
- 这一步先把 `MovieGraphProfile` 框架搭起来
- 初始时每部电影已经有：
  - genres
  - regions
  - languages
  - year
  - content_type

#### 0.2.4 导演和演员为什么再从 Neo4j 补

`_load_people_and_genres_from_neo4j()` 分别跑了三类 Cypher：

- `(Person)-[:DIRECTED]->(Movie)`
- `(Person)-[:ACTED_IN]->(Movie)`
- `(Movie)-[:HAS_GENRE]->(Genre)`

这样做的好处是：

- 导演 / 演员关系直接取图数据库里的最终结构
- 不必重新解析 MySQL 里的 `actor_ids` / `director_ids`
- 演员顺序 `rel.order` 也能一起保留

尤其是演员排序很关键，因为：

- `top_actors` 会取 `ord <= 5`
- `kg_path` 会用 `actor_order_limit` 限制演员证据

#### 0.2.5 inverted index 是什么

`_build_relation_indexes()` 会为每种关系建一个倒排索引：

```python
relation -> entity_id -> set[mid]
```

例如：

- `director -> 诺兰 -> {盗梦空间, 星际穿越, 信条}`
- `genre -> 科幻 -> {...很多电影...}`

这个结构特别适合 `kg_path`：

因为要找“和 seed 电影共享某个导演/演员/类型”的候选电影时，倒排索引可以直接查，不需要实时跑 Cypher。

#### 0.2.6 entity degree 是什么

同时还会统计：

```python
relation -> entity_id -> degree
```

意思是：

- 某个导演连接了多少电影
- 某个演员连接了多少电影
- 某个类型覆盖了多少电影

这个值会被 `safe_idf()` 用来做惩罚：

```python
safe_idf(degree) = 1 / log(2 + degree)
```

实体越热门，权重越低。

所以：

- “科幻” 这种大类型贡献会被压低
- 冷门导演、冷门演员更容易产生更强区分度

#### 0.2.7 triples 是给谁用的

`build_triples()` 是专门服务 `kg_embed` 的。

它会把图谱导出成三元组：

```python
(head, relation, tail)
```

同时会输出：

- `movie_mids`
- `entity_types`
- `relation_types`

并支持两种模式：

- `core`
  - 只用 `director / actor / genre`
- `expanded`
  - 额外加入 `region / language / content_type / year_bucket`

这个开关是 `kg_embed` 结果差异的核心之一。

### 0.3 推荐路由层在做什么

文件：

`db-backend/app/routers/recommend.py`

这一层不是算法本身，而是把算法运行时包起来。

它做了 5 件事：

1. 算法实例缓存
2. 并发槽位控制
3. 推荐超时保护
4. 推荐结果电影详情补全
5. 推荐解释图构建

#### 0.3.1 为什么算法实例要缓存

函数：

```python
_get_algorithm_instance(algo_name)
```

每种算法实例都会放进 `_algorithm_instances`。

原因很简单：

- `content` 会缓存整套特征矩阵
- `item_cf` 会缓存评分矩阵
- `kg_path` 会缓存 evidence
- `kg_embed` 会缓存 embedding 工件

如果每次请求都新建实例，缓存价值就没了。

#### 0.3.2 为什么要有并发槽位

每个算法还会分配一个 `BoundedSemaphore`。

作用是：

- 控制某个算法同时只能跑有限个任务
- 防止重算法被打爆

所以 `/personal` 里如果 `slot.acquire(blocking=False)` 失败，就直接返回 `503`。

#### 0.3.3 `/personal` 的完整流程

入口：

```python
get_personal_recommendations()
```

流程是：

1. 校验算法名
2. 取算法实例
3. 取并发槽位
4. 在线程池里跑 `algo.recommend()`
5. 用 `asyncio.wait_for()` 做超时保护
6. 算法返回后，用 `_enrich_movie_details()` 批量查电影详情
7. 返回前端所需的结构：

```python
{
  "algorithm": ...,
  "algorithm_display_name": ...,
  "cold_start": ...,
  "items": [...],
  "total": ...
}
```

所以要记住：

“算法层只负责吐 mid、score、reason；路由层再把电影标题、封面、评分、类型补全给前端。”

#### 0.3.4 `/explain` 的完整流程

入口：

```python
explain_recommendation()
```

流程是：

1. 先去 MySQL 查目标电影基本信息
2. 再调用 `_load_positive_movies_for_user()` 取当前用户正反馈电影
3. 再进入 `_build_recommendation_explain_payload()`

解释分两种：

- 有共享证据：
  - `_build_overlap_explanation()`
  - 找共享导演、共享演员、共享类型
- 没有共享证据：
  - `_build_target_context_explanation()`
  - 退化成展示目标电影自身的导演、演员、类型

这意味着：

“解释接口不是把算法内部所有计算过程完全还原，而是把推荐结果翻译成用户能理解的结构化证据图。”

## 1. `content` 源码走读

文件：

`db-backend/app/algorithms/content_based.py`

这是最容易理解的算法，因为它不依赖用户间关系，也不依赖图谱路径，只看电影自己的内容特征。

### 1.1 读哪些数据

在 `_load_data()` 里，一次性查整张 `movies` 表：

```sql
SELECT douban_id, name, genres, regions, year, douban_score
FROM movies
WHERE douban_id IS NOT NULL
```

也就是说，这个算法只看：

- `genres`
- `regions`
- `year`

不看：

- 导演
- 演员
- 用户相似用户
- 图数据库关系

所以它是最纯的“内容基线”。

### 1.2 缓存怎么建

`_load_data()` 会构造这些缓存：

- `_movie_data`
  - `mid -> {name, genres, regions, year, score}`
- `_genre_list`
- `_region_list`
- `_genre_idx`
- `_region_idx`
- `_movie_mids`
- `_feature_matrix`
- `_mid_to_matrix_idx`

其中最关键的是：

```python
_feature_matrix: 所有电影的特征矩阵
```

这意味着推荐时不会每次重建向量，而是提前把所有电影的向量全部算好。

### 1.3 特征向量怎么构造

函数：

```python
_make_feature_vector(mid)
```

向量结构是：

```text
[genre one-hot | region one-hot | year bucket(5维)]
```

如果全库有：

- 32 种 genre
- 20 种 region

那总维度就是：

`32 + 20 + 5 = 57`

其中最后 5 维表示年份桶：

- `< 1990`
- `1990s`
- `2000s`
- `2010s`
- `2020+`

### 1.4 用户画像怎么计算

`recommend()` 里先取用户正反馈电影：

```python
positive_movies = self.get_user_positive_movies(...)
```

然后拿这些正反馈电影的向量做加权平均。

权重直接来自评分：

```python
positive_weights = np.array([float(m["rating"]) for m in positive_movies])
valid_weights = valid_weights / total_weight
user_vec = np.average(pos_vectors, axis=0, weights=valid_weights)
```

也就是说：

- 用户更高分的电影
- 对最终兴趣向量贡献更大

### 1.5 为什么点积就等于余弦相似度

因为它在 `_load_data()` 里已经把所有电影向量做了归一化：

```python
self._feature_matrix = self._feature_matrix / norms
```

之后又把用户画像向量也做了归一化：

```python
user_vec = user_vec / u_norm
```

因此：

```python
scores = np.dot(self._feature_matrix, user_vec)
```

本质上就是：

```text
cos(movie_vec, user_vec)
```

数学上因为：

```text
cos(a, b) = (a · b) / (|a| |b|)
```

两边都归一化后，分母变成 1，于是余弦相似度就等于点积。

### 1.6 候选怎么排除

这一步要特别注意：

```python
rated_mids = self.get_user_all_rated_mids(...)
exclude_all = rated_mids | exclude_mids
```

含义是：

- 用户已经评分/偏好的电影不推荐
- 前端额外要求排除的电影也不推荐

但是：

- `exclude_from_training` 只用于构造画像时排除
- `exclude_mids` 才是最终结果排除

### 1.7 推荐理由怎么生成

它先看用户画像向量中权重大于 `0.1` 的 genre：

```python
if user_vec[i] > 0.1:
    user_top_genres.append(g)
```

然后取前三个类型作为用户 top genres。

对候选电影：

- 如果候选电影和用户 top genres 有交集
  - 生成：
    - `因为你偏好 xxx 类型的电影`
- 否则
  - 生成：
    - `基于你的观影偏好特征匹配`

### 1.8 一句话总结它怎么算分

```text
候选分数 = 候选电影内容向量 与 用户加权兴趣向量 的余弦相似度
```

### 1.9 手算小例子

假设用户喜欢两部电影：

- A：剧情 / 犯罪，评分 5.0
- B：剧情 / 悬疑，评分 4.0

则用户画像会更偏向：

- 剧情
- 犯罪
- 悬疑

且 A 的权重比 B 更高。

如果候选电影 C 是：

- 剧情 / 犯罪

那 C 和用户画像的点积就会更大。

如果候选电影 D 是：

- 动画 / 家庭

那 D 的相似度就很低。

### 1.10 优势与局限

优势：

- 简单稳定
- 速度快
- 不依赖用户间共现

局限：

- 只看浅层元数据
- 不利用导演、演员等关系
- 很难表达深层语义

## 2. `item_cf` 源码走读

文件：

`db-backend/app/algorithms/item_cf.py`

这个算法的核心思想是：

“如果大量用户同时喜欢 A 和 B，那么喜欢 A 的用户也可能喜欢 B。”

### 2.1 读哪些数据

它只读两类 MySQL 数据：

1. 评分表 `user_movie_ratings`
2. 电影名 `movies`

核心 SQL：

```sql
SELECT user_id, mid, rating FROM user_movie_ratings
SELECT douban_id, name FROM movies
```

注意：

- 它不依赖 Neo4j
- 它也不依赖图缓存
- 它完全建立在用户评分矩阵上

### 2.2 缓存怎么建

`_load_data()` 会构造：

- `_item_users`
  - `mid -> {user_id: rating}`
- `_user_items`
  - `user_id -> [mid1, mid2, ...]`
- `_movie_names`
- `_item_norms`

其中最关键的是前两个：

#### `_item_users`

它是 item 为中心的评分矩阵。

例如：

```python
{
  "1292052": {1: 5.0, 3: 4.0, 8: 4.5},
  "1291546": {1: 4.0, 2: 5.0, 8: 4.0},
}
```

#### `_user_items`

它是 user 为中心的倒排结构。

例如：

```python
{
  1: ["1292052", "1291546"],
  8: ["1292052", "1291546", "1292720"],
}
```

### 2.3 `item_norms` 为什么要预计算

物品间余弦相似度公式是：

```text
sim(i, j) = dot(i, j) / (||i|| * ||j||)
```

其中：

- `i` 是电影 i 的用户评分向量
- `j` 是电影 j 的用户评分向量

范数部分：

```python
sqrt(sum(v ** 2 for v in users.values()))
```

如果每次比较物品都现算范数，会重复很多次，所以 `_load_data()` 提前把每个物品的范数存进 `_item_norms`。

### 2.4 候选集怎么生成

在 `recommend()` 里，先拿到用户的正反馈电影：

```python
positive_mids = [seed1, seed2, ...]
```

然后通过这些 seed 的评分用户，把候选电影扩出来：

```python
for seed_mid in positive_mids:
    for u in self._item_users.get(seed_mid, {}):
        candidate_mids.update(self._user_items.get(u, []))
```

这段逻辑的意思是：

1. 找到所有给 seed 电影打过分的用户
2. 再把这些用户打过分的其他电影加入候选集

所以候选集不是全量电影，而是“和用户喜欢电影有共评用户的电影”。

### 2.5 相似度怎么计算

内嵌函数：

```python
item_cosine_sim(mid_a, mid_b)
```

步骤：

1. 取出两个电影的评分用户字典
2. 找共同评分用户
3. 若共同用户数小于 2，直接返回 0
4. 否则算点积
5. 再除以两个物品向量范数乘积

代码逻辑：

```python
common_users = set(users_a.keys()) & set(users_b.keys())
if len(common_users) < 2:
    return 0.0
dot = sum(users_a[u] * users_b[u] for u in common_users)
return dot / (norm_a * norm_b)
```

### 2.6 为什么共同用户数必须至少为 2

如果共同用户只有 1 个：

- 相似度会非常不稳定
- 很容易被单个用户的评分噪声带偏

这是一种简单但有效的噪声过滤。

### 2.7 最终分数怎么聚合

对每个候选电影，它会遍历所有 seed：

```python
candidate_scores[candidate_mid] += sim * weight
```

其中：

```python
weight = positive_weights.get(seed_mid, 1.0) / 5.0
```

也就是说：

```text
候选分数 = Σ [ seed 与候选的相似度 × seed评分权重 ]
```

如果用户特别喜欢某个 seed 电影，这个 seed 对候选分数的贡献也更大。

### 2.8 为什么要临时屏蔽测试电影的当前用户评分

这段很关键：

```python
temporary_masks = {}
for masked_mid in exclude_from_training:
    if masked_mid in self._item_users and user_id in self._item_users[masked_mid]:
        val = self._item_users[masked_mid].pop(user_id)
        temporary_masks[(masked_mid, user_id)] = val
```

它的目的不是线上业务，而是修复离线评测的数据泄露。

为什么会泄露？

假设当前用户同时评了：

- seed 电影 A
- 测试电影 T

如果不把当前用户对 T 的评分从矩阵里临时移除，那么：

- A 和 T 会因为“当前用户自己”而产生虚假的共现
- 导致 T 被异常高估

所以这里做的是：

- 评测时先把当前用户对测试电影的评分暂时从评分矩阵里挖掉
- 算完以后再恢复

这一步是代码里非常容易被忽略、但非常体现作者严谨性的细节。

### 2.9 推荐理由怎么生成

算法会记录“和候选最相似的 seed 电影”：

```python
if sim > best_sim:
    best_sim = sim
    best_seed = seed_mid
```

最后把它翻译成：

```text
因为你喜欢《某电影》，推荐评分行为相似的电影
```

所以这里的解释不是图谱解释，而是协同过滤的“相似物品解释”。

### 2.10 一句话总结它怎么算分

```text
候选分数 = 用户高分种子电影 与 候选电影 的 item-item cosine 相似度加权和
```

### 2.11 和 `content` 的区别

`content`：

- 比较的是电影内容特征和用户画像

`item_cf`：

- 比较的是电影和电影在用户评分矩阵中的相似性

所以：

- `content` 更依赖电影属性
- `item_cf` 更依赖用户行为共现

## 3. `kg_path` 源码走读

文件：

`db-backend/app/algorithms/kg_path.py`

这个算法的思想是：

“如果候选电影和用户喜欢的电影在图谱里共享重要实体，或者存在有意义的二跳桥接路径，那么候选就应该得高分。”

### 3.1 它依赖什么数据

它几乎不直接查数据库，而是依赖两部分：

1. `BaseRecommender` 提供的正反馈 seed
2. `GraphMetadataCache` 提供的图谱缓存

所以它的性能基础其实不是 Cypher 查询，而是预先构建好的内存索引。

### 3.2 用户上下文怎么拿

函数：

```python
_get_user_context()
```

它会缓存：

```python
(positive_movies, rated_mids)
```

其中：

- `positive_movies`
  - 图谱算法专用 seed 集合
- `rated_mids`
  - 用户所有已交互电影，用于候选排除

### 3.3 evidence bundle 是什么

函数：

```python
_get_evidence_bundle()
```

这是 `kg_path` 的核心中间层。

它先根据：

- `user_id`
- `exclude_from_training`
- `actor_order_limit`
- 是否启用二跳
- 各类 per-seed limit

构造一个复杂缓存 key。

缓存命中的话，直接返回已经算好的证据。

缓存未命中时，调用：

```python
_fetch_evidence_from_graph()
```

### 3.4 一跳证据怎么找

`_fetch_evidence_from_graph()` 会对每个 seed 电影分别做三件事：

1. 共享导演
2. 共享演员
3. 共享类型

它不是实时跑图数据库，而是直接查：

- `director_index`
- `actor_index`
- `genre_index`

也就是：

```python
GraphMetadataCache.inverted_index(...)
```

### 3.5 `_build_one_hop_records()` 是如何生成证据的

这一步本质上是：

“对 seed 电影的每个关系实体，找出所有也拥有该实体的候选电影。”

例如：

- seed 电影导演是诺兰
- inverted index 告诉你诺兰关联了哪些电影
- 那这些电影都成为潜在候选

然后对每个候选统计共享实体数量。

#### 导演 / 演员

对导演和演员，强度是：

```text
strength = seed_weight × shared_count
```

#### 类型

对类型，强度是：

```text
strength = seed_weight × min(shared_count / 3, 1.0)
```

为什么类型单独处理？

因为类型通常更泛，不适合像导演/演员那样线性放大。

### 3.6 演员为什么还要看 `actor_order_limit`

对于演员关系，有额外限制：

```python
cand_profile.actor_orders.get(entity_id, 9999) > actor_order_limit
```

默认只看演员表靠前的演员，比如前 `5` 个。

这样做是为了避免：

- 长演员表电影引入大量弱关系噪声
- 路径过于泛化

### 3.7 二跳证据怎么找

函数：

```python
_build_two_hop_records()
```

它构造的是一个“演员桥接”的二跳路径：

```text
seed电影 --演员A--> bridge电影 --演员B--> candidate电影
```

不是简单地说“共享演员”，而是：

- seed 先通过某个演员连到桥接电影
- 再从桥接电影的其他演员扩到候选电影

最终 relation 类型记成：

```python
"actor_actor"
```

这表示：

“我是通过演员二跳桥接发现的候选，而不是直接共享演员。”

### 3.8 `_select_genres()` 为什么优先选稀有类型

类型太热门时区分度很差。

它会先按：

```python
GraphMetadataCache.entity_degree(REL_GENRE, genre)
```

升序排序，也就是优先保留连接电影更少的类型。

然后再用：

```python
genre_max_degree
```

过滤超热门类型。

所以：

- “剧情” 这种大类很可能被弱化甚至过滤
- “黑色电影”“公路片”这类稀有类型更可能保留

### 3.9 最终分数怎么计算

核心函数：

```python
_score_candidates()
```

它先定义关系权重：

```python
director_weight
actor_weight
genre_weight
two_hop_weight
```

然后对每条证据：

```text
contribution = relation_weight × strength × penalty
```

其中 penalty 来自：

```python
_entity_penalty()
```

如果开启 `use_degree_penalty`：

```text
penalty = 平均 safe_idf(entity_degree)
```

所以总分是：

```text
候选分数 = Σ 每条图谱证据的加权贡献
```

### 3.10 推荐理由怎么生成

函数：

```python
_reason_for_record()
```

会根据 relation 生成不同文本：

- 共同导演
- 共同演员
- 同属某类型
- 通过演员二跳关系发现

然后 `_score_candidates()` 会按贡献值排序，最多保留 3 条不重复理由。

### 3.11 一句话总结它怎么算分

```text
候选分数 = 共享导演 / 演员 / 类型 / 二跳演员桥接 的证据贡献总和
```

### 3.12 为什么它解释性强但效果可能弱

因为它非常依赖显式结构重叠：

- 有共享关系时很合理
- 没共享关系时就没有证据

所以它擅长解释，但泛化能力通常弱于嵌入类方法。

## 4. `kg_embed` 源码走读

文件：

`db-backend/app/algorithms/kg_embed.py`

这是最复杂的算法，也是当前图谱方法里表现最好的一个。

它其实分成两段：

1. 离线：训练或加载 TransE 嵌入
2. 在线：基于嵌入和实体重叠做推荐

### 4.1 它依赖什么

依赖三部分：

1. `GraphMetadataCache` 导出的三元组
2. 本地 embedding 工件文件
3. 用户正反馈 seed

所以它不是实时查图数据库算分，而是：

- 先把图谱压成向量空间
- 在线只做向量运算和少量结构重叠统计

### 4.2 `_load_or_train()` 在做什么

这个函数是 KG-Embed 的入口门卫。

逻辑顺序是：

1. 先看共享缓存 `_shared_artifacts_by_scope`
2. 没命中就看磁盘上有没有 embedding 文件
3. 如果磁盘有，直接加载
4. 如果磁盘没有，再看是否允许在线训练
5. 如果不允许训练，返回 `None`
6. 如果允许训练，执行 `_train_transe()`

这里的 scope 有两种：

- `core`
- `expanded`

### 4.3 `core` 和 `expanded` 的区别

由 `GraphMetadataCache.build_triples()` 决定。

#### `core`

只保留三类关系：

- `DIRECTED`
- `ACTED_IN`
- `HAS_GENRE`

#### `expanded`

除了上面三类，还加入：

- `IN_REGION`
- `IN_LANGUAGE`
- `HAS_CONTENT_TYPE`
- `IN_YEAR_BUCKET`

这就是评测报告里“扩图三元组”优于“原始三关系”的根本原因。

### 4.4 三元组是怎么导出的

`build_triples()` 会遍历每部电影的 profile，把它展开成：

```python
(head, relation, tail)
```

例如：

- `person_1047973 --DIRECTED--> movie_1292052`
- `movie_1292052 --HAS_GENRE--> genre_犯罪`
- `movie_1292052 --IN_REGION--> region_美国`

而且默认 `include_inverse=True`，所以还会补反向边：

- `movie_1292052 --DIRECTED_REV--> person_1047973`

这样做的好处是：

- 嵌入训练时，图结构方向信息更完整

### 4.5 TransE 训练流程是什么

函数：

```python
_train_transe()
```

核心流程是：

1. 导出 triples
2. 给 entity 和 relation 编号
3. 构造 `entity_to_idx`、`relation_to_idx`
4. 按 `VALIDATION_RATIO = 0.1` 划分 train / val
5. 随机初始化 entity embedding 和 relation embedding
6. 批量训练
7. 每隔若干 epoch 在验证集上评估
8. early stop
9. 保存 `.npz` 和 `.json`

### 4.6 负采样怎么做

函数：

```python
_sample_negative_batch()
```

关键点是：它不是乱替换，而是“按实体类型替换”。

比如：

- `(person, DIRECTED, movie)`

就只能：

- 用另一个 `person` 替换头
- 或用另一个 `movie` 替换尾

不能把 `genre` 乱塞进去。

这一步能保证负样本仍然在类型上合理。

### 4.7 TransE 的核心公式

TransE 的直觉是：

```text
head + relation ≈ tail
```

理想情况下：

```text
e_h + e_r ≈ e_t
```

代码里正例距离：

```python
pos_dist = entity_emb[h] + relation_emb[r] - entity_emb[t]
pos_score = sum(pos_dist ** 2)
```

负例也同样计算：

```python
neg_score = sum(neg_dist ** 2)
```

损失函数：

```text
loss = max(0, margin + pos_score - neg_score)
```

也就是：

- 希望正例距离更小
- 负例距离更大
- 并且至少拉开一个 `margin`

### 4.8 在线推荐时，用户向量怎么建

函数：

```python
_get_user_components()
```

它先取所有 seed 电影在 embedding 空间中的向量，再按评分权重做平均：

```python
user_vec = np.average(seed_vectors, axis=0, weights=seed_weights)
```

这就是用户在图谱嵌入空间里的“兴趣中心”。

### 4.9 三个在线分量是什么

`kg_embed` 在线打分不是只靠一个分量，而是三个：

#### 1. `centroid_scores`

```python
centroid_scores = movie_matrix @ user_vec
```

含义：

“候选电影和用户兴趣中心的相似度”

#### 2. `max_seed_scores`

```python
seed_sims = movie_matrix @ seed_vectors.T
max_seed_scores = np.max(seed_sims * seed_weights, axis=1)
```

含义：

“候选电影和用户最喜欢某部 seed 电影的最大相似度”

#### 3. `entity_overlap_scores`

由 `_entity_overlap_components()` 生成。

它不是纯向量相似，而是：

“候选电影与用户喜欢电影在结构实体上重叠得有多强”

### 4.10 `entity_overlap_scores` 怎么算

对每个 seed 电影、每个关系实体：

```text
contribution = rating_weight × relation_weight × safe_idf(entity_degree)
```

这里 relation_weight 是写死的：

- 导演 `1.0`
- 演员 `0.6`
- 类型 `0.4`
- 地区 `0.2`
- 语言 `0.2`
- 内容形式 `0.1`
- 年代桶 `0.1`

然后：

- 如果候选电影也拥有这个实体
- 就把 contribution 加到候选的 overlap 分数上

所以 overlap 分量本质上是：

“结构实体重叠的累积分”

### 4.11 为什么还要做归一化

三个分量量纲不一样：

- centroid 是点积
- max_seed 是最大相似度
- overlap 是累加分

不能直接加。

所以用 `_normalize_vector()` 做 min-max 归一化：

```text
normalized = (value - min) / (max - min)
```

这样三类分量都被压到可比较的区间里。

### 4.12 最终分数怎么融合

如果 `use_fusion_ranking=True`：

```text
final_score =
  centroid_weight * centroid_scores
  + max_seed_weight * max_seed_scores
  + entity_overlap_weight * overlap_scores
```

当前较优配置里常见的是：

- `centroid_weight = 0.2`
- `max_seed_weight = 0.2`
- `entity_overlap_weight = 0.6`

也就是说当前版本更重视结构实体重叠。

### 4.13 推荐理由怎么生成

函数：

```python
_reason_for_movie()
```

逻辑是：

- 如果 overlap 分量最大
  - 优先说“偏好相同导演/演员/类型”
- 如果 max_seed 更强
  - 说“与您高评分电影在嵌入空间中最相近”
- 否则
  - 说“基于知识图谱嵌入空间的语义相似性推荐”

所以它的解释是“向量语义 + 结构重叠”的混合解释。

### 4.14 一句话总结它怎么算分

```text
候选分数 = 用户中心相似度 + 最强 seed 相似度 + 结构实体重叠分量 的加权融合
```

### 4.15 为什么 `expanded relations` 更强

因为 `expanded` 模式引入了：

- 地区
- 语言
- 内容形式
- 年代桶

这样图谱结构更丰富，embedding 能学习到更多维度的相似性。

所以：

- `core` 更像“导演/演员/类型”三关系图
- `expanded` 更像“结构信息更完整的电影知识图”

## 5. `cfkg` 主链路走读

文件：

`db-backend/app/algorithms/cfkg.py`

这个类最容易被误解。

它不是一种全新单独算法，而是一个融合框架。

### 5.1 它初始化了什么

在 `__init__()` 里：

```python
self._item_cf = ItemCFRecommender()
self._kg_embed = KGEmbedRecommender(...)
self._kg_path = KGPathRecommender(...)
self._content = ContentBasedRecommender()
```

这说明：

`cfkg` 的本质是“组装现有分支”。

### 5.2 `DEFAULT_CONFIG` 的含义

关键参数：

- `item_cf_weight`
- `kg_embed_weight`
- `kg_path_weight`
- `item_cf_recall`
- `kg_embed_recall`
- `kg_path_recall`
- `content_recall`
- `content_fallback_weight`
- `min_candidate_pool`

这组参数同时控制：

1. 每个分支召回多少候选
2. 融合时各分支占多大权重
3. 候选池太小的时候是否引入 `content` 保底

### 5.3 推荐流程怎么走

`recommend()` 的主流程是：

1. 跑 `item_cf`
2. 跑 `kg_embed`
3. 跑 `kg_path`
4. 合并候选
5. 若候选池太小，补 `content`
6. 重新归一化可用分支权重
7. 计算每个候选的最终 `raw_score`
8. 输出理由和来源算法

### 5.4 为什么 `_safe_recommend()` 很重要

它把分支调用包起来：

```python
try:
    recommender.recommend(...)
except:
    return []
```

这意味着：

- 某个分支挂了
- 不会把整个 CFKG 拖死

例如：

- `kg_embed` 嵌入工件缺失
- `kg_path` 某次 evidence 计算异常

这时 CFKG 仍然能退化运行。

### 5.5 权重为什么还要再归一化

函数：

```python
_resolve_branch_weights()
```

如果某分支没有结果，它的配置权重就不该继续占比。

例如：

原始配置：

- item_cf = 0.6
- kg_embed = 0.25
- kg_path = 0.15

如果 `kg_path` 没有结果，那么可用权重会重新归一化成：

```text
item_cf = 0.6 / 0.85
kg_embed = 0.25 / 0.85
```

所以它不是机械地按原权重相加，而是对“实际有输出的分支”重新分配。

### 5.6 候选怎么合并

函数：

```python
_merge_branch_results()
```

每个候选电影都会形成一个 payload：

```python
{
  "scores": {
    "item_cf": ...,
    "kg_embed": ...,
    "kg_path": ...
  },
  "reasons": {
    "item_cf": ...,
    "kg_embed": ...,
    "kg_path": ...
  }
}
```

它不是直接把分支结果拼在一起，而是按 `mid` 聚合。

### 5.7 什么时候会触发 `content` fallback

两种情况：

1. `item_cf` 分支没有结果
2. 合并后候选池小于 `min_candidate_pool`

这时会额外跑一次 `content`，把内容基线的候选补进去。

注意：

- `content` 在 CFKG 中不是主分支
- 它更像一个保底召回器

### 5.8 最终分数怎么计算

每个候选会先把各分支得分乘各自权重：

```text
weighted_contribution(branch) = branch_score × normalized_weight
```

然后：

```text
raw_score = Σ weighted_contribution(branch)
```

如果三大分支都没贡献，但 `content` 有结果，就会退化成：

```text
raw_score = content_score × content_fallback_weight
```

最后再用当前候选集中最大 raw_score 做归一化。

### 5.9 推荐理由怎么排序

函数：

```python
_ordered_reasons()
```

理由优先级固定为：

```text
kg_path -> kg_embed -> item_cf -> content
```

这说明系统在展示层更偏好先展示“图谱解释”。

来源算法排序则来自：

```python
_ordered_sources()
```

它按加权贡献从高到低排序。

所以一个候选最终会输出：

- `reasons`
- `source_algorithms`

### 5.10 为什么当前最佳参数里 `item_cf_weight` 最高

从评测看，`ItemCF` 当前整体效果最好。

所以调参后最佳配置里：

- `item_cf_weight` 最高

这说明当前 CFKG 的实际有效信息来源，仍然主要是协同过滤。

从工程上看这很正常：

- `item_cf` 精度强
- `kg_embed` 和 `kg_path` 提供补充语义和解释
- CFKG 当前还处在“融合增强”阶段，而不是彻底超越 `item_cf`

### 5.11 一句话总结它怎么算分

```text
候选分数 = item_cf + kg_embed + kg_path 分支得分的加权和，必要时再加 content 保底
```

### 5.12 为什么它是主链路但不是当前最优

因为：

- 它代表的是系统最终推荐框架
- 不是单一分支算法

从结构上它最完整，但从当前评测看还没有把多分支优势完全转化成最强指标。

## 6. 评测器是怎么证明这些结果的

文件：

`db-backend/app/algorithms/evaluator.py`

这部分很重要，因为老师问“你怎么证明算法效果”的时候，你不能只说“我跑了结果”。

### 6.1 评测协议

核心常量：

- `POSITIVE_THRESHOLD = 3.5`
- `NUM_NEGATIVES = 99`
- `USER_SPLIT_SEED = 42`
- `VALIDATION_RATIO = 0.2`
- `NEGATIVE_SAMPLE_SEEDS = [42, 52, 62, 72, 82]`

评测协议是：

```text
validation + test sampled leave-one-out
```

即：

1. 每个用户保留一个测试正样本
2. 从全库里随机抽 99 个负样本
3. 构成 100 个候选
4. 看测试正样本能否排进 Top-K

### 6.2 为什么 `Recall@K` 和 `Hit Rate@K` 一样

因为每个用户只保留了 1 个正样本。

所以：

- Top-K 里命中它，则 recall = 1，hit = 1
- 没命中，则 recall = 0，hit = 0

这也是报告里强调主表看 `HR@K` 与 `NDCG@K` 的原因。

### 6.3 指标怎么累计

函数：

```python
evaluate_algorithm()
```

对每个用户、每个负采样 seed，会累计：

- hits
- precision_sum
- recall_sum
- ndcg_sum
- coverage_mids
- diversity_sum

#### precision

如果命中：

```text
precision@k += 1 / k
```

#### recall

因为只有一个正样本：

```text
recall@k += 1 或 0
```

#### ndcg

用 `_ndcg_at_k()` 计算：

```text
DCG = 1 / log2(rank + 1)
```

#### diversity

用 `_diversity_at_k()` 计算推荐列表间的 Jaccard 差异度。

它先取电影的 relation token 集合，再算：

```text
1 - JaccardSimilarity
```

### 6.4 为什么要分验证集和测试集

因为：

- 有些算法有参数网格
- 不能直接在测试集上调参

所以流程是：

1. 验证集选最优参数
2. 测试集只做最终汇报

这一步让结果更规范。

## 8. 当前实验结果应该怎么解释

基于当前报告：

### 8.1 为什么 `ItemCF` 最强

因为公开豆瓣用户数据本身就包含较强协同行为信号。

在这种数据条件下：

- 用户-电影评分矩阵已经足够有信息量
- item-item 共现关系非常强

所以它自然能拿到最好指标。

### 8.2 为什么 `KG-Embed` 是最佳图谱方法

因为它同时利用了：

- 图谱嵌入空间语义
- seed 电影最大相似度
- 结构实体重叠

比单纯走显式路径更能泛化。

### 8.3 为什么 `CFKG` 还没超过 `ItemCF`

当前原因很可能是：

- 融合权重仍偏保守
- 候选融合策略还不够强
- 图谱分支带来的增益还不足以完全抵消融合噪声
- 计算耗时也比较高

但它作为系统主链路仍然成立，因为它代表“精度 + 解释 + 结构语义”的综合方向。

## 10. 你背诵时最实用的总结版

如果老师让你一句话讲每个算法，可以这样说：

- `content`
  - “用电影元数据构造向量，再和用户兴趣向量做余弦相似度。”
- `item_cf`
  - “用用户评分矩阵计算电影和电影之间的相似度，再把相似度按用户高分种子加权汇总。”
- `kg_path`
  - “找候选电影和用户喜欢电影之间的共享导演、演员、类型和二跳桥接路径，再把这些路径证据加权求和。”
- `kg_embed`
  - “先把知识图谱训练成嵌入空间，再综合用户中心相似度、最强 seed 相似度和实体重叠分量做融合评分。”
- `cfkg`
  - “把 `item_cf`、`kg_embed`、`kg_path` 的结果融合起来，并在必要时用 `content` 做保底召回。”

这五句话，是你后面答辩时最值得先背熟的部分。
