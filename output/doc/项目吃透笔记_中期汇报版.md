# 基于知识图谱的电影推荐系统 项目吃透笔记

这份笔记不是按文件清单罗列，而是按“你明天最需要讲清楚的系统模块”来整理。目标只有一个：让你能用自己的话，把这个项目的结构、数据流、算法和当前结论讲出来。

## 1. 先用一句话说清项目

这是一个围绕豆瓣电影数据构建的端到端推荐系统，完整链路是：

`Playwright 爬虫采集 -> MySQL 原始数据与用户行为 -> 清洗标准化 -> ETL 导入 Neo4j -> FastAPI 提供查询/推荐/解释接口 -> Vue 前端展示`

它不是单纯的“图谱可视化项目”，也不是单纯的“推荐算法实验”；它是一个把数据采集、知识图谱、推荐算法、离线评测和前端展示串起来的完整系统。

## 2. 核心事实与汇报时必须统一的口径

- 后端接口总数：`58`
- 前端页面总数：`13`
- 推荐算法总数：`5`
- 后端测试文件：`11`
- 当前正式离线评测报告时间：`2026-03-15`
- 当前结论：
  - `ItemCF` 是整体最优方法
  - `KG-Embed` 是当前表现最好的图谱方法
  - `CFKG` 是当前主链路，但效果和耗时都还有优化空间

## 3. 模块化理解

下面每个模块都按 6 个问题来讲：`职责`、`输入`、`输出`、`核心实现`、`为什么这样设计`、`可能追问`

### 3.1 爬虫采集模块

#### 职责

从豆瓣页面采集电影和影人数据，解决反爬、并发抓取、断点恢复和任务锁问题。

#### 输入

- 豆瓣电影或影人页面
- 任务队列表中的待抓取 ID
- 代理、浏览器、请求头、Cookie 等运行配置

#### 输出

- `movies` 表中的电影原始信息
- `person` 表中的影人信息
- 供后续图谱构建使用的原始字段，例如 `actor_ids`、`director_ids`、`genres`、`regions`

#### 核心实现

- `db-spiders/crawl_movie.py`
  - 基于 `Playwright`
  - 支持并发 worker
  - 使用 `crawl_status`、`crawl_locked_at`、`crawl_worker` 进行分布式任务锁
- `db-spiders/crawl_person.py`
  - 异步 Playwright
  - 集成代理池 `ProxyManager`
  - 重点抓影人详情字段，如出生地、职业、简介等

#### 为什么这样设计

- 豆瓣页面并不是稳定的静态 HTML，请求式抓取容易被限制，所以用真实浏览器模拟更稳
- 电影和影人分开抓，便于控制并发和重试策略
- 任务锁是为了支持中断恢复和多 worker 协作，避免重复抓取

#### 可能追问

- 为什么不用 `Requests` 或 `Scrapy`？
  - 因为目标站点存在较强反爬和动态加载，Playwright 更接近真实用户行为，更容易稳定采集
- 为什么电影和影人拆成两个脚本？
  - 数据结构、抓取频率、失败重试策略不同，拆开后更容易控制复杂度

### 3.2 数据清洗与质量模块

#### 职责

把采集来的半结构化数据统一格式、修正空值、规范字段，为后续图谱导入和推荐算法提供可用数据。

#### 输入

- MySQL 中原始采集数据

#### 输出

- 清洗后的 `movies` 和 `person` 表
- 数据质量报告

#### 核心实现

- `data_processing/clean_pipeline.py`
  - 清洗 `genres`、`regions`、`languages`
  - 把空字符串统一转为 `NULL`
  - 把无意义的 `0` 分和 `0` 票数改为 `NULL`
  - 检查 `actor_ids` / `director_ids` 格式
- `data_processing/reports/data_quality_report.md`
  - 当前规模：`135,990` 部电影，`198,067` 位影人
  - 缺失率较高字段：`douban_score`、`storyline`、`alias`、`mins`

#### 为什么这样设计

- 爬虫数据天然脏乱，不先清洗，图谱节点和关系会出现大量重复与异常
- 清洗逻辑直接在 MySQL 中做，减少中间文件和额外同步步骤

#### 可能追问

- 为什么评分缺失率高？
  - 系统抓的是大规模电影条目，不是只有热门电影，很多长尾作品本来就没有足够评分
- 这些缺失值会怎么影响推荐？
  - 会影响基于内容和排序的某些字段，但图谱结构关系仍然可用于查询和推荐

### 3.3 图谱建模与 ETL 模块

#### 职责

把 MySQL 的结构化数据转为 Neo4j 中的属性图，支撑图谱查询、关系路径推荐和嵌入训练。

#### 输入

- 清洗后的 MySQL 数据

#### 输出

- Neo4j 中的节点和关系

#### 核心实现

- `data_processing/etl_to_neo4j.py`
  - 节点：
    - `Movie`
    - `Person`
    - `Genre`
    - `Region`
    - `Language`
    - `ContentType`
    - `YearBucket`
  - 关系：
    - `DIRECTED`
    - `ACTED_IN`
    - `HAS_GENRE`
    - `IN_REGION`
    - `IN_LANGUAGE`
    - `HAS_CONTENT_TYPE`
    - `IN_YEAR_BUCKET`
- `documents/schema_design.md`
  - 说明了节点、关系、索引和约束设计

#### 为什么这样设计

- `Movie`、`Person`、`Genre` 是推荐和图谱展示最核心的三类实体
- `Region`、`Language`、`ContentType`、`YearBucket` 是为了增强图谱结构特征，给 `KG-Embed` 这类方法提供更多关系信息
- 采用属性图模型，是因为 Neo4j 更适合做关系查询、路径搜索和图可视化

#### 可能追问

- 为什么同时用 MySQL 和 Neo4j？
  - MySQL 更适合存原始数据、用户表、认证和行为数据；Neo4j 更适合存图结构和做路径/邻接查询，两者分工明确
- 为什么 `Person` 不拆成导演和演员两个节点？
  - 同一个人可能同时担任导演和演员，统一建模更符合真实世界，也减少实体重复

### 3.4 后端接口与服务分层模块

#### 职责

对外提供电影查询、影人查询、图谱探索、统计分析、用户行为、推荐和推荐解释等接口。

#### 输入

- 前端请求
- MySQL / Neo4j 中的数据

#### 输出

- JSON API 响应

#### 核心实现

- 应用入口：`db-backend/app/main.py`
- 路由层：`db-backend/app/routers/`
  - `auth`
  - `users`
  - `movies`
  - `persons`
  - `graph`
  - `stats`
  - `recommend`
  - `admin`
  - `proxy`
- 服务层：`db-backend/app/services/`
  - `movie_service.py`
  - `person_service.py`
  - `graph_service.py`
  - `user_service.py`
  - 认证与管理员相关服务

#### 为什么这样设计

- 路由层只做参数接收和返回，服务层负责业务逻辑，结构清晰
- 电影搜索和用户行为更适合走 MySQL
- 图谱详情、路径和统计更适合走 Neo4j

#### 可能追问

- 后端为什么是 FastAPI？
  - 类型标注和接口文档友好，适合快速开发并且易于组织复杂接口
- 为什么接口会这么多？
  - 因为系统不仅有推荐，还覆盖了搜索、详情、图谱、统计、认证、用户画像和评测报告

### 3.5 推荐算法与解释模块

#### 职责

基于用户的评分、喜欢和想看行为，给出个性化推荐，并为推荐结果生成可解释信息。

#### 输入

- 用户行为：`rating`、`like`、`want_to_watch`
- 图谱结构
- 电影元数据

#### 输出

- 推荐列表
- 推荐分数
- 推荐原因
- 可视化解释图

#### 核心实现

- 算法注册表：`db-backend/app/algorithms/__init__.py`
- 五种算法：
  - `content`
  - `item_cf`
  - `kg_path`
  - `kg_embed`
  - `cfkg`
- 推荐入口：`db-backend/app/routers/recommend.py`
- 解释逻辑：
  - 若目标推荐电影和用户历史正反馈电影存在共享导演、演员、类型，则构造解释图
  - 若不存在重叠证据，则退化为展示目标电影自身的导演、演员、类型上下文

#### 为什么这样设计

- `content` 和 `item_cf` 是基线方法，用来对照图谱方法是否真的有价值
- `kg_path` 强调显式关系和可解释性
- `kg_embed` 强调潜在语义和泛化能力
- `cfkg` 把 `item_cf + kg_embed + kg_path` 融合起来，作为系统主链路

#### 关键细节

- 在 `BaseRecommender` 里，`like` 会被映射成合成评分 `4.5`，`want_to_watch` 会映射成 `4.0`
- 冷启动阈值是 `3`，少于 3 个有效信号会判定为冷启动用户
- `CFKG` 当前最佳参数里，`item_cf_weight` 最高，说明当前融合主链路仍然主要依赖协同过滤召回

#### 可能追问

- 为什么 `CFKG` 不是当前最优还要作为主链路？
  - 因为它不是单一算法，而是系统主推荐框架，目标是把协同过滤的精度和图谱方法的解释性结合起来；当前版本已经可用，但还要继续调权重和候选融合策略
- 为什么 `KG-Embed` 比 `KG-Path` 更强？
  - `KG-Embed` 能学习更丰富的结构语义，对稀疏场景的泛化更好；`KG-Path` 更强在解释，但分数上受显式关系覆盖限制

### 3.6 评测协议与实验结果模块

#### 职责

离线评估不同推荐算法的效果，并生成标准化报告。

#### 输入

- 公开豆瓣用户行为数据
- 五种推荐算法的推荐结果

#### 输出

- Markdown 报告
- JSON 报告

#### 核心实现

- `db-backend/app/algorithms/evaluator.py`
- 报告文件：
  - `db-backend/reports/eval_results.md`
  - `db-backend/reports/eval_results.json`

#### 当前评测协议

- 用户来源：公开豆瓣用户
- 验证 / 测试用户数：`42 / 168`
- 评测方法：`sampled leave-one-out`
- 候选集：`1 positive + 99 negatives`
- 指标：
  - `HR@K`
  - `NDCG@K`
  - `Coverage`
  - `Diversity`
  - `Time`

#### 当前结果要怎么说

- `ItemCF` 是当前整体最优
- `KG-Embed` 是当前图谱方法中表现最好的
- `CFKG` 的 `NDCG@10` 不差，但 `HR@10` 和响应时间还不够理想

#### 可能追问

- 为什么采用 `sampled leave-one-out`？
  - 对推荐系统离线评测很常见，能在计算量可控的情况下对不同算法做公平对比
- 为什么 `Recall@K` 没单独强调？
  - 因为当前协议下每个用户只保留一个测试正样本，所以 `Recall@K` 和 `Hit Rate@K` 数值等价

## 4. 四条必须讲清的端到端链路

### 4.1 从豆瓣页面到 MySQL

1. Playwright 打开豆瓣页面
2. 解析电影或影人字段
3. 把结果落到 MySQL 的 `movies` / `person` 表
4. 更新任务状态，避免重复抓取

### 4.2 从 MySQL 到 Neo4j 图谱

1. 清洗 MySQL 中的电影和影人字段
2. 统一类型、地区、语言、演员、导演等字段格式
3. 用 `etl_to_neo4j.py` 创建节点和关系
4. 在 Neo4j 中形成可查询的图结构

### 4.3 从用户行为到个性化推荐

1. 用户在前端登录后，对电影进行评分、喜欢或想看
2. 行为写入 MySQL 的用户行为表
3. 推荐接口读取这些行为，构建正反馈集合
4. 5 种算法之一或 CFKG 主链路生成推荐结果
5. 前端展示推荐卡片和推荐理由

### 4.4 从推荐结果到解释与展示

1. 用户点击推荐结果
2. 前端请求 `/api/recommend/explain`
3. 后端在图缓存中寻找共享导演、演员、类型等证据
4. 返回 `reason_paths`、`nodes`、`edges`
5. 前端将解释信息以抽屉和图谱方式展示

## 6. 你最容易讲错的点

- 不要说“CFKG 当前效果最好”
  - 真实情况是 `ItemCF` 最强，`KG-Embed` 是当前最佳图谱方法
- 不要把项目说成“只是图谱展示”
  - 它是带推荐、解释、评测和前端展示的完整系统
- 不要只说“用了 Neo4j 很高级”
  - 要说清它和 MySQL 的分工
- 不要把冷启动说成“没有解决”
  - 真实情况是：系统做了阈值判断、热门回退和行为补充机制
- 不要把 AI 说成“帮我写了点代码而已”
  - 更稳的口径是：AI 辅助了实现效率，但需求拆解、模块整合、验证测试和结果取舍由你主导
