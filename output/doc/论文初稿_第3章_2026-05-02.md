# 第 3 章 系统总体设计

本章在需求分析的基础上，对基于知识图谱的电影推荐系统进行总体设计。系统采用前后端分离架构，将数据采集、关系型数据存储、数据清洗、知识图谱构建、后端接口服务、前端页面展示和推荐评测分层组织。整体设计重点在于明确 MySQL 与 Neo4j 的职责分工，保证电影数据、用户行为、图谱查询和推荐算法之间的数据流转清晰可追溯。

## 3.1 系统总体架构

系统总体架构按照数据流向可以划分为采集层、数据层、图谱层、服务层、表现层和评测层六个部分。

采集层负责电影和影人数据的获取。系统使用 Playwright 采集电影条目和影人信息，采集过程通过任务状态、锁定时间、采集 worker 和失败标记控制任务流转。采集结果写入 MySQL 中的原始业务表，为后续清洗和图谱构建提供数据来源。

数据层以 MySQL 为核心，保存电影条目、影人信息、采集辅助数据、用户账号、用户会话、用户评分和用户偏好等结构化数据。MySQL 负责承载原始业务数据和用户行为数据，适合处理按主键查询、分页查询、用户行为增删改查和会话管理等场景。数据处理模块在该层基础上执行字段清洗、空值处理、质量分析和 ETL 数据准备。

图谱层以 Neo4j 为核心，保存电影知识图谱。系统将电影、影人、类型、地区、语言、内容形式和年代分桶等对象建模为节点，将执导、参演、所属类型、所属地区、语言、内容形式和年代分桶等语义联系建模为关系。图谱层主要支撑电影详情关联查询、影人关联查询、图谱探索、统计分析和推荐解释。

服务层由 FastAPI 后端实现，位于前端和数据资源之间。后端启动时初始化 MySQL 连接池和 Neo4j Driver，并预热推荐图谱缓存与 KG-Embed 工件。服务层按业务模块提供认证、用户行为、电影、影人、图谱、统计、推荐和图片代理接口，同时负责 Token 鉴权、请求参数校验、推荐算法调用和结果组装。

表现层由 Vue3 前端实现。前端通过 Vue Router 组织页面，通过 Axios 调用后端 `/api` 接口，通过 Pinia 管理用户认证状态，通过 Element Plus 构建交互组件，并使用 ECharts 展示图谱和统计图表。表现层面向游客和注册用户提供首页、搜索、电影库筛选、电影详情、影人详情、图谱探索、统计看板、登录注册、个人中心和推荐中心等页面。

评测层用于验证推荐模块效果和系统链路可用性。推荐算法包括 `content`、`item_cf`、`kg_path`、`kg_embed` 和 `cfkg` 五类，正式离线评测采用 `1 positive + 499 negatives` 的 sampled leave-one-out 协议。评测报告和在线链路验收报告不直接参与在线业务请求，但为第 5 章的测试与实验分析提供依据。

从数据流角度看，系统主线为：Playwright 采集电影与影人数据，写入 MySQL；数据处理脚本清洗数据并导入 Neo4j；FastAPI 同时访问 MySQL 和 Neo4j，对外提供查询、图谱、统计、用户行为和推荐接口；Vue3 前端调用接口完成页面展示；推荐算法读取用户评分、喜欢和想看等行为，并结合电影内容特征与知识图谱结构生成推荐结果和解释信息。

## 3.2 数据库总体设计

系统采用 MySQL 与 Neo4j 双数据库设计。MySQL 负责保存结构化业务数据和用户行为数据，Neo4j 负责保存图结构关系和支撑图谱查询。两类数据库不是互相替代的关系，而是按照数据访问特点进行分工：MySQL 适合保存原始字段、用户行为和会话记录；Neo4j 适合表达实体之间的多跳关系，并为图谱探索和推荐解释提供结构化依据。

### 3.2.1 MySQL 数据库设计

MySQL 数据库包含电影采集入口表、电影原始数据表、影人原始数据表、影人采集辅助表和用户行为相关表。根据当前 DDL 导出结果，主要表的用途如表 3-1 所示。

**表 3-1 MySQL 主要业务表设计**

| 表名 | 当前记录数 | 主要用途 |
|---|---:|---|
| `subjects` | 192,035 | 豆瓣电影条目来源和采集任务入口，保存采集状态、锁定时间和 worker 标识 |
| `movies` | 192,032 | 保存电影元数据，包括名称、别名、封面、年份、类型、地区、语言、片长、评分、票数、剧情简介和演职员 ID |
| `person` | 236,882 | 保存影人元数据，包括姓名、性别、英文名、中文名、出生日期、出生地、职业和简介 |
| `person_obj` | 236,890 | 保存待采集影人 ID 和采集状态，用于影人采集流程 |
| `users` | 518 | 保存注册用户账号、邮箱、密码哈希、昵称、头像、状态和登录时间 |
| `user_sessions` | 89 | 保存 refresh token 哈希、用户代理、IP、过期时间和撤销时间 |
| `user_movie_prefs` | 84,983 | 保存用户喜欢和想看行为，按用户和电影建立唯一约束 |
| `user_movie_ratings` | 107,076 | 保存用户评分和短评，按用户和电影建立唯一约束 |
| `user_search_history` | 0 | 保存用户搜索关键词、结果数量和搜索时间 |

其中，`subjects` 和 `person_obj` 偏向采集任务控制，`movies` 和 `person` 是电影领域原始业务数据的核心表，`users`、`user_sessions`、`user_movie_prefs` 和 `user_movie_ratings` 是注册用户行为和推荐输入的核心表。电影表以 `douban_id` 作为主键，影人表以 `person_id` 作为主键，用户行为表通过 `user_id` 关联普通用户，并通过 `mid` 对应电影 ID。

从逻辑关系看，电影与影人的原始关联主要保存在 `movies` 表的 `director_ids` 和 `actor_ids` 字段中，后续 ETL 根据这些字段生成图数据库中的 `DIRECTED` 和 `ACTED_IN` 关系。用户侧数据则采用独立表保存，评分表记录 `0.5` 到 `5.0` 的评分值，偏好表记录 `like` 和 `want_to_watch` 两类偏好。该设计使电影知识数据和用户行为数据既能独立维护，又能在推荐算法中共同使用。

### 3.2.2 Neo4j 图数据库设计

Neo4j 数据库用于保存电影知识图谱。图数据库以实体节点和关系边表达电影领域知识，适合处理多跳关系查询、路径分析和图谱可视化。系统当前 ETL 主图谱包含 7 类节点和 7 类关系，其规模口径以 `output/doc/数据库与图谱计数证据_2026-05-01.md` 中的实测结果为准。

Neo4j 与 MySQL 的数据映射关系主要包括三类。第一，`movies` 表导入为 `Movie` 节点，保存电影名称、内容形式、评分、票数、上映日期、年份、地区、语言、片长、剧情简介、别名和详情页 URL 等属性。第二，`person` 表导入为 `Person` 节点，保存影人姓名、性别、英文名、中文名、出生日期、去世日期、出生地、职业和简介等属性。第三，电影表中的类型、地区、语言、内容形式和年份字段被拆分或派生为维度节点，并与电影节点建立关系。

系统在 ETL 阶段为 `Movie.mid`、`Person.pid`、`Genre.name`、`Region.name`、`Language.name`、`ContentType.name` 和 `YearBucket.name` 创建唯一约束，并为电影标题、电影评分和影人姓名创建索引。唯一约束用于减少重复节点，索引用于提升常用查询字段的定位效率。Neo4j 中存在少量用户行为同步关系，但电影知识图谱 ETL 主规模只统计 7 类主节点和 7 类主关系，不将少量用户行为同步关系并入主图谱关系总数。

## 3.3 知识图谱 Schema 设计

电影知识图谱 Schema 的目标是用较稳定的实体和关系表达电影领域数据，使后端查询、前端图谱展示和推荐解释能够共享同一套图结构。当前 Schema 采用属性图模型，节点表示电影、影人和结构化维度，关系表示影人与电影、电影与维度之间的语义连接。

表 3-2 给出了知识图谱节点设计。

**表 3-2 Neo4j 节点设计表**

| 节点标签 | 主键 | 来源 | 说明 | 当前数量 |
|---|---|---|---|---:|
| `Movie` | `mid` | `movies.douban_id` | 电影实体，保存名称、内容形式、评分、票数、年份、地区、语言、片长、简介等属性 | 192,032 |
| `Person` | `pid` | `person.person_id` | 影人实体，导演和演员统一建模为 Person，通过关系区分职责 | 236,883 |
| `Genre` | `name` | `movies.genres` | 类型维度节点，如剧情、喜剧等 | 32 |
| `Region` | `name` | `movies.regions` | 制片地区维度节点 | 563 |
| `Language` | `name` | `movies.languages` | 语言维度节点 | 1,271 |
| `ContentType` | `name` | `movies.type` | 内容形式维度节点 | 2 |
| `YearBucket` | `name` | `movies.year` 派生 | 年代分桶节点，包括 `before_1990`、`1990s`、`2000s`、`2010s`、`2020s_plus` | 5 |

表 3-3 给出了知识图谱关系设计。

**表 3-3 Neo4j 关系设计表**

| 关系类型 | 起点 | 终点 | 说明 | 当前数量 |
|---|---|---|---|---:|
| `DIRECTED` | `Person` | `Movie` | 影人执导电影 | 138,660 |
| `ACTED_IN` | `Person` | `Movie` | 影人参演电影，关系上保留演员顺序属性 | 1,040,953 |
| `HAS_GENRE` | `Movie` | `Genre` | 电影所属类型 | 339,613 |
| `IN_REGION` | `Movie` | `Region` | 电影制片地区 | 216,865 |
| `IN_LANGUAGE` | `Movie` | `Language` | 电影语言 | 210,180 |
| `HAS_CONTENT_TYPE` | `Movie` | `ContentType` | 电影内容形式 | 192,032 |
| `IN_YEAR_BUCKET` | `Movie` | `YearBucket` | 电影所属年代分桶 | 191,294 |

以上关系总数为 2,329,597，构成当前 ETL 主图谱关系规模。扩展维度关系包括 `IN_REGION`、`IN_LANGUAGE`、`HAS_CONTENT_TYPE` 和 `IN_YEAR_BUCKET`，总数为 810,371。该 Schema 既保留了电影与影人之间的核心关系，也补充了地区、语言、内容形式和年代等可用于筛选、统计和推荐的结构化维度。

在 Schema 设计上，导演和演员没有拆分为不同节点类型，而是统一使用 `Person` 节点，再通过 `DIRECTED` 和 `ACTED_IN` 关系区分职责。这样可以避免同一影人在不同身份下形成重复实体，也便于查询某位影人的全部作品和合作者。类型、地区、语言等字段从电影属性拆分为独立节点后，可以支持类型分布、地区或语言相关统计，以及基于共同实体的推荐解释。

## 3.4 后端服务设计

后端服务采用 FastAPI 实现，整体按入口层、路由层、服务层、数据库连接层和算法层组织。

入口层位于 `db-backend/app/main.py`。应用创建时配置 CORS 中间件、请求日志中间件和健康检查接口，并在生命周期启动阶段初始化 MySQL 连接池和 Neo4j Driver。启动阶段还会预热推荐图谱缓存和 KG-Embed 工件，使推荐接口在运行时能够复用图谱元数据和已有嵌入结果。

路由层负责定义对外 API。系统当前主线包含认证、用户行为、电影、影人、图谱、统计、推荐和图片代理模块，具体设计如表 3-4 所示。

**表 3-4 后端服务模块设计表**

| 模块 | 路由文件 | 接口前缀 | 主要职责 |
|---|---|---|---|
| 系统 | `app/main.py` | `/health` | 健康检查 |
| 认证 | `app/routers/auth.py` | `/api/auth` | 用户注册、登录、登出、刷新 Token、当前用户信息 |
| 用户行为 | `app/routers/users.py` | `/api/users` | 喜欢、想看、评分、用户画像分析、用户画像图谱 |
| 电影 | `app/routers/movies.py` | `/api/movies` | 电影搜索、类型列表、高分榜、筛选、详情、演职员 |
| 影人 | `app/routers/persons.py` | `/api/persons` | 影人搜索、详情、作品、合作者 |
| 图谱 | `app/routers/graph.py` | `/api/graph` | 电影图、影人图、全局概览、最短路径、共同电影 |
| 统计 | `app/routers/stats.py` | `/api/stats` | 类型、年代、评分、合作网络、趋势和散点统计 |
| 推荐 | `app/routers/recommend.py` | `/api/recommend` | 个性化推荐、推荐解释、算法列表、离线评估报告 |
| 代理 | `app/routers/proxy.py` | `/api/proxy` | 图片代理 |

服务层位于 `app/services/`，主要封装电影、影人、图谱和用户行为等业务查询逻辑。路由函数通过依赖注入获取 MySQL 连接、Neo4j Session 或当前登录用户，再调用服务层函数完成具体查询。该结构可以减少路由函数中的复杂查询逻辑，使接口定义和业务实现保持相对清晰。

数据库连接层由 `app/db/mysql.py` 和 `app/db/neo4j.py` 实现。MySQL 连接层使用 `DBUtils` 连接池管理连接，读取 `DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASS`、`DB_NAME` 和 `DB_POOL_SIZE` 等配置；Neo4j 连接层使用 Neo4j 官方 Driver，并通过 `NEO4J_URI`、`NEO4J_USER`、`NEO4J_PASS` 和连接池大小配置创建驱动。配置项统一由 `pydantic-settings` 从环境变量或 `.env` 文件读取。

算法层位于 `app/algorithms/`，通过统一注册表维护可用推荐算法。推荐路由根据请求参数选择算法，默认使用 `cfkg`。当用户新增或删除评分、喜欢和想看记录时，用户行为路由会通知推荐运行时失效相关算法状态，使后续推荐尽量反映新的用户行为。

## 3.6 推荐模块总体设计

推荐模块连接用户行为数据、电影内容特征和知识图谱结构，是系统个性化服务的核心部分。推荐输入主要包括用户评分、喜欢和想看行为；候选电影及其属性来自 MySQL 和 Neo4j；推荐输出通过后端 `/api/recommend/personal` 接口返回给前端推荐页面，并通过 `/api/recommend/explain` 接口提供解释图和理由。

系统当前注册了五类推荐算法，如表 3-6 所示。

**表 3-6 推荐算法总体设计表**

| 算法标识 | 类型 | 输入信号 | 设计定位 |
|---|---|---|---|
| `content` | 基于内容的推荐 | 电影类型、地区、年代等内容特征 | 作为浅层内容基线，适合用户行为较少时提供基础候选 |
| `item_cf` | 基于物品的协同过滤 | 用户评分矩阵和物品相似度 | 利用群体行为发现相似电影 |
| `kg_path` | 基于知识图谱路径的推荐 | 用户正反馈电影与候选电影之间的图谱路径 | 利用共同导演、演员、类型等结构关系生成候选和解释 |
| `kg_embed` | 基于知识图谱嵌入的推荐 | 图谱三元组和用户正反馈信号 | 将电影、影人和关系映射为向量表示，用于结构相似度排序 |
| `cfkg` | 融合推荐主链路 | `kg_embed` 候选与 `item_cf` 辅助信号 | 综合图谱结构和协同过滤结果，作为系统默认推荐链路 |

推荐调用流程可以概括为四步。第一，前端推荐页面发起请求，携带算法名称、推荐数量和需要排除的电影 ID。第二，后端根据算法名称从注册表中获取推荐器实例，并根据当前用户 ID 调用对应算法。第三，算法根据用户评分、喜欢和想看行为生成候选电影及分数，后端补充电影详情后返回前端。第四，用户查看某个推荐结果解释时，后端根据目标电影和用户历史正反馈电影构造解释图，返回共同导演、共同演员、同类类型等匹配实体和路径理由。

推荐模块还包含运行时控制设计。后端为每类算法维护实例缓存和任务槽位，推荐计算超时或异常时会替换算法运行时状态；用户评分或偏好变化后，相关算法缓存会被清理。该设计的目的在于让推荐结果尽量反映用户最新行为，并避免单次推荐任务长期占用后端资源。由于当前系统主要在本地开发和验收环境中运行，本文将评价范围限定在推荐链路可用性、用户行为变更后的状态更新和接口调用边界上。

从总体设计角度看，`cfkg` 是系统默认推荐链路，但它并不替代其他算法。`content`、`item_cf`、`kg_path` 和 `kg_embed` 既可以作为推荐效果对比方法，也可以为融合链路提供不同类型的候选和解释依据。通过统一算法注册表和统一推荐接口，前端可以在推荐中心切换算法，后端也可以在保持接口稳定的前提下继续优化算法实现。
