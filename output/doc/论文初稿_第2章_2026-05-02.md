# 第 2 章 相关技术与需求分析

本章在第 1 章研究背景的基础上，结合系统实际采用的技术栈和已实现功能，对电影推荐系统的相关技术、使用场景、功能需求和非功能需求进行分析。系统当前主线围绕电影数据采集、关系型数据库存储、知识图谱构建、前后端分离服务、图谱可视化和个性化推荐展开，面向游客和注册用户两类使用场景提供电影检索、详情浏览、图谱探索、统计分析、用户行为记录、推荐结果展示与推荐解释等功能。

## 2.1 相关技术

### 2.1.1 Playwright 数据采集技术

Playwright 是一种面向现代浏览器的自动化工具，能够驱动 Chromium、Firefox 和 WebKit 等浏览器执行页面访问、元素定位、点击、等待和数据抽取等操作。与只请求静态 HTML 的采集方式相比，Playwright 更适合处理页面存在动态渲染、异步加载或交互行为的场景。本系统使用 Playwright 采集电影和影人数据，并结合任务状态、失败标记和断点恢复机制控制采集过程，使采集程序能够在中断后继续处理未完成任务。

在本课题中，数据采集不是孤立的辅助步骤，而是后续知识图谱构建和推荐实验的数据来源。采集结果首先进入 MySQL 原始业务表，再经过字段清洗和质量分析后导入 Neo4j。由于外部页面结构和网络状态可能变化，采集流程需要关注任务幂等、错误记录和可恢复性，避免少量异常页面影响整体数据处理。

### 2.1.2 MySQL 关系型数据库

MySQL 是常用的关系型数据库管理系统，适合存储结构明确、事务边界清晰、需要按表维护的数据。本系统将电影条目、影人信息、采集辅助数据、用户账号、用户会话、用户评分和用户偏好等数据存储在 MySQL 中。关系型数据库的优势在于表结构清晰、查询条件稳定，并且便于通过主键、外键或索引维护业务数据的一致性。

在系统设计中，MySQL 主要承担两类职责。第一，保存采集后的原始电影和影人数据，作为数据清洗、质量分析和图谱 ETL 的基础。第二，保存用户侧业务数据，包括注册用户、登录会话、评分记录、喜欢和想看等行为记录。这些行为数据一方面用于个人中心和用户画像展示，另一方面也作为个性化推荐算法的重要输入。通过将原始业务数据和用户行为数据保存在 MySQL 中，系统能够保留可回溯的数据来源，并为后续统计和推荐提供稳定数据支撑。

### 2.1.3 Neo4j 图数据库与知识图谱

知识图谱以实体和关系为核心组织数据，适合表达电影领域中多类型实体之间的关联。Neo4j 是基于属性图模型的图数据库，节点可以表示电影、影人、类型、地区、语言、内容形式和年代分桶等实体，关系可以表示执导、参演、所属类型、所属地区、语言、内容形式和年代分桶等连接。相比只依赖关系表连接查询，图数据库在多跳关系查询、路径分析和关联可视化方面更直观。

本系统使用 Neo4j 承载电影知识图谱。图谱 Schema 以 `Movie`、`Person`、`Genre`、`Region`、`Language`、`ContentType` 和 `YearBucket` 七类节点为主体，并围绕 `DIRECTED`、`ACTED_IN`、`HAS_GENRE`、`IN_REGION`、`IN_LANGUAGE`、`HAS_CONTENT_TYPE` 和 `IN_YEAR_BUCKET` 等关系组织电影领域知识。该图结构既支持电影详情和影人详情中的关联查询，也支持电影图谱、影人图谱、全局概览、最短路径和共同电影等图谱探索功能。知识图谱相关研究表明，结构化实体关系能够为推荐系统提供辅助语义和解释依据[1][7][C1]，这也是本系统引入 Neo4j 的主要原因。

### 2.1.4 FastAPI 后端服务框架

FastAPI 是基于 Python 类型标注的 Web API 框架，支持请求参数校验、依赖注入、自动生成接口文档和异步任务处理等能力。本系统后端使用 FastAPI 连接 MySQL 与 Neo4j，并向前端提供统一的 REST 风格接口。后端路由按业务职责划分为认证、用户行为、电影、影人、图谱、统计、推荐和图片代理等模块，便于按功能边界组织代码。

在系统中，FastAPI 主要承担三方面工作。第一，封装数据查询服务，将 MySQL 中的电影、影人和用户行为数据以及 Neo4j 中的图谱数据以接口形式提供给前端。第二，实现认证与用户行为接口，包括注册、登录、登出、Token 刷新、当前用户信息、评分、喜欢、想看、用户画像和用户画像图谱等功能。第三，提供推荐算法调用和推荐解释接口，将后端算法输出转化为前端可展示的数据结构。通过依赖注入管理数据库连接和当前用户信息，后端能够保持较清晰的接口边界。

### 2.1.5 Vue3 前端框架与组件生态

Vue3 是面向前端界面构建的渐进式 JavaScript 框架，适合实现组件化页面和响应式交互。本系统前端采用 Vue3 与 Vite 构建项目，使用 Vue Router 组织页面路由，使用 Pinia 管理用户认证状态，使用 Axios 封装后端 API 请求，使用 Element Plus 构建表单、按钮、抽屉、分页和提示等常用交互组件。前端页面包括首页、搜索页、电影库筛选页、推荐页、电影详情页、影人详情页、图谱页面、统计页面、登录注册页面和个人中心页面。

前端与后端通过 `/api` 前缀进行接口通信。Axios 请求拦截器负责为已登录用户附加 Bearer Token，响应拦截器负责处理 401 状态下的 Token 刷新和全局错误提示。路由层面对个人中心等登录后功能设置访问控制，未登录用户访问受保护页面时会跳转到登录页。该设计能够将公开浏览功能和登录后用户行为功能区分开，符合系统当前面向游客和注册用户两类场景的需求。

### 2.1.6 ECharts 图谱与统计可视化

ECharts 是常用的数据可视化库，支持折线图、柱状图、饼图、散点图、关系图等多种图表形式。本系统使用 ECharts 展示电影知识图谱、统计看板和部分推荐解释信息。对于电影和影人详情，图谱组件可以展示目标节点及其关联影人、类型等节点；对于全局图谱探索，前端可以展示从 Neo4j 查询得到的节点和边；对于统计页面，前端可以展示类型分布、年代分布、评分分布、合作网络和评分票数散点图等结果。

可视化的作用不只是美化页面，更重要的是降低用户理解复杂关系的成本。电影知识图谱包含大量节点和关系，如果只以文本或列表方式呈现，用户难以直观看到电影与影人、类型、地区之间的连接。通过图谱和统计图表展示，系统能够把结构化数据转化为更容易浏览和比较的信息，从而支撑电影探索和推荐解释。

### 2.1.7 推荐算法基础

推荐系统通常根据用户行为、物品属性和外部知识生成候选物品并排序。基于内容的推荐利用电影类型、地区、年代等内容特征计算相似性，适合在用户行为较少时提供基础结果；基于物品的协同过滤利用用户评分或偏好行为计算物品之间的相似性，能够从群体行为中发现用户可能感兴趣的电影；基于知识图谱路径的推荐利用电影、影人、类型等实体之间的路径关系生成推荐依据，具有较好的解释性；基于知识图谱嵌入的推荐将实体和关系映射到向量空间，通过向量相似度刻画结构关联；融合推荐则将多类信号组合起来，以获得更稳健的排序结果[1][3][6][C5][C6]。

本系统实现的推荐算法包括 `content`、`item_cf`、`kg_path`、`kg_embed` 和 `cfkg` 五类。其中 `content` 侧重电影元数据特征，`item_cf` 侧重用户行为相似性，`kg_path` 侧重结构路径和解释，`kg_embed` 侧重图谱实体表示，`cfkg` 则融合知识图谱嵌入和物品协同过滤结果作为主推荐链路。论文后续章节将结合实现过程和离线评测结果进一步分析各算法的输入、候选生成、排序方式和效果表现。

## 2.2 用户角色分析

系统当前面向两类使用场景：游客场景和注册用户场景。游客可以在不登录的情况下浏览公开电影数据、搜索电影或影人、查看详情、使用图谱探索和统计分析功能；注册用户在游客能力基础上，还可以登录系统，记录评分、喜欢和想看行为，查看个人中心、用户画像、个性化推荐和推荐解释。

游客场景强调公开数据查询和图谱探索。用户进入系统后，可以通过首页了解电影数据概况和推荐入口，通过搜索页查询电影或影人，通过电影库筛选页按类型、内容形式、年份、评分等条件筛选电影，也可以进入电影详情或影人详情页面查看演职员、作品和合作者信息。对于希望理解电影之间关联的用户，系统提供电影关联图、影人关联图、全局图谱概览、最短路径和共同电影查询等图谱探索能力。

注册用户场景强调用户行为记录和个性化服务。用户注册并登录后，可以对电影进行评分，标记喜欢或想看，并在个人中心查看评分和偏好记录。系统根据用户历史行为构建用户画像，并将评分、喜欢和想看等信号作为推荐算法输入，生成个性化推荐结果。推荐页面和推荐解释抽屉能够展示候选电影与用户偏好之间的关系，使推荐结果不仅可浏览，也能够被用户理解。

## 2.3 功能需求

根据系统路由、前端页面和后端接口，本文将当前功能需求划分为公开浏览、检索筛选、详情展示、图谱探索、统计分析、认证与用户行为、个性化推荐和推荐解释八类。

### 2.3.1 公开浏览需求

系统应提供公开首页，使用户能够在进入系统后获得基本的数据概览和电影入口。首页需要展示统计概览、高分电影或推荐入口，帮助用户快速进入搜索、筛选、详情和推荐等页面。该类功能不要求用户登录，属于游客可访问的基础能力。

### 2.3.2 检索与筛选需求

系统应支持电影和影人的关键词搜索。电影搜索需要根据关键词返回匹配电影，并支持分页；影人搜索需要根据姓名关键词返回匹配影人。电影库筛选功能应支持按类型、内容形式、年份范围、最低评分等条件筛选电影，并支持按加权排序、评分或票数等方式排序。检索与筛选功能用于解决用户在大量电影条目中定位目标内容的问题。

### 2.3.3 详情展示需求

系统应提供电影详情和影人详情页面。电影详情需要展示电影名称、年份、评分、地区、语言、类型、剧情简介、封面和演职员等信息，并为登录用户提供评分、喜欢和想看等行为入口。影人详情需要展示影人基本信息、参演或执导作品以及合作者信息。详情展示功能既服务于普通浏览，也为用户后续行为记录和推荐解释提供上下文。

### 2.3.4 图谱探索需求

系统应支持基于 Neo4j 的电影知识图谱探索。电影关联图需要展示指定电影与相关影人、类型等节点的连接；影人关联图需要展示指定影人与作品、合作者之间的连接；全局图谱概览需要从图数据库中抽取部分节点和边展示整体关系；路径查询需要在两个节点之间寻找最短路径；共同电影查询需要查找两个影人之间共同参与的电影。为避免图谱页面加载过多数据，图谱接口需要支持节点数量、边数量、深度和路径长度等限制参数。

### 2.3.5 统计分析需求

系统应提供电影知识图谱的统计分析能力。统计页面需要展示电影数量、影人数量、类型数量和关系数量等总体统计，并进一步提供类型分布、年代分布、评分分布、参演最多演员、执导最多导演、合作网络、类型共现、类型年代趋势、评分年代变化和评分票数散点图等视图。统计分析功能有助于从整体角度理解数据集结构和电影领域分布特征。

### 2.3.6 认证与用户行为需求

系统应支持用户注册、登录、登出、刷新 Token 和获取当前用户信息。登录后，用户可以添加或取消喜欢、想看等偏好记录，也可以创建、更新、查询或删除电影评分。系统还应提供用户画像分析和用户画像图谱，帮助用户了解自身观影偏好。对于需要登录的个人中心页面，前端路由应进行访问控制；对于需要身份的接口，后端应根据 Token 获取当前用户。

### 2.3.7 个性化推荐需求

系统应根据注册用户的评分、喜欢和想看等行为生成个性化推荐结果。推荐接口需要支持算法选择、推荐数量、排除已交互电影等参数，使用户能够查看不同算法或主链路推荐的结果。系统实现的推荐算法包括基于内容、基于物品协同过滤、基于知识图谱路径、基于知识图谱嵌入和 CFKG 融合推荐。推荐功能应与用户行为模块联动，当用户新增或删除评分、喜欢和想看记录时，推荐运行时缓存需要相应失效，以减少旧行为对推荐结果的影响。

### 2.3.8 推荐解释需求

系统应提供推荐解释能力，使用户能够理解推荐结果与自身偏好之间的关系。推荐解释可以基于目标电影与用户历史正反馈电影之间的共同导演、共同演员或共同类型生成，也可以在缺少重合证据时展示目标电影自身的导演、演员和类型上下文。前端推荐解释抽屉需要展示解释图中的节点、边、匹配实体和路径理由，帮助用户从图谱角度理解候选电影的推荐依据。

## 2.4 非功能需求

### 2.4.1 可维护性需求

系统应按照模块边界组织代码，降低后续修改成本。后端需要按路由层、服务层、数据库连接层和算法层划分职责，认证、用户行为、电影、影人、图谱、统计和推荐接口分别维护；前端需要按页面、API 封装、状态管理和通用组件组织代码，使页面逻辑和接口调用保持清晰边界。推荐算法通过统一注册表管理，有助于在不改变前端调用方式的情况下扩展或替换算法实现。

### 2.4.2 数据一致性与可追溯性需求

系统的数据链路应保持来源清晰。电影和影人原始数据首先进入 MySQL，清洗后再导入 Neo4j，避免直接在图数据库中修改原始来源。用户账号、会话、评分和偏好等业务数据保存在 MySQL 中，推荐模块按用户行为读取输入数据。对于电影知识图谱，节点和关系应由 ETL 流程统一生成，并通过唯一约束和索引减少重复节点。这样可以保证数据在采集、清洗、建图、查询和推荐之间具有较好的可追溯性。

### 2.4.3 采集与处理可靠性需求

数据采集过程应支持任务状态、失败标记和断点恢复。当采集过程中出现页面异常、网络波动或字段缺失时，系统应记录失败状态，并允许后续继续处理未完成或失败任务。数据清洗阶段需要对空字符串、异常评分、票数字段和缺失字段进行统一处理，避免不规范字段直接进入图谱构建和推荐算法。该需求主要保障数据基础的稳定性，而不是追求采集速度指标。

### 2.4.4 图谱查询控制需求

图谱查询容易因为节点和边数量过大而影响页面展示和接口响应。系统应在图谱接口中设置合理的查询限制，例如电影图和影人图的深度限制、节点数量限制、边数量限制和超时时间限制，全局概览也需要控制种子节点、节点总数和边总数。最短路径查询应限制最大跳数，避免无界路径搜索。通过这些限制，系统可以在保证图谱探索可用性的同时降低单次查询的资源占用。

### 2.4.5 认证与访问控制需求

系统应区分公开访问功能和登录后功能。搜索、筛选、详情、图谱探索和统计分析属于公开浏览能力，游客可以直接访问；个人中心、评分、喜欢、想看、用户画像和个性化推荐等功能需要用户登录后使用。前端路由通过 `requiresAuth` 标记保护个人中心页面，Axios 请求拦截器在存在访问令牌时自动附加 Bearer Token，响应拦截器在访问令牌失效时尝试刷新 Token。后端通过当前用户依赖获取登录用户信息，保证用户行为数据归属于具体账号。

### 2.4.6 推荐服务可用性需求

推荐服务需要在用户行为变化后尽量使用新的偏好数据。系统在用户添加或删除偏好、创建或删除评分后，会触发推荐运行时状态失效，使受影响算法在后续调用时重新加载相关状态。推荐接口还应设置推荐任务超时时间和单算法任务槽位限制，避免单次推荐计算长期占用后端资源。由于当前证据主要来自本地开发和验收环境，本文将评价范围限定在推荐链路可用性、行为变更后的状态更新和接口调用边界上。

### 2.4.7 前端交互与错误处理需求

前端页面应保证基本交互一致性。对于搜索、筛选和推荐结果，页面应支持加载状态、错误提示和空结果提示；对于需要分页的列表，应通过页码和每页数量控制返回结果；对于图谱和统计图表，应在接口返回后再渲染图形，避免空数据导致页面异常。全局 API 封装需要统一处理请求基础路径、超时时间、Token 附加、Token 刷新和错误提示，使页面层不必重复实现底层请求逻辑。

综上，本系统的需求重点是在电影数据、知识图谱、用户行为和推荐算法之间建立稳定的应用链路。游客侧功能负责公开数据查询和图谱探索，注册用户侧功能负责行为记录、个人画像和推荐服务，非功能需求则围绕数据可追溯、接口可维护、图谱查询可控、认证访问清晰和推荐服务可用展开。
