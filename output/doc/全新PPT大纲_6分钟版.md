# 中期汇报全新 PPT 大纲（6 分钟版）

这份大纲不继承旧 `pptx` 结构，直接按“老师最容易听懂、最不容易追着问崩”的顺序重新组织。总页数固定为 `8` 页。

## 整体叙事主线

先讲“为什么做”，再讲“系统怎么搭起来”，再讲“现在已经做成什么”，最后讲“结果如何、问题在哪、下一步做什么”。

总的逻辑顺序是：

`问题 -> 方法 -> 系统 -> 数据与图谱 -> 推荐设计 -> 已完成成果 -> 实验与问题 -> 收尾`

## Slide 1 封面与课题定位

### 核心结论

这不是单点算法实验，而是一个从数据采集到推荐展示的端到端系统。

### 页面内容

- 标题：`基于知识图谱的电影推荐系统`
- 副标题：`从数据采集到推荐展示的端到端系统`
- 汇报人信息
- 4 个开场指标：
  - `135,990` 电影记录
  - `58` 后端接口
  - `5` 推荐算法
  - `13` 前端页面

### 视觉重点

- 深色封面
- 大标题 + 指标标签
- 右侧做抽象系统网络图或流水线图，不堆正文

### 建议时长

`20-30s`

### 口头重点

“我这个项目不是只做图谱展示，而是把采集、图谱、推荐、评测和前端串起来做成了一个完整系统。”

## Slide 2 研究问题与课题目标

### 核心结论

做图谱不是为了图谱本身，而是为了解决推荐中的稀疏、冷启动和解释性问题。

### 页面内容

左侧讲 3 个问题：

- 评分数据稀疏
- 新用户冷启动明显
- 推荐结果缺少解释

右侧讲 3 个目标：

- 构建电影知识图谱
- 设计多算法推荐方法
- 完成可演示系统

底部一句总结：

`图谱是结构化知识底座，推荐与解释才是最终落地场景。`

### 视觉重点

- 左右双栏卡片
- 左边用偏暖色表示问题，右边用偏绿色表示目标

### 建议时长

`40-50s`

### 口头重点

“我的课题不是单纯做数据库建模，而是想把图谱真正用于电影推荐场景。”

## Slide 3 总体技术路线与系统架构

### 核心结论

系统采用“双库 + 分层”的结构：MySQL 管原始和行为数据，Neo4j 管图结构和关系查询。

### 页面内容

主流程图：

`Playwright 爬虫 -> MySQL -> 数据清洗 -> Neo4j -> FastAPI -> Vue`

下方再拆两块：

- `MySQL`：
  - 原始电影/影人数据
  - 用户认证与行为数据
- `Neo4j`：
  - Movie / Person / Genre 等节点
  - 路径查询、图谱展示、图谱推荐特征

右侧补一个四层结构：

- 采集层
- 处理层
- 服务层
- 展示层

### 视觉重点

- 中间做一条清晰的流水线
- 下方做 MySQL / Neo4j 分工对照

### 建议时长

`50-60s`

### 口头重点

“双库并不是重复存储，而是因为关系图查询和业务型数据管理适合不同数据库。”

## Slide 4 数据与图谱构建

### 核心结论

图谱来自真实采集数据，不是手工构造；图谱设计同时服务于查询展示和推荐算法。

### 页面内容

左侧数据规模：

- `135,990` Movies
- `198,067` Persons
- `32` Genres
- `1,443,933` 预估关系规模

左下角补数据质量要点：

- `30.52%` 电影缺评分
- `36.76%` 电影缺剧情简介
- 因此需要专门的数据清洗流程

右侧图谱模式：

- 节点：
  - `Movie`
  - `Person`
  - `Genre`
  - `Region`
  - `Language`
  - `ContentType`
  - `YearBucket`
- 关系：
  - `DIRECTED`
  - `ACTED_IN`
  - `HAS_GENRE`
  - `IN_REGION`
  - `IN_LANGUAGE`
  - `HAS_CONTENT_TYPE`
  - `IN_YEAR_BUCKET`

底部补 ETL 链路：

`crawl -> clean_pipeline -> etl_to_neo4j -> graph query`

### 视觉重点

- 左边做统计卡片
- 右边做简化版 schema 图

### 建议时长

`45-55s`

### 口头重点

“我在建模时没有只停留在电影、演员、导演三个节点，还增加了地区、语言、年份分桶等结构节点，为后面的图谱嵌入方法提供更多关系信息。”

## Slide 5 推荐系统设计

### 核心结论

推荐部分不是单一算法，而是“基线方法 + 图谱方法 + 融合主链路”的组合系统。

### 页面内容

上方先讲用户信号：

- 评分
- 喜欢
- 想看

中间讲 5 个算法的角色：

- 基线方法：
  - `content`
  - `item_cf`
- 图谱方法：
  - `kg_path`
  - `kg_embed`
- 主链路：
  - `cfkg`

右侧讲两项系统能力：

- 冷启动判断：有效信号阈值 `3`
- 推荐解释：返回 `reason_paths / nodes / edges`

底部结论：

`CFKG 是当前主链路，但不宣称它是当前最优，而是重点优化对象。`

### 视觉重点

- 做成“信号输入 -> 算法分支 -> 融合输出”的结构图
- 用单独卡片展示冷启动和解释能力

### 建议时长

`60-70s`

### 口头重点

“这 5 种算法不是并列摆设，而是分别承担基线对照、显式解释、潜在语义和融合推荐的不同角色。”

## 设计原则

- 只保留 8 页，不扩页
- 每页只讲 1 个核心结论
- 以图、卡片、结构图为主，不做高密度大段文字
- 第 1 页和第 8 页用深色，其余内容页用浅底，形成“深-浅-深”的节奏
- 不把 `CFKG` 说成当前最优
- 不用“功能计划”充数，只讲已经完成和下一步明确事项

## 演讲节奏建议

- 第 1 页：开场定题
- 第 2-5 页：主体方法
- 第 6 页：中期完成情况
- 第 7 页：结果与问题
- 第 8 页：收束

这一套结构的优点是：老师即使中途打断提问，你也始终站在“目标清楚、系统已做成、问题也知道”的主动位置上。
