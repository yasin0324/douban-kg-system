## 摘要

随着互联网影视内容的持续增长，电影信息已经呈现出数据规模大、关系复杂和语义关联丰富等特点。传统的关系型数据展示方式能够满足基础检索需求，但在多实体关联查询、图结构探索和个性化推荐等场景下存在表达能力不足的问题。豆瓣电影平台积累了大量电影、导演、演员、评分和类型信息，为构建垂直领域知识图谱提供了良好的数据基础。基于此，本文围绕豆瓣电影数据，设计并实现了一套集数据采集、数据治理、知识图谱构建、后端服务、前端展示和推荐评估于一体的电影推荐系统。

本文首先基于 Playwright 设计电影与影人采集程序，并结合任务锁、代理切换、失败重试和监控脚本提升采集稳定性；随后利用 Python 与 Pandas 对原始数据进行清洗、缺失分析和格式统一，并通过 ETL 脚本将 MySQL 中的电影、影人及扩展属性导入 Neo4j，构建 `Movie`、`Person`、`Genre`、`Region`、`Language`、`ContentType` 和 `YearBucket` 等节点及多类语义关系组成的豆瓣电影知识图谱。在系统实现方面，后端基于 FastAPI 构建电影检索、影人查询、图谱探索、统计分析、用户认证、用户行为和推荐接口；前端基于 Vue3、Vite、Element Plus 与 ECharts 实现首页、搜索页、电影详情页、影人详情页、图谱可视化页、推荐页、统计页和个人中心等功能，形成从数据层到展示层的完整闭环。

在推荐实验方面，本文实现了基于内容的推荐、基于物品的协同过滤、基于知识图谱路径的推荐以及基于知识图谱嵌入的推荐四类算法，并采用 validation/test sampled leave-one-out 协议进行离线评估。实验结果表明，在 `Recall@10` 指标上，`item_cf` 达到 `0.8335 ± 0.0025`，表现最佳；`kg_embed` 达到 `0.5801 ± 0.0210`，在引入图谱语义信息和可解释性方面具有较好的综合表现。系统同时完成了图谱服务、推荐算法与评测模块的关键测试验证，相关 16 项测试均已通过。研究结果说明，知识图谱能够有效支撑电影信息组织、关系探索与推荐分析，所实现系统具有一定的工程实用价值和论文研究价值。

**关键词：** 豆瓣电影；知识图谱；推荐系统；Neo4j；FastAPI；Vue3

## Abstract

With the rapid growth of online movie content, film information has become large-scale, highly connected, and semantically rich. Traditional relational data organization can satisfy basic retrieval needs, but it is less effective for multi-entity association queries, graph exploration, and personalized recommendation. Douban Movie provides abundant data on films, directors, actors, ratings, and genres, which offers a solid foundation for building a domain-specific knowledge graph. Based on this background, this thesis designs and implements a movie recommendation system integrating data collection, data governance, knowledge graph construction, backend services, frontend visualization, and recommendation evaluation.

This work first develops movie and person crawlers with Playwright and improves crawling stability through task locking, proxy switching, retry strategies, and monitoring scripts. Then Python and Pandas are used to clean raw data, analyze missing fields, and normalize formats. An ETL pipeline imports structured data from MySQL into Neo4j and builds a Douban movie knowledge graph with nodes including `Movie`, `Person`, `Genre`, `Region`, `Language`, `ContentType`, and `YearBucket`, together with multiple semantic relationships. On the system side, a FastAPI backend is implemented to provide movie search, person query, graph exploration, statistical analysis, authentication, user behavior, and recommendation APIs. A Vue3 frontend with Vite, Element Plus, and ECharts is also developed to support the home page, search page, movie detail page, person detail page, graph visualization page, recommendation page, statistics page, and profile page, forming a complete end-to-end workflow.

For recommendation experiments, four algorithms are implemented, including content-based recommendation, item-based collaborative filtering, knowledge-graph path-based recommendation, and knowledge-graph embedding recommendation. A validation/test sampled leave-one-out protocol is adopted for offline evaluation. Experimental results show that `item_cf` achieves the best `Recall@10` of `0.8335 ± 0.0025`, while `kg_embed` reaches `0.5801 ± 0.0210` and provides a better balance between graph semantics and explainability. In addition, key tests for graph services, recommendation algorithms, and evaluation modules are executed successfully, with 16 related tests passing. The results indicate that the knowledge graph can effectively support movie information organization, relationship exploration, and recommendation analysis, and the implemented system has both engineering and academic value.

**Keywords:** Douban Movie; knowledge graph; recommender system; Neo4j; FastAPI; Vue3

## 第一章 绪论

### 1.1 研究背景与意义

近年来，电影内容生产和传播速度不断提升，用户在观看、评分、收藏和搜索过程中持续生成大量行为数据。对于电影平台而言，如何从庞杂的影视信息中组织实体关系、提升检索效率并给出更符合用户兴趣的推荐结果，已经成为一个兼具研究价值和应用价值的问题。传统关系数据库更适合处理结构清晰、关联层级较浅的数据，而电影场景中的导演、演员、类型、地区、语言和用户偏好之间往往存在复杂的多跳关系，仅依赖二维表结构不利于进行语义组织和图式分析。

知识图谱作为一种面向语义关联的知识组织方式，能够将实体、属性和关系统一表示为图结构，为复杂关系查询、路径探索、可视化展示和推荐解释提供支撑。在电影领域，知识图谱可以将电影、影人、类型等核心实体构造成可计算的关系网络，为“某位导演的长期合作演员有哪些”“两位影人之间通过哪些影片产生关联”“某部电影为何被推荐给当前用户”等问题提供更自然的建模方式。

豆瓣电影数据具有题材丰富、结构较完整、用户活跃度高等特点，是构建垂直电影知识图谱的理想对象。围绕豆瓣电影构建知识图谱，不仅能够增强电影信息的结构化表达能力，还可以为个性化推荐和关系可视化提供统一的数据基础。从工程角度看，这一课题涉及数据采集、数据清洗、双库存储、后端接口设计、前端可视化和推荐算法评测等多个环节，具有较强的综合实践价值；从研究角度看，该课题能够验证知识图谱在电影推荐场景中的应用效果，并形成可分析、可比较的实验结果。因此，开展基于豆瓣电影知识图谱的推荐系统研究具有较为明确的理论意义和现实意义。

### 1.2 国内外研究现状

在知识图谱研究方面，国外较早将知识图谱应用于搜索增强、语义组织和智能推荐等场景，图数据库、图嵌入和关系推理技术已经形成较成熟的方法体系。通用知识图谱的发展推动了实体对齐、关系表示和多源知识融合等研究方向，也为垂直领域图谱构建提供了可借鉴的方法。在电影领域，围绕 IMDB 等影视数据形成的结构化知识库研究说明，影视实体之间的导演、演员、类型和作品合作关系非常适合采用图结构进行表达。

在推荐系统研究方面，协同过滤和基于内容的推荐是最经典的两类方法。协同过滤通过挖掘用户行为中的共现规律生成推荐结果，在用户反馈较充分时通常具有较好效果；基于内容的推荐则依赖物品属性建模，能够在一定程度上缓解协同过滤的冷启动问题。随着知识图谱研究的发展，越来越多的方法开始尝试将实体关系、路径特征和图嵌入引入推荐场景，以增强推荐模型对语义关联的利用能力，同时提升结果解释性。

国内相关研究更多集中在垂直领域知识图谱构建、图谱问答、图谱增强推荐等方向，面向电影领域的研究通常会结合影片属性、演员关系和用户行为展开系统实现与实验分析。总体来看，现有研究已经证明知识图谱能够改善推荐系统对复杂语义关系的表达能力，但在真实工程环境中，仍需要解决采集稳定性、数据质量控制、双数据库协同和实验评估规范化等问题。基于此，本文以完整工程闭环为目标，从采集、治理、建图、服务到推荐评估展开系统化实现。

第四，完成推荐算法设计与离线评估。本文实现四类推荐算法，并采用统一的 sampled leave-one-out 协议进行离线实验，分析不同方法在准确率、多样性、覆盖率和时间开销上的差异，验证知识图谱在推荐任务中的作用。

## 第二章 关键技术与理论基础

### 2.1 Playwright 采集技术

本系统的数据采集模块采用 Playwright 实现，而非传统的 Scrapy 框架。其原因在于豆瓣电影页面在实际访问过程中具有较明显的动态特征和反爬限制，单纯依赖静态请求更容易受到访问限制。Playwright 能够驱动真实浏览器环境，模拟较接近真实用户的页面访问过程，并支持页面等待、脚本执行、代理配置和上下文隔离，因而更适合构建电影和影人详情页的稳定采集逻辑。

在具体实现中，电影采集模块采用并发 worker 模式，通过数据库状态字段实现任务锁定，避免多线程或多机器重复抓取同一电影；影人采集模块则结合代理池与异步 Playwright 机制，提高长时间运行时的稳定性。为了增强系统鲁棒性，采集链路还加入了陈旧锁清理、失败任务释放、代理切换、启动抖动与请求间隔控制等策略。这些机制使采集模块不再是一次性的实验脚本，而是具备可恢复、可监控和可扩展特征的工程化采集子系统。

### 2.2 MySQL 与 Neo4j 双数据库协同

系统在存储层采用 MySQL 与 Neo4j 双数据库协同方案。MySQL 主要负责原始电影数据、影人数据、用户账号、用户偏好、用户评分和会话信息等结构化记录的保存；Neo4j 则负责电影知识图谱的图结构存储和关系查询。二者的分工体现了“事务型数据与图关系数据分离”的设计思路。

MySQL 的优势在于数据表结构稳定、事务支持完善、适合分页检索与用户行为管理，因此电影和影人的关键词搜索、用户注册登录、评分和偏好记录等功能优先使用 MySQL 完成。Neo4j 更适合处理图遍历、最短路径和多实体关联问题，因此电影详情、影人详情、图谱可视化、常见合作者分析、路径查询和统计分析等模块主要依赖 Neo4j。

双库协同虽然增加了系统设计复杂度，但也提升了整体性能与语义表达能力。具体而言，搜索类接口避免了在图数据库中执行大范围模糊匹配，图谱类接口则避免了关系型数据库中多表复杂连接带来的开销，从而形成较合理的技术分工。

### 2.3 FastAPI 与 Vue3 前后端技术

后端框架选择 FastAPI，主要考虑其类型提示友好、异步支持完善、自动文档能力强且便于组织模块化接口。系统将后端拆分为路由层、服务层和数据访问层：路由层负责参数校验与接口注册，服务层负责业务逻辑实现，数据库连接层负责 MySQL 连接池与 Neo4j 驱动管理。此种分层方式有利于降低接口耦合度，并便于后续测试与维护。

前端采用 Vue3 + Vite 作为基础框架，结合 Vue Router 管理页面路由，Pinia 管理用户认证状态，Element Plus 构建界面组件，ECharts 实现知识图谱和统计图表展示。Vue3 的组合式 API 有助于将推荐逻辑、用户行为和页面展示拆分为更清晰的模块，Vite 则保证了较高的开发效率和构建速度。前后端通过 REST API 进行交互，使系统具备较好的模块边界和后续扩展空间。

### 2.4 知识图谱表示方法

知识图谱通常使用“实体 - 关系 - 实体”或“实体 - 属性 - 值”的方式表示现实世界中的知识对象及其联系。对于电影场景而言，电影和影人是最核心的两类实体，导演、演员、类型、地区、语言、年代分桶等信息则可以通过属性和关系组织成图结构。本文采用属性图模型进行表示，节点不仅具备类型标签，还可附带评分、年份、封面、职业等属性，从而兼顾图关系表示和业务字段存储。

在 Schema 设计方面，本文将电影定义为 `Movie` 节点，将导演和演员统一为 `Person` 节点，通过不同关系区分其角色；将类型定义为 `Genre` 节点，并补充 `Region`、`Language`、`ContentType` 和 `YearBucket` 等由现有字段派生出的结构节点，以支持后续推荐实验中对影片结构特征的使用。该设计既覆盖了前端展示与图谱查询所需的核心语义，也为知识图谱路径推荐和嵌入推荐提供了结构基础。

### 2.5 推荐算法原理

#### 2.5.1 基于内容的推荐

基于内容的推荐算法从电影元数据中抽取特征，并通过比较候选电影与用户兴趣画像之间的相似度生成结果。本文使用电影的类型、地区和年份分桶构成特征向量，再根据用户历史正向电影的加权平均向量计算候选相似度，核心思想可表示为：

\[
score(i, u)=cos(\mathbf{v}_i,\mathbf{p}_u)
\]

其中，\(\mathbf{v}_i\) 表示电影 \(i\) 的特征向量，\(\mathbf{p}_u\) 表示用户 \(u\) 的兴趣画像向量。该方法实现简单、可解释性强，但无法充分利用图谱中的深层语义关系。

#### 2.5.2 基于物品的协同过滤

基于物品的协同过滤通过用户评分矩阵挖掘电影之间的行为相似性。若两部电影经常被相同用户同时高分评价，则认为两者具有较强关联。本文使用余弦相似度度量物品间相似性，并根据用户已喜欢电影对候选电影进行加权打分：

\[
sim(i,j)=\frac{\sum_{u \in U_{ij}} r_{u,i}r_{u,j}}
{\sqrt{\sum_{u \in U_i} r_{u,i}^2}\sqrt{\sum_{u \in U_j} r_{u,j}^2}}
\]

协同过滤在用户行为充分时通常能够获得较好效果，但对评分矩阵稀疏性较敏感，因此在冷启动场景下存在一定局限。

#### 2.5.3 基于知识图谱路径的推荐

基于知识图谱路径的推荐从用户正向电影出发，寻找在导演、演员和类型等关系上与其存在共享实体的候选电影。该方法不仅可以利用一跳关系，还可以进一步结合两跳路径扩展候选范围，并通过实体度惩罚抑制高频热门实体带来的噪声。本文采用基于共享实体的加权打分策略，可概括表示为：

\[
score(i)=\sum_{s \in S_u}\sum_{e \in E(s)\cap E(i)} w_s \cdot w_r \cdot idf(e)
\]

其中，\(S_u\) 为用户正向电影集合，\(E(\cdot)\) 为电影在某类关系上的实体集合，\(w_s\) 表示种子电影权重，\(w_r\) 表示关系类型权重，\(idf(e)\) 用于降低高频实体的影响。该方法较易生成推荐解释，但在复杂语义建模能力上仍有一定限制。

#### 2.5.4 基于知识图谱嵌入的推荐

知识图谱嵌入推荐首先将图谱中的结构三元组导出为训练样本，通过嵌入学习得到电影与实体的低维向量表示，再基于用户兴趣电影在嵌入空间中的分布对候选影片进行排序。本文的嵌入模块采用 TransE 风格的结构训练方式，以结构三元组和逆关系构建训练数据，并基于验证集进行早停。在线推荐阶段，将用户正向电影的加权质心相似度、与单部种子影片的最大相似度以及实体重叠得分进行融合，形成最终推荐分数。该方法能够同时利用图谱结构语义和局部实体证据，在准确率与可解释性之间取得较为均衡的效果。

### 2.6 推荐评估指标

为了客观比较不同算法表现，本文使用 `Recall@K`、`NDCG@K`、`Hit Rate@K`、`Precision@K`、`Coverage@20`、`Diversity@10` 和平均推荐时间等指标。

`Recall@K` 用于衡量目标正样本是否出现在前 \(K\) 个结果中；`Precision@K` 用于衡量前 \(K\) 个结果中命中目标的比例；`Hit Rate@K` 与单正样本场景下的召回率具有一致解释；`NDCG@K` 强调命中位置越靠前贡献越大。覆盖率反映推荐结果覆盖到不同电影的广度，多样性衡量推荐列表内部的差异程度，平均时间则用于反映算法的实际计算开销。这些指标共同构成了准确率、广度、列表质量与效率的综合评估框架。

## 第三章 系统分析与总体设计

### 3.1 系统需求分析

从用户使用角度出发，系统需要满足以下几类需求。第一类是基础浏览需求，即用户能够查询电影和影人信息，查看电影评分、年份、类型、导演、演员及相关详情；第二类是图谱探索需求，即用户能够以电影或影人为中心查看关系网络，并查询两个实体之间的最短路径或共同作品；第三类是用户交互需求，即系统需要支持注册登录、喜欢、想看、评分和个人中心管理；第四类是推荐需求，即系统能够基于用户历史行为给出个性化推荐，并展示一定程度的推荐理由；第五类是系统验证需求，即需要通过统计接口、数据质量分析和离线评估结果证明系统具有一定可用性与实验价值。

从工程实现角度看，上述需求进一步对应为采集层、数据治理层、图谱构建层、服务层和展示层五个部分。系统既要支持结构化数据处理，又要支持图关系查询和推荐实验，因此必须在架构设计阶段就明确各层职责与数据流向。

### 3.2 系统总体架构设计

本文系统采用“数据采集层 - 数据治理层 - 存储层 - 服务层 - 展示层”的分层结构。数据采集层负责从豆瓣页面抓取电影与影人数据；数据治理层负责清洗、统计与 ETL；存储层由 MySQL 和 Neo4j 构成；服务层由 FastAPI 提供统一接口；展示层由 Vue3 前端完成交互与可视化。

`[插图占位：图 3-1 系统总体架构图]`

这种分层设计具有三方面优势。其一，各层关注点清晰，爬虫、ETL、接口和页面可以独立迭代；其二，双数据库分工明确，避免所有问题都堆叠到同一数据库；其三，推荐模块既可以调用图谱结构信息，也可以结合 MySQL 中的用户行为数据，从而形成完整闭环。

### 3.3 数据流程设计

系统的数据流程可以概括为以下步骤：首先从豆瓣页面采集电影和影人信息并写入 MySQL；其次对采集结果进行清洗、缺失分析和格式标准化；然后将符合 Schema 要求的数据以批量方式导入 Neo4j 构建知识图谱；之后由 FastAPI 同时面向 MySQL 和 Neo4j 提供查询、统计和推荐接口；最后由前端页面调用接口完成浏览、图谱展示和推荐结果呈现。

在此过程中，用户评分与偏好数据保存在 MySQL 中，既直接服务于用户中心功能，也作为推荐算法构建用户画像的重要依据。图谱缓存模块会从 MySQL 和 Neo4j 联合加载图谱元信息，用于支撑路径推荐与嵌入推荐的特征准备工作。

## 第四章 系统实现

### 4.1 采集子系统实现

#### 4.1.1 电影采集模块

电影采集模块以 Playwright 同步接口为基础实现，通过多 worker 并发抓取电影详情页面。为避免在分布式或多线程环境下重复抓取同一任务，系统在 `subjects` 表中引入 `crawl_status`、`crawl_locked_at` 和 `crawl_worker` 等字段，采集程序在获取任务时先尝试锁定，再进入页面访问和字段解析流程。若任务执行过程中出现异常，系统会重置状态以便后续重试；若 worker 意外退出导致锁长时间未释放，后台逻辑会将超时任务恢复为待采集状态。

电影页面解析内容包括电影名称、年份、封面、剧情简介、评分、评分人数、地区、语言、导演、演员及演员与导演标识信息等字段。为适应豆瓣页面结构变化，采集模块对关键字段提取保持了较强的容错逻辑，并对不存在页面和字段缺失情况进行统计。

#### 4.1.2 影人采集模块

影人采集模块采用异步 Playwright 和代理池策略。与电影采集相比，影人页面在字段结构和请求节奏上具有一定差异，因此系统单独实现了影人任务表和抓取流程。模块通过动态代理切换、请求节流、Cookie 注入、User-Agent 随机化等方式降低被封禁风险，同时支持从电影中解析出的 `person_id` 集合出发批量补全影人信息。

#### 4.1.3 监控与容错机制

为了提高采集链路可维护性，系统补充了进度检查、监控脚本和锁重置脚本，用于查看当前采集状态、发现失败任务并在必要时快速恢复任务队列。总体上，采集子系统不再仅是单一脚本，而是包含任务调度、状态管理与故障恢复能力的工程化实现。

### 4.2 数据清洗与 ETL 实现

#### 4.2.1 数据清洗实现

原始采集结果存在分隔符不统一、空字符串与空值混用、名称两端空格、评分为零等问题。为此，本文实现了数据清洗脚本，对 `genres`、`regions`、`languages` 等字段统一使用斜杠分隔，对空字符串统一转为 `NULL`，并对电影名和影人名执行去空格处理。此外，还对 `actor_ids` 和 `director_ids` 的格式进行检查，确保后续关系抽取和图谱导入的稳定性。

#### 4.2.2 数据质量分析

#### 4.2.3 ETL 与图谱导入实现

ETL 脚本从 MySQL 中读取电影、影人及扩展字段数据，先在 Neo4j 中创建唯一约束与索引，再利用 `UNWIND` 批量导入节点和关系。导入过程将 `genres`、`regions` 和 `languages` 等多值字段拆分为集合，并生成相应的扩展节点，再依据电影和影人的关系信息建立 `DIRECTED`、`ACTED_IN`、`HAS_GENRE` 等关系。对于电影年份，系统进一步构造 `YearBucket` 节点，将影片归入 `before_1990`、`1990s`、`2000s`、`2010s` 和 `2020s_plus` 等分桶，以便后续推荐实验中使用。

### 4.3 后端服务实现

#### 4.3.1 后端架构与依赖管理

后端采用 FastAPI 实现，入口文件负责初始化 MySQL 连接池、Neo4j 驱动和推荐缓存资源。系统通过依赖注入统一管理 MySQL 连接、Neo4j 会话以及 JWT 鉴权逻辑，使各路由模块可以专注于接口层定义。整体架构围绕“路由 - 服务 - 数据访问”三层组织，既方便职责划分，也便于进行单元测试。

#### 4.3.2 电影与影人查询接口实现

电影搜索接口采用 MySQL 实现，先尝试标题与别名的直接模糊匹配，在未命中时再使用分词结果进行多关键词搜索，从而提升中文搜索体验。电影详情接口则直接从 Neo4j 中读取 `Movie` 节点及其关联的导演、演员和类型信息，并统一转换为前端可用的数据结构。影人详情和合作关系查询等接口与之类似，主要通过 Neo4j 的图模式匹配实现。

#### 4.3.3 图谱与统计接口实现

图谱服务支持以电影或影人为中心的一跳或两跳展开，并允许限制节点和边数量，防止图形过于拥挤。系统会在图谱返回结果中补充节点类型、边类型、数量统计和查询时间等信息，以便前端展示。除此之外，后端还实现了最短路径、共同作品、类型分布、年代分布、评分分布、头部影人统计等多类统计接口，为论文实验分析和前端看板提供数据支持。

#### 4.3.4 用户认证与行为模块实现

用户模块包括注册、登录、刷新令牌、登出、偏好记录、评分记录和行为汇总等功能。系统采用 JWT 进行鉴权，使用 `access_token` 与 `refresh_token` 区分短期访问与长期会话刷新，并对刷新令牌进行哈希存储。用户行为方面，系统支持“喜欢”和“想看”两类偏好，以及 0.5 至 5.0 区间的评分记录。行为汇总接口会统计用户去重后的交互电影数量，用于推荐模块中的冷启动判断。

### 4.4 推荐模块实现

#### 4.4.1 推荐接口与算法注册

推荐模块将四类算法统一注册在算法表中，并通过 `/api/recommend/personal` 暴露个性化推荐接口。调用时，用户可以传入算法名称和推荐数量，后端会根据用户行为构建画像，再调用对应算法生成推荐列表。为防止并发请求导致同一算法实例过载，系统为每类算法维护了独立的并发信号量，并设置统一的推荐超时阈值。

#### 4.4.2 图谱缓存与推荐解释

知识图谱路径推荐和嵌入推荐都需要快速访问电影与实体之间的映射关系，因此系统在启动阶段预热图谱元数据缓存，构建电影画像、倒排索引和结构三元组。推荐解释接口会结合用户正向电影和目标推荐电影的共享导演、演员、类型等实体生成解释图结构，当缺少明显重叠证据时，则退化为基于目标电影上下文的解释结果。这一机制使推荐页面不仅能展示推荐列表，还能展示基础图谱证据。

`[插图占位：图 4-1 推荐流程图]`

#### 4.4.3 四类推荐算法实现

基于内容的推荐从 MySQL 中一次性加载全部电影的类型、地区和年份特征，构建标准化特征矩阵，并根据用户历史正向电影的加权平均特征向量进行余弦相似度排序。

基于物品的协同过滤从用户评分表中加载全局评分矩阵，计算物品间的余弦相似度，并根据用户历史高评分电影对候选电影进行聚合打分。算法在评估场景中显式排除测试目标电影对训练集的影响，以避免数据泄漏。

基于知识图谱路径的推荐利用缓存中的导演、演员和类型倒排索引，寻找与用户正向电影共享实体的候选电影，并支持引入两跳路径和实体度惩罚。该方法的特点是能够直接返回“共同导演”“共同演员”“同类类型”等较直观的解释。

基于知识图谱嵌入的推荐先从图谱结构中导出三元组，再训练 TransE 风格的实体与关系向量，并在推荐阶段综合使用用户嵌入质心、最大种子相似度和实体重叠得分进行融合排序。该方法在结构语义建模能力上强于单纯的一跳路径匹配。

#### 4.4.4 离线评估模块实现

离线评估脚本采用 validation/test sampled leave-one-out 协议。首先从评分大于等于 3.5 的用户行为中选取正向样本，并要求每位用户至少拥有足够的历史交互；然后按照固定随机种子将用户划分为验证集和测试集；对于每个测试用户，保留最后一个正向样本作为目标电影，再从未交互电影中采样 99 个负样本，与目标电影共同构成候选集合。算法会先在验证集上完成参数搜索，再在测试集上输出多 seed 的均值与标准差，并同时生成 JSON 与 Markdown 格式报告。

### 4.5 前端展示实现

前端页面围绕电影浏览、图谱探索和推荐展示展开。首页用于展示电影列表、统计概览和个性化推荐预览；搜索页支持电影与影人的双标签检索；电影详情页展示评分、类型、剧情简介和演职信息，并允许用户执行喜欢、想看和评分操作；影人详情页展示影人信息、作品列表和合作者关系；图谱页基于 ECharts 力导向图展示电影或影人的邻域网络；路径页用于展示两个实体之间的最短路径；推荐页集中展示四类算法的推荐结果和解释抽屉；统计页用于展示图谱总体统计与分布图表；个人中心则用于查看偏好与评分记录。

前端通过组合式逻辑封装推荐请求、重刷历史、反馈行为和冷启动状态判断。当用户交互不足时，推荐页会退化为热门电影展示；当用户具备足够行为后，则可切换不同算法生成个性化结果。这种实现方式既保证了系统交互完整性，也为论文中的冷启动讨论提供了实际依据。

## 第五章 实验与结果分析

### 5.1 实验环境与数据基础

本文实验所使用的数据来源于系统当前采集与治理后的正式数据集。根据数据质量报告，系统共包含 135,990 条电影记录和 198,067 条影人记录，预估知识图谱关系总量约为 1,443,933 条。实验中使用的推荐算法均基于现有代码实现和正式评估报告，不引入额外未实现模块。

在实验平台方面，后端服务基于 Python 3.11 和 FastAPI 实现，图谱存储使用 Neo4j 5.x，关系型数据存储使用 MySQL 8.x。推荐评估采用统一离线评测脚本执行，结果输出为 JSON 和 Markdown 报告，用于论文分析与后续追踪。

### 5.2 数据质量结果分析

从数据规模上看，系统已经具备较为丰富的电影实体和影人实体基础，足以支持图谱构建与推荐实验。从字段缺失情况看，名称、年份、地区、职业等基础字段总体较完整，而评分、剧情简介、片长、出生地等字段缺失较高。这一现象在垂直影视数据采集中具有一定代表性，其主要原因包括页面字段本身缺失、不同类型影视条目元信息不一致，以及反爬和采集失败导致的局部信息缺漏。

这些缺失字段对系统的影响并不完全相同。对于电影搜索、详情展示和基础图谱构建而言，影片名、年份、导演、演员和类型等字段是核心依赖，目前整体可用性较高；对于评分排序、推荐解释丰富度和影人信息展示而言，剧情简介、评分和片长等字段缺失则会对页面完整性和特征表达造成一定影响。因此，当前系统已经具备可用于实验和展示的基础质量，但仍需要在补采和治理层面继续优化。

### 5.3 系统功能验证

从功能实现角度看，系统已经完成从数据采集到前端展示的核心闭环。后端能够提供电影、影人、图谱、统计、认证、用户行为和推荐接口；前端已实现首页、搜索、电影详情、影人详情、推荐、图谱、路径、统计、登录注册和个人中心等主要页面。图谱服务能够返回节点、边和元信息，用于 ECharts 可视化；推荐服务能够返回推荐结果、推荐原因和解释子图；离线评估模块能够生成正式报告文件。

在验证层面，本文对图谱服务、推荐算法和评估模块相关测试进行了抽样运行，当前共 16 项关键测试通过。这说明系统的核心逻辑在代码层面具备一定稳定性，也为论文中对“系统已实现”的描述提供了支撑。

### 5.4 推荐算法对比设计

本文比较的推荐算法共有四类，其设计目的与特点如表 5-1 所示。

**表 5-1 四类推荐算法对比**

| 算法 | 核心依据 | 优点 | 局限性 |
| --- | --- | --- | --- |
| `content` | 类型、地区、年份等内容特征 | 简单直观、易解释、无需复杂图计算 | 难以捕捉深层关系语义 |
| `item_cf` | 用户评分行为共现 | 准确率高、传统基线稳定 | 依赖行为密度，冷启动较弱 |
| `kg_path` | 导演、演员、类型等共享路径 | 解释性强，关系证据清晰 | 准确率受路径覆盖限制 |
| `kg_embed` | 结构三元组嵌入与融合排序 | 可利用图谱语义，效果与解释较均衡 | 训练与推理开销相对较高 |

### 5.5 离线评估协议

推荐评估采用 validation/test sampled leave-one-out 协议，具体过程如下。

1. 从用户评分数据中筛选评分大于等于 3.5 的正向样本。
2. 要求每位用户至少具备足够的历史交互记录，以确保画像能够形成。
3. 按固定随机种子将用户划分为验证集和测试集，其中验证集用于参数搜索，测试集用于最终指标统计。
4. 对每位测试用户，选择其最后一个正向样本作为目标电影，并从未交互电影中采样 99 个负样本。
5. 针对每个算法重复执行 5 组负采样 seed，计算多组实验结果的均值与标准差。

该协议的优点在于能够降低单次随机采样带来的波动，同时通过将目标电影排除出训练画像避免评估中的数据泄漏问题。与只看一次命中率的粗糙方法相比，该协议更适合作为本科论文中的正式离线评估方案。

### 5.6 推荐实验结果

根据 2026 年 3 月 12 日生成的正式评估报告，在 `K=10` 条件下，不同算法的主要指标如表 5-2 所示。

**表 5-2 离线评估结果（K=10）**

| 算法 | Recall@10 | NDCG@10 | Hit@10 | Precision@10 |
| --- | --- | --- | --- | --- |
| `content` | 0.4186 ± 0.0145 | 0.2162 ± 0.0065 | 0.4186 ± 0.0145 | 0.0419 ± 0.0014 |
| `item_cf` | 0.8335 ± 0.0025 | 0.8131 ± 0.0030 | 0.8335 ± 0.0025 | 0.0833 ± 0.0003 |
| `kg_path` | 0.2025 ± 0.0030 | 0.0835 ± 0.0034 | 0.2025 ± 0.0030 | 0.0203 ± 0.0003 |
| `kg_embed` | 0.5801 ± 0.0210 | 0.3346 ± 0.0124 | 0.5801 ± 0.0210 | 0.0580 ± 0.0021 |

进一步比较覆盖率、多样性和平均时间等指标，结果如表 5-3 所示。

**表 5-3 覆盖率、多样性与时间结果**

| 算法 | Coverage@20 | Diversity@10 | Avg Time (s) |
| --- | --- | --- | --- |
| `content` | 0.0231 ± 0.0001 | 0.8416 ± 0.0018 | 0.3137 |
| `item_cf` | 0.0225 ± 0.0001 | 0.8928 ± 0.0011 | 0.0179 |
| `kg_path` | 0.0225 ± 0.0001 | 0.8963 ± 0.0011 | 2.1786 |
| `kg_embed` | 0.0230 ± 0.0000 | 0.8424 ± 0.0021 | 3.8461 |

从结果可以看出，`item_cf` 在准确率方面表现最优，说明当前数据集中用户评分行为具有较强的协同信号，传统协同过滤仍然是非常强的基线方法。`content` 的结果位于中间水平，说明仅依靠影片元数据特征可以提供一定程度的相关性，但无法充分刻画更复杂的语义关系。`kg_path` 的准确率相对较低，但其多样性较高，并且推荐理由更容易解释，适合用作关系可解释性较强的方法参考。`kg_embed` 虽未超过 `item_cf`，但显著优于 `content` 和 `kg_path`，说明知识图谱结构嵌入对提升推荐效果具有积极作用。

### 5.7 参数优化与消融分析

在参数优化阶段，`kg_path` 的最佳参数组合包括：`actor_order_limit=5`、`actor_weight=0.8`、`director_weight=1.0`、`genre_weight=0.6`、`two_hop_weight=0.4`、`enable_two_hop=true` 和 `use_degree_penalty=true`。这说明在路径推荐中，导演关系的重要性最高，演员和类型是有效补充，同时适当引入两跳路径与实体频率惩罚有助于改善排序质量。

`kg_embed` 的最佳参数组合包括：`centroid_weight=0.2`、`entity_overlap_weight=0.6`、`max_seed_weight=0.2`、`use_expanded_relations=true` 和 `use_fusion_ranking=true`。该结果表明，单纯依赖嵌入空间距离并不足以取得最优表现，融合实体重叠信号能够进一步增强推荐的稳定性与可解释性。

消融实验方面，`kg_path` 在 `1-hop`、`+2-hop` 和 `+IDF weighting` 三组配置下的 `Recall@10` 均为 `0.2025`，但 `NDCG@10` 从 `0.0802` 提升至 `0.0835`，说明两跳路径和实体度惩罚对排序位置有一定改善作用。`kg_embed` 在“原始三关系”“扩图三元组”和“扩图三元组+融合排序”三组消融配置下的 `Recall@10` 分别为 `0.4783`、`0.4633` 和 `0.5366`，说明仅增加结构关系并不能自动提升效果，融合排序机制才是改善结果的重要因素。

### 5.8 结果讨论

综合实验结果可以得到以下几点结论。

第一，当前数据集中的用户行为信号较强，因此基于物品的协同过滤能够取得最高准确率。这表明在电影推荐场景中，用户评分行为本身依然具有很高价值，知识图谱方法并不必然在所有指标上超过传统方法。

第二，知识图谱方法更适合在“结构语义增强”和“可解释推荐”两个方面体现优势。`kg_path` 虽然在准确率上不占优，但更容易给出“共同导演”“共同演员”“同类类型”等可理解证据；`kg_embed` 则在利用结构语义方面更进一步，能够在准确率与解释性之间达到较为均衡的效果。

第三，系统的当前限制也较为明显。路径推荐的性能仍受候选规模和共享关系覆盖度影响，嵌入推荐则存在训练和推理开销较高的问题。此外，原始数据的字段缺失和双数据库协同复杂度，也都会对系统表现带来一定影响。因此，后续优化应同时关注数据质量、算法效率和推荐解释质量。

## 第六章 总结与展望

### 6.1 工作总结

本文围绕豆瓣电影数据完成了一套电影知识图谱推荐系统的设计与实现。首先，基于 Playwright 构建了电影和影人采集链路，并结合任务锁、代理切换和监控脚本提升采集稳定性；其次，对原始数据进行了清洗、缺失分析和 ETL 处理，构建了包含电影、影人、类型、地区、语言、内容形式和年代分桶等节点的电影知识图谱；再次，基于 FastAPI 与 Vue3 实现了电影查询、影人查询、图谱探索、统计分析、用户交互与推荐展示功能，形成了从数据层到应用层的完整系统；最后，实现了四类推荐算法并通过正式离线评估报告对其表现进行了分析比较。

从实验结果看，当前系统已经具备较为清晰的工程主线和实验支撑。传统 `item_cf` 在准确率上表现最优，说明用户行为信息仍是非常有效的推荐信号；`kg_embed` 在引入知识图谱结构语义后表现出较好的综合能力，说明图谱增强推荐在电影场景中具有实际价值；`kg_path` 则体现了知识图谱在推荐解释方面的优势。总体而言，本文工作验证了豆瓣电影知识图谱在电影信息组织、图谱探索和推荐分析中的可行性。

### 6.2 不足之处

尽管系统已经完成主要功能，但仍存在以下不足。首先，豆瓣页面结构变化和反爬策略会直接影响采集稳定性，采集链路仍需继续优化。其次，剧情简介、评分、片长和部分影人属性字段的缺失率较高，会影响图谱完整性与推荐特征质量。再次，MySQL 与 Neo4j 双库协同提升了系统表达能力，但也增加了接口开发、缓存维护和一致性管理的复杂度。最后，知识图谱路径推荐和嵌入推荐虽然具有研究价值，但在准确率和时间开销方面仍未超过最强的协同过滤基线。

### 6.3 未来工作展望

未来可以从以下几个方面继续改进。

1. 持续优化采集链路，增强对页面变化和代理波动的适应能力，并对关键缺失字段执行定向补采。
2. 在图谱层面引入更多电影属性和更细粒度的关系特征，提高知识表达的完整性。
3. 优化推荐模块的缓存策略和候选生成效率，降低知识图谱方法的时间开销。
4. 进一步提升推荐解释质量，使解释结果与用户行为和图谱证据之间的对应关系更加清晰。
5. 在后续研究中探索融合评论文本、时间序列行为或更复杂图神经网络模型的推荐方法，以提升知识图谱推荐的效果上限。
