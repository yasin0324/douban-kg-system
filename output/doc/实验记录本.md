## 日期：2026-02-01 ~ 2026-02-04

### 实验内容：分布式爬虫与代理管理系统

#### 实验目的

为爬虫添加代理 IP 轮换和分布式支持，提高爬取效率并避免 IP 被封禁。

#### 实验环境/工具

- Python 3.11 + Playwright
- MySQL 8.x（任务协调）
- 代理 IP 服务（KDL 快代理等）

#### 实验步骤

1. **代理管理器** (`proxy_crawler.py`)：
   - 实现代理 IP 池，从多个 API 地址获取代理；
   - 支持代理有效性检测和自动切换；
   - 实现失败重试机制和代理轮换策略。

2. **分布式任务锁**：
   - 在 `subjects` 表中添加 `crawl_locked_at` 和 `crawl_worker` 字段；
   - 爬取前通过 SQL UPDATE 实现即时任务锁定：
     ```sql
     UPDATE subjects
     SET crawl_status='crawling', crawl_locked_at=NOW(), crawl_worker=%s
     WHERE crawl_status='pending'
     LIMIT 10
     ```
   - 支持多台机器通过唯一 worker ID 并行爬取。

3. **过期锁清理**：
   - 实现 `reset_stale_tasks` 函数，自动清理超过 30 分钟的锁；
   - 添加 `scripts/reset_all_locks.py` 手动重置脚本。

4. **爬取失败处理**：
   - 失败的电影立即释放状态锁（status 回退为 pending）；
   - 支持不完整数据的重爬 (`recrawl_incomplete_movies.py`)。

5. **Schema 设计文档**编写 (`documents/schema_design.md`)。

#### 实验结果

- 代理池正常工作，支持多 API 源切换。
- 分布式爬取测试：2 台机器并行爬取，无任务冲突。
- 截至本阶段，`subjects` 表中已录入约 94,000+ 条电影 URL。
- 数据库 Schema 设计文档完成。

#### 结果分析

分布式方案有效提升了爬取效率，代理轮换降低了被封概率。MySQL 的行级锁机制足以支撑数台机器的并发协调，无需引入 Redis 等中间件。

## 日期：2026-02-23

### 实验内容：数据清洗与 ETL 至 Neo4j 知识图谱

#### 实验目的

对 MySQL 中的原始爬取数据进行清洗标准化，然后通过 ETL 管道将数据导入 Neo4j 图数据库，构建电影领域知识图谱。

#### 实验环境/工具

- Python 3.11 + PyMySQL + neo4j-driver
- MySQL 8.x（数据源）
- Neo4j 5.x（目标图数据库）

#### 实验步骤

**第一步：数据清洗** (`data_processing/clean_pipeline.py`)

1. 类型字段标准化：将 `genres` 中的空格分隔符统一为 `/`，去除重复类型。
2. 地区字段标准化：统一 `regions` 格式。
3. 语言字段标准化：统一 `languages` 格式。
4. 去除各字段首尾空白字符。

**第二步：ETL 导入** (`data_processing/etl_to_neo4j.py`)

分 11 个阶段批量导入：

| 阶段 | 操作 | 说明 |
|------|------|------|
| Phase 1 | 创建约束和索引 | Movie.mid, Person.pid, Genre.name 等唯一性约束 |
| Phase 2 | 导入 Genre 节点 | 从 movies.genres 提取所有不重复类型 |
| Phase 2B-E | 导入 Region/Language/ContentType/YearBucket 节点 | 从电影元数据派生 |
| Phase 3 | 导入 Movie 节点 | 包含 mid, title, rating, year, cover 等属性 |
| Phase 4 | 导入 Person 节点 | 包含 pid, name, sex, birth 等属性 |
| Phase 5 | 创建 DIRECTED 关系 | Person → Movie |
| Phase 6 | 创建 ACTED_IN 关系 | Person → Movie（含 order 属性） |
| Phase 7 | 创建 HAS_GENRE 关系 | Movie → Genre |
| Phase 8-11 | 创建扩展关系 | IN_REGION, IN_LANGUAGE, HAS_CONTENT_TYPE, IN_YEAR_BUCKET |

使用 UNWIND 批量导入优化性能，批次大小 2000 条。

#### 实验结果

**知识图谱 Schema 结构图**：

```
                    ┌─────────────┐
                    │   Person    │
                    │ (pid, name) │
                    └──────┬──────┘
                   DIRECTED│ ACTED_IN
                           ▼
┌──────────┐    HAS_GENRE    ┌───────────────────────┐    IN_REGION     ┌──────────┐
│  Genre   │◄────────────────│        Movie          │───────────────►  │  Region  │
│ (name)   │                 │ (mid, title, rating,  │                  │ (name)   │
└──────────┘                 │  year, cover, url)    │                  └──────────┘
                             └───────────┬───────────┘
                        IN_LANGUAGE │         │ HAS_CONTENT_TYPE
                                    ▼         ▼
                             ┌──────────┐ ┌─────────────┐
                             │ Language │ │ ContentType │
                             │ (name)   │ │ (name)      │
                             └──────────┘ └─────────────┘
                                    │
                              IN_YEAR_BUCKET
                                    ▼
                             ┌─────────────┐
                             │ YearBucket  │
                             │ (name)      │
                             └─────────────┘
```

**导入数据量统计**：

| 类别 | 数量 |
|------|------|
| Movie 节点 | ~140,000 |
| Person 节点 | ~200,000 |
| Genre 节点 | ~32 |
| Region 节点 | ~400 |
| Language 节点 | ~900 |
| ContentType 节点 | 2 (movie/tv) |
| YearBucket 节点 | 5 |
| DIRECTED 关系 | ~100,000 |
| ACTED_IN 关系 | ~800,000 |
| HAS_GENRE 关系 | ~250,000 |
| **总关系数** | **~1,700,000** |

**第三步：导入验证**

运行验证查询，以"肖申克的救赎"为例：

```cypher
MATCH (m:Movie)
WHERE m.title CONTAINS '肖申克'
OPTIONAL MATCH (d:Person)-[:DIRECTED]->(m)
OPTIONAL MATCH (a:Person)-[:ACTED_IN]->(m)
OPTIONAL MATCH (m)-[:HAS_GENRE]->(g:Genre)
RETURN m.title, m.rating, collect(DISTINCT d.name), collect(DISTINCT a.name)[0..5], collect(DISTINCT g.name)
```

结果：肖申克的救赎（评分 9.7），导演：弗兰克·德拉邦特，类型：犯罪/剧情。验证通过。

> **[截图占位：Neo4j Browser 中执行验证查询的结果界面，展示电影节点与关联影人、类型节点的图谱可视化]**

#### 结果分析

ETL 管道运行稳定，批量 UNWIND 操作大幅提升了导入效率（约 10 分钟完成全量导入）。知识图谱规模属于小型图谱，单机 Neo4j (1GB RAM) 即可轻松承载。7 种节点类型和 7 种关系类型覆盖了电影领域的核心语义关系。

### 实验内容：CFKG 主链路与基线推荐算法实现

#### 实验目的

实现完整的多算法推荐框架，包括基线算法（Item-CF、Content-Based）和知识图谱路径推荐（KG-Path），并建立 CFKG 融合主链路。

#### 实验环境/工具

- Python 3.11
- MySQL + Neo4j
- 算法代码目录：`db-backend/app/algorithms/`

#### 实验步骤

1. **算法基类设计** (`algorithms/base.py`，254 行)：
   ```python
   class BaseRecommender(ABC):
       @abstractmethod
       def recommend(self, user_id, n, exclude_mids, exclude_from_training):
           ...

       # 信号权重定义
       PREF_SIGNAL_WEIGHT = {"like": 0.7, "want_to_watch": 0.25}
       PREF_COMPAT_RATING = {"like": 3.5, "want_to_watch": 1.25}
   ```
   - 统一的用户正反馈电影获取逻辑（评分 >= 3.5 为正样本）；
   - 偏好信号（喜欢/想看）转换为等效评分权重。

2. **Item-CF 协同过滤** (`algorithms/item_cf.py`，212 行)：
   - 基于用户共同评分的物品-物品余弦相似度；
   - 从用户的正反馈电影出发，找到相似电影；
   - 支持数据泄露防护（评估时临时屏蔽测试样本）。

3. **Content-Based 基于内容推荐** (`algorithms/content_based.py`，220 行)：

## 日期：2026-03-11 ~ 2026-03-12

### 实验内容：KG-Embed（TransE 嵌入）推荐算法与消融实验

#### 实验目的

实现基于 TransE 知识图谱嵌入的推荐算法，通过向量空间中的语义相似度发现潜在的推荐候选；并通过消融实验验证各组件的贡献。

#### 实验环境/工具

- Python 3.11 + NumPy
- Neo4j 三元组数据
- 算法代码：`algorithms/kg_embed.py`（766 行）

#### 实验步骤

1. **TransE 嵌入训练**：
   - 从图谱缓存中提取三元组 (head, relation, tail)；
   - 核心关系（3 种）：Director, Actor, Genre；
   - 扩展关系（额外 4 种）：Region, Language, ContentType, YearBucket；
   - 加入反向关系（如 director_REV）以增强信息流。
   - 训练参数：
     ```python
     embedding_dim = 64
     learning_rate = 0.01
     margin = 1.0
     max_epochs = 200
     batch_size = 4096
     early_stopping_patience = 3
     validation_ratio = 0.1
     ```
   - 损失函数：Margin-based ranking loss
     ```
     L = max(0, margin + ||h + r - t||₂ - ||h' + r - t'||₂)
     ```

2. **推荐策略**：
   - 计算用户偏好向量：正反馈电影嵌入向量的加权质心；
   - 候选打分（融合排序）：
     ```python
     score = centroid_weight * centroid_similarity
           + entity_overlap_weight * entity_overlap_score
           + max_seed_weight * max_seed_similarity
     ```
   - entity_overlap：候选与 seed 的共同导演/演员/类型数量。

3. **消融实验设计**：
   - **消融 1**：仅使用核心三关系（Director, Actor, Genre）；
   - **消融 2**：加入扩展三元组（+Region, Language, ContentType, YearBucket）；
   - **消融 3**：扩展三元组 + 融合排序（centroid + entity_overlap + max_seed）。

#### 实验结果

**KG-Embed 消融实验结果**：

| 消融实验 | HR@10 | NDCG@10 | 说明 |
|---------|-------|---------|------|
| 原始三关系 | 0.3869 ± 0.0192 | 0.2243 ± 0.0084 | 仅 Director/Actor/Genre |
| +扩展三元组 | 0.4845 ± 0.0071 | 0.2720 ± 0.0144 | +Region/Language/ContentType/YearBucket |
| +融合排序 | 0.5250 ± 0.0069 | 0.3142 ± 0.0076 | +entity_overlap + max_seed |

**消融分析**：
- 扩展三元组带来 +25.2% 的 HR@10 提升（0.3869 → 0.4845），说明地区、语言等属性信息对推荐有显著帮助；
- 融合排序在此基础上再提升 +8.4%（0.4845 → 0.5250），entity_overlap 通过显式匹配共同实体进一步提高精度。

#### 结果分析

TransE 嵌入成功将知识图谱的结构信息编码为低维向量，使得语义相似度计算成为可能。扩展关系的加入丰富了嵌入的语义信息，融合排序则结合了嵌入相似度和显式实体重叠两种信号，互为补充。

### 实验内容：KG-Path 消融实验与候选剪枝优化

#### 实验目的

通过消融实验验证 KG-Path 算法中二跳路径和 IDF 度惩罚的贡献，并优化候选剪枝策略。

#### 实验步骤

1. 设计三组消融实验：
   - **1-hop**：仅使用一跳路径（共同导演/演员/类型直接关联）；
   - **+2-hop**：启用二跳路径（通过中间演员扩展候选集）；
   - **+IDF weighting**：在 2-hop 基础上加入 IDF 度惩罚。

2. 关键参数：
   ```python
   BEST_KG_PATH_PARAMS = {
       "actor_order_limit": 5,     # 仅考虑前 5 位演员
       "actor_weight": 0.6,
       "director_weight": 0.8,     # 导演权重最高
       "genre_weight": 0.2,
       "enable_two_hop": True,
       "two_hop_weight": 0.2,
       "use_degree_penalty": True,
   }
   ```

#### 实验结果

**KG-Path 消融实验结果**：

| 消融实验 | HR@10 | NDCG@10 | 说明 |
|---------|-------|---------|------|
| 1-hop | 0.3595 ± 0.0072 | 0.2448 ± 0.0045 | 仅一跳路径 |
| +2-hop | 0.3702 ± 0.0070 | 0.2524 ± 0.0016 | 启用二跳路径 |
| +IDF weighting | 0.3702 ± 0.0070 | 0.2573 ± 0.0022 | 加入度惩罚 |

**分析**：
- 二跳路径带来 +2.9% 的 HR@10 提升，扩展了候选集覆盖面；
- IDF 度惩罚对 HR@10 无额外提升，但 NDCG@10 小幅改善（+1.9%），说明排序质量有所提高。

#### 结果分析

KG-Path 的优势在于推荐的可解释性——每条推荐都可以追溯到具体的图谱路径（如"因为你喜欢A电影的导演也执导了B电影"）。IDF 惩罚在排序层面有帮助，避免了"多产演员"主导推荐结果的问题。

### 实验内容：离线评估框架实现与全面参数调优

#### 实验目的

实现严格的离线评估框架，对所有推荐算法进行系统性评估和参数调优，得出可靠的性能对比结论。

#### 实验环境/工具

- Python 3.11
- 评估代码：`algorithms/evaluator.py`（772 行）
- 测试集：168 名公开豆瓣用户

#### 实验步骤

1. **评估协议设计**：
   - 方法：Sampled Leave-One-Out（每用户留 1 个正样本 + 99 个随机负样本）；
   - 多 seed 鲁棒性：使用 5 个随机种子 [42, 52, 62, 72, 82]，报告均值和标准差；

2. **评估指标**（K = 5, 10, 20）：
   - HR@K (Hit Rate)：推荐列表是否命中正样本；
   - NDCG@K：考虑排序位置的增益度量；
   - Coverage@K：推荐结果覆盖的电影比例；
   - Diversity@K：推荐列表内部的多样性。

3. **参数调优**（在 42 名验证集用户上进行）：
   - KG-Path：actor_order_limit, actor_weight, director_weight, genre_weight, enable_two_hop, two_hop_weight, use_degree_penalty；
   - KG-Embed：centroid_weight, entity_overlap_weight, max_seed_weight, use_expanded_relations, use_fusion_ranking；
   - CFKG：item_cf_weight, kg_embed_weight, kg_path_weight, item_cf_recall。

4. **最终测试**（在 168 名测试集用户上评估最优参数）。

#### 实验结果

**推荐算法离线评估报告（K=10）**：

| 算法 | HR@10 | NDCG@10 | Coverage@20 | Diversity@10 | 平均耗时 (s) |
|------|-------|---------|-------------|--------------|-------------|
| 基于物品的协同过滤 (Item-CF) | **0.8083 ± 0.0069** | **0.5992 ± 0.0071** | 0.0233 | 0.9046 | 0.50 |
| 基于知识图谱嵌入 (KG-Embed) | 0.5333 ± 0.0208 | 0.3330 ± 0.0100 | 0.0239 | 0.8370 | 6.53 |
| CFKG 主链路 | 0.4393 ± 0.0095 | 0.3156 ± 0.0064 | 0.0234 | 0.8935 | 28.53 |
| 基于知识图谱路径 (KG-Path) | 0.3702 ± 0.0070 | 0.2573 ± 0.0022 | 0.0235 | 0.8984 | 3.48 |
| 基于内容 (Content-Based) | 0.3512 ± 0.0119 | 0.1767 ± 0.0067 | 0.0238 | 0.8249 | 0.23 |

**完整 K 值对比**：

| 算法 | HR@5 | HR@10 | HR@20 |
|------|------|-------|-------|
| Item-CF | 0.7595 | 0.8083 | 0.8583 |
| KG-Embed | 0.3988 | 0.5333 | 0.7369 |
| CFKG | 0.3381 | 0.4393 | 0.4988 |
| KG-Path | 0.2595 | 0.3702 | 0.4476 |
| Content-Based | 0.1988 | 0.3512 | 0.5536 |

**最优参数**：

- **CFKG**：`item_cf_recall=300, item_cf_weight=0.6, kg_embed_weight=0.25, kg_path_weight=0.15`
- **KG-Path**：`actor_order_limit=5, actor_weight=0.6, director_weight=0.8, genre_weight=0.2, enable_two_hop=true, two_hop_weight=0.2, use_degree_penalty=true`
- **KG-Embed**：`centroid_weight=0.2, entity_overlap_weight=0.6, max_seed_weight=0.2, use_expanded_relations=true, use_fusion_ranking=true`

#### 结果分析

1. **Item-CF 准确性最高**（HR@10: 0.81），但这是因为协同过滤直接利用了用户-物品评分矩阵的相似性，在数据充足时表现优异。

2. **KG-Embed 排名第二**（HR@10: 0.53），TransE 嵌入成功捕获了图谱的结构化语义信息，尤其在扩展三元组和融合排序优化后提升显著。

3. **CFKG 综合表现均衡**（HR@10: 0.44, Diversity: 0.89），作为融合算法在准确性、多样性和可解释性之间取得了较好的平衡。

4. **KG-Path 的核心优势在于可解释性**（HR@10: 0.37），每条推荐都可追溯到具体的图谱路径，这是纯 CF 方法无法提供的。

5. **Content-Based 是最快的兜底方案**（0.23s），虽然精度较低，但在冷启动场景下不可或缺。

6. **多样性方面**：Item-CF (0.90) 和 KG-Path (0.90) 多样性最高，Content-Based (0.82) 最低。

### 实验内容：偏好信号权重统一与中期总结

#### 实验目的

统一所有推荐算法中偏好信号（评分、喜欢、想看）的权重处理逻辑，消除不同算法间的不一致性；整理项目文档，准备中期答辩材料。

#### 实验环境/工具

- Python 3.11
- 算法代码：`algorithms/base.py`

#### 实验步骤

1. **偏好信号权重统一** (`algorithms/base.py`)：
   - 将原本分散在各算法中的信号权重处理逻辑统一到基类：
     ```python
     # 信号类型及其权重
     PREF_SIGNAL_WEIGHT = {
         "like": 0.7,           # 喜欢 → 0.7 权重
         "want_to_watch": 0.25, # 想看 → 0.25 权重
     }
     PREF_COMPAT_RATING = {
         "like": 3.5,           # 喜欢等效评分 3.5
         "want_to_watch": 1.25, # 想看等效评分 1.25
     }
     KG_SIGNAL_PRIORITY = {
         "rating": 0,     # 评分优先级最高
         "like": 1,
         "want_to_watch": 2,
     }
     ```
   - 统一 `get_user_positive_movies()` 方法，处理评分和偏好两种信号源；
   - 为 KG 类算法实现 `get_user_positive_movies_for_kg()`，限制 seed 数量（最多 30 条评分 + 10 条喜欢）以控制图遍历范围。

2. **推荐解释增强**：
   - KG-Embed 的实体重叠原因返回影人名称而非 ID；
   - 前端推荐详情抽屉展示可读的推荐路径。

3. **中期答辩材料准备**：
   - 编写中期汇报 PPT 大纲与讲稿；
   - 整理推荐算法源码走读文档；
   - 更新评估结果报告。

#### 实验结果

- 所有 5 种算法使用统一的信号权重处理逻辑，代码一致性大幅提升。
- 推荐解释文案更加可读：
  - 示例："推荐《无间道》：因为你喜欢的《重庆森林》的导演王家卫也执导了该片"→ 改为使用影人名称而非 ID。

> **[截图占位：推荐结果页面，展示推荐列表和推荐原因说明]**

#### 结果分析

信号权重统一是重要的工程优化，确保了不同算法对同一用户行为的理解一致。"like" 权重 0.7 高于 "want_to_watch" 权重 0.25 是合理的——主动标记"喜欢"的电影比"想看"更能反映用户的真实偏好。
