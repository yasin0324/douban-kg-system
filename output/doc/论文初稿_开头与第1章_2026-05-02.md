# 基于知识图谱的电影推荐系统设计与实现

# Design and Implementation of a Movie Recommendation System Based on a Knowledge Graph

## 摘 要

随着网络电影资源和用户观影行为数据持续积累，传统依赖列表浏览、关键词检索或单一评分排序的电影服务难以充分表达影片、影人、类型、地区、语言和用户兴趣之间的复杂关联，推荐结果也容易出现依据不清、解释性不足等问题。针对上述问题，本文设计并实现了一个基于知识图谱的电影推荐系统。系统以豆瓣电影相关数据为对象，使用 Playwright 完成电影与影人数据采集，将原始业务数据、用户信息和用户行为存储在 MySQL 中，并经过字段清洗、规范化处理和质量分析后导入 Neo4j，形成包含电影、影人、类型、地区、语言、内容形式和年代分桶等实体的电影知识图谱。在应用层面，系统采用 FastAPI 构建后端服务，提供电影检索、影人查询、图谱探索、统计分析、用户认证、用户行为、个性化推荐和推荐解释接口；前端采用 Vue3、Element Plus 和 ECharts 实现搜索、筛选、详情展示、图谱可视化、统计看板、个人中心和推荐页面。在推荐模块中，系统实现了基于内容、基于物品协同过滤、基于知识图谱路径、基于知识图谱嵌入以及融合推荐 CFKG 五类算法，并通过推荐解释接口展示与用户偏好相关的图谱证据。实验与验收结果表明，在 `1 positive + 499 negatives` 的 sampled leave-one-out 离线评测协议下，CFKG 主链路在 `K=10` 时达到 `HR@10=0.7910`、`NDCG@10=0.5240`；系统在线链路验收覆盖 46 个场景，均通过验证。本文工作形成了从数据采集、数据清洗、知识图谱构建到前后端应用和推荐评测的完整工程闭环，为电影领域知识图谱推荐系统的实现提供了一个可复现的实践案例。

**关键词：** 知识图谱; 电影推荐; Neo4j; FastAPI; Vue3; 推荐解释

## Abstract

With the continuous growth of online movie resources and user behavior data, movie services based only on list browsing, keyword search, or single-score ranking are insufficient for representing the complex relations among movies, persons, genres, regions, languages, and user interests. Their recommendation results are also often difficult to explain. To address these issues, this thesis designs and implements a movie recommendation system based on a knowledge graph. The system uses Douban movie-related data as the data source. Playwright is used to collect movie and person data, while MySQL stores raw business data, user information, and user behaviors. After field cleaning, normalization, and data quality analysis, the processed data are imported into Neo4j to construct a movie knowledge graph containing entities such as movies, persons, genres, regions, languages, content types, and year buckets. At the application layer, FastAPI is used to build backend services for movie search, person query, graph exploration, statistical analysis, authentication, user behaviors, personalized recommendation, and recommendation explanation. The frontend is implemented with Vue3, Element Plus, and ECharts, providing pages for search, filtering, detail display, graph visualization, statistical dashboards, user profiles, and recommendations. In the recommendation module, five algorithms are implemented, including content-based recommendation, item-based collaborative filtering, knowledge graph path-based recommendation, knowledge graph embedding-based recommendation, and the fused CFKG recommendation pipeline. Recommendation explanations are provided through graph-related evidence associated with user preferences. The evaluation and acceptance results show that, under the sampled leave-one-out protocol with `1 positive + 499 negatives`, the CFKG pipeline achieves `HR@10=0.7910` and `NDCG@10=0.5240`. The online acceptance report covers 46 scenarios, all of which pass the verification. This work forms a complete engineering pipeline from data collection, data cleaning, and knowledge graph construction to frontend-backend application development and recommendation evaluation, providing a reproducible practice case for movie recommendation systems based on knowledge graphs.

**Keywords:** knowledge graph; movie recommendation; Neo4j; FastAPI; Vue3; recommendation explanation

# 第 1 章 绪论

## 1.1 研究背景与意义

互联网电影平台长期积累了大量影片条目、影人资料、类型标签、制片地区、语言、上映年份、评分人数和用户行为数据。对于普通用户而言，电影资源数量增加一方面扩大了选择空间，另一方面也提高了信息筛选成本。仅依靠关键词检索或按评分、热度进行列表排序时，用户往往需要在大量条目中反复比较，难以快速发现与个人兴趣相匹配的影片。推荐系统的目标正是从用户历史行为和物品特征中挖掘潜在偏好，帮助用户缩小选择范围，提高信息获取效率[6][C6]。

传统推荐方法通常可以分为基于内容的推荐和协同过滤推荐。基于内容的推荐依赖电影的类型、地区、年代、导演、演员等元数据刻画物品相似性，适合在用户行为不足时提供基础推荐；协同过滤则利用用户之间或物品之间的交互相似性，能够从群体行为中发现潜在兴趣。然而，在电影领域中，影片与影人、类型、地区、语言等实体之间存在天然的多关系结构，单纯依赖平面特征向量或用户评分矩阵难以表达多跳语义关系。例如，两部电影可能并不属于同一类型，但可能共享导演、主演、制片地区或相近年代；某位用户对某个演员的偏好也可能通过其参演作品扩展到其他相关影片。若推荐系统不能利用这些结构化关系，就容易损失可解释的领域知识。

知识图谱以实体和关系为基本组织方式，能够将电影、影人、类型、地区、语言、内容形式和年代等信息统一表达为图结构。近年来，知识图谱在推荐系统中的应用受到广泛关注，相关研究表明，外部知识或领域知识能够缓解数据稀疏、冷启动和推荐解释不足等问题[1][2][C1]。在电影推荐场景中，知识图谱不仅可以作为推荐算法的特征来源，也可以支持关系查询、路径分析、图谱可视化和推荐解释。与只给出“猜你喜欢”的推荐结果相比，基于图谱的系统可以进一步说明候选电影与用户偏好之间的关联，例如共享类型、共同影人、相近年代或用户历史正反馈所连接的结构路径。

本文围绕电影推荐这一典型应用场景，设计并实现一个基于知识图谱的电影推荐系统。系统将数据采集、MySQL 存储、数据清洗、Neo4j 图谱构建、FastAPI 后端服务、Vue3 前端应用、推荐算法和离线评测串联起来，形成完整的软件工程实现链路。与只关注单一算法模型的研究不同，本文更侧重于将电影领域数据组织、图谱建模、业务接口、前端交互和推荐评测结合起来，说明知识图谱推荐系统在本科软件工程项目中的设计方法和实现过程。

本文研究具有以下意义。第一，从数据组织角度看，系统将电影条目、影人信息和结构化维度统一建模，有助于提高电影领域数据的可查询性和可扩展性。第二，从推荐应用角度看，系统同时实现内容推荐、物品协同过滤、图谱路径推荐、图谱嵌入推荐和融合推荐链路，便于比较不同类型推荐方法在同一数据环境下的表现。第三，从用户体验角度看，系统通过图谱探索、统计看板和推荐解释增强了电影浏览与推荐结果的可理解性。第四，从工程实践角度看，系统覆盖采集、清洗、建图、服务、前端和评测等环节，可为同类垂直领域知识图谱应用的开发提供参考。

本文研究范围聚焦于本地开发与验收环境下的系统设计、功能实现和推荐评测。系统的重点在于基于真实项目代码和实验报告，完成一个面向电影领域的知识图谱推荐系统设计与实现，并通过功能链路验收和离线推荐评测验证核心功能的可用性。

## 1.2 国内外研究现状

推荐系统研究起步较早，协同过滤、矩阵分解、基于内容的推荐以及混合推荐等方法构成了较长时间内的基础技术路线。协同过滤方法通过用户-物品交互矩阵发现相似用户或相似物品，能够在不显式理解物品内容的情况下产生推荐结果；基于内容的方法则利用物品属性构建特征表示，在用户行为较少或物品冷启动场景下具有一定优势。随着深度学习和图学习方法的发展，推荐系统逐渐从浅层相似性计算扩展到更复杂的表示学习、序列建模和图结构建模方向[4][6]。

知识图谱为推荐系统引入了更加明确的实体和关系语义。国外相关研究较早关注知识图谱推荐的技术体系，包括基于路径的方法、基于嵌入表示的方法和基于图神经网络的方法等。路径类方法通常利用用户偏好物品与候选物品之间的实体连接关系计算推荐分数，并能较自然地生成解释；嵌入类方法将实体和关系映射到低维向量空间，通过向量计算刻画语义关联；图神经网络类方法则进一步利用邻域传播机制建模高阶关系[1][3][8]。近年来，知识图谱推荐研究还关注对比学习、自监督学习、多视图建模和可解释推荐等问题，试图在复杂图结构中筛选更有效的知识信号[11][12][14]。

在电影推荐领域，知识图谱具有较强的适配性。电影数据天然包含影片、导演、演员、类型、地区、语言和年份等结构化信息，适合用属性图或 RDF 图进行组织。IMDb、DBpedia、Wikidata 等开放数据源为国外电影知识图谱研究提供了基础数据，相关工作常将电影实体与外部知识库关联，再用于个性化推荐、语义搜索或解释生成[7][10]。电影场景中的知识图谱推荐不仅关注“推荐什么”，还关注“为什么推荐”，例如通过共同演员、共同导演、相似类型或用户历史兴趣路径解释推荐原因。

国内关于知识图谱推荐系统的研究近年也较为活跃。相关综述从知识表示、知识融合、推荐模型和应用场景等角度总结了知识图谱推荐的发展脉络，并指出知识图谱能够为推荐系统提供语义关联、辅助特征和解释依据[C1][C2][C3]。在方法层面，国内研究既关注知识图谱嵌入、图卷积、图注意力等模型，也关注用户偏好传播、协同知识图谱和混合推荐等方向[C5][C9][C12]。这些研究为电影推荐系统的设计提供了方法参考，但在具体工程实现中仍需结合数据规模、系统复杂度、开发周期和可复现实验条件进行取舍。

从现有研究和工程实践看，知识图谱推荐系统仍面临若干实际问题。首先，知识图谱质量直接影响推荐效果，实体属性缺失、关系不完整和数据噪声都会影响后续算法表现。其次，图谱推荐方法往往需要在准确性、计算耗时和解释性之间权衡，复杂模型未必适合所有本地系统或本科项目环境。再次，很多研究侧重算法指标，而完整应用系统还需要处理数据采集、数据库设计、接口封装、前端交互、权限控制和测试验收等工程问题。因此，本文将研究重点放在面向电影领域的端到端系统实现上，在保证事实可追溯的前提下，综合使用 MySQL、Neo4j、FastAPI、Vue3 和多类推荐算法完成系统设计。

## 1.3 研究内容

本文以“基于知识图谱的电影推荐系统设计与实现”为题，围绕电影领域数据组织、知识图谱建模、推荐算法实现和前后端系统开发展开研究。主要研究内容如下。

第一，完成电影与影人数据采集及预处理。系统使用 Playwright 采集电影条目和影人信息，将采集结果写入 MySQL。针对原始字段中存在的空值、格式不一致和部分字段缺失等问题，系统通过数据清洗流程完成字段规范化、空字符串处理、评分和票数字段处理，并输出数据质量分析结果，为后续图谱构建和推荐实验提供数据基础。

第二，设计电影知识图谱 Schema 并完成 Neo4j 图谱构建。系统将电影、影人、类型、地区、语言、内容形式和年代分桶建模为节点，将执导、参演、所属类型、所属地区、语言、内容形式和年代分桶建模为关系。通过 MySQL 到 Neo4j 的 ETL 脚本，系统完成节点导入、关系导入、约束索引创建和导入验证，使电影领域数据能够以图结构支持查询、探索和推荐。

第三，设计并实现后端服务。后端采用 FastAPI 构建，连接 MySQL 与 Neo4j，为前端提供统一接口。接口范围覆盖电影搜索、电影详情、影人搜索、影人详情、图谱探索、统计分析、用户注册登录、用户评分与偏好、个性化推荐以及推荐解释等功能。后端同时负责数据库连接、鉴权、推荐算法调用和部分缓存预热逻辑。

第四，设计并实现前端交互应用。前端采用 Vue3、Vue Router、Pinia、Element Plus、Axios 和 ECharts 实现电影推荐系统的主要页面，包括首页、搜索页、电影库筛选页、电影详情页、影人详情页、图谱探索页、统计看板、登录注册页、个人中心和推荐中心。前端通过图谱可视化、统计图表和推荐解释抽屉展示系统结果，提高用户对电影关系和推荐依据的理解。

第五，实现多类推荐算法并进行离线评测。系统实现基于内容的推荐、基于物品的协同过滤、基于知识图谱路径的推荐、基于知识图谱嵌入的推荐和 CFKG 融合推荐。推荐输入主要来自用户评分、喜欢和想看等行为。评测阶段采用公开豆瓣用户数据，在 `1 positive + 499 negatives` 的 sampled leave-one-out 协议下比较不同算法的 `HR@K` 与 `NDCG@K` 等指标，同时分析覆盖率、多样性和算法耗时，为推荐模块的效果判断提供依据。

第六，完成系统测试与验收分析。本文在后续章节中将结合单元测试、回归测试、接口链路验收和离线推荐评测，说明系统核心功能的实现情况。测试与实验分析以当前项目已有报告和代码证据为依据，评价重点放在功能链路、推荐效果和工程可用性上。

## 1.4 论文组织结构

本文共分为六章，各章内容安排如下。

第 1 章为绪论。本章介绍电影推荐系统和知识图谱推荐的研究背景与意义，概述国内外相关研究现状，说明本文的主要研究内容和论文结构。

第 2 章为相关技术与需求分析。本章将介绍 Playwright、MySQL、Neo4j、FastAPI、Vue3、ECharts 和推荐算法等相关技术，并结合系统面向的游客和注册用户两类使用场景分析功能需求与非功能需求。

第 3 章为系统总体设计。本章将从总体架构、数据库设计、知识图谱 Schema、后端服务设计、前端结构设计和推荐模块总体设计等方面说明系统的整体方案，重点阐述 MySQL 与 Neo4j 的职责划分以及推荐模块在系统中的位置。

第 4 章为系统详细设计与实现。本章将结合实际代码说明数据采集、数据清洗、MySQL 到 Neo4j 的 ETL、图谱查询、用户认证与用户行为、推荐算法和前端页面的实现过程，突出系统各模块之间的数据流转关系。

第 5 章为系统测试与实验分析。本章将说明测试环境、测试方法、功能测试结果、在线链路验收和推荐算法离线评测结果，并对不同推荐算法的表现及消融实验进行分析。

第 6 章为总结与展望。本章将总结本文完成的主要工作，分析系统当前仍存在的不足，并从数据更新、特征扩展、图谱计算效率、推荐解释和部署监控等方面提出后续改进方向。
