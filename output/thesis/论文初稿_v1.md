# 基于知识图谱的电影推荐系统设计与实现

## 摘要

传统电影推荐系统在面对冷启动、数据稀疏和推荐理由不透明等问题时，往往难以同时兼顾准确性与可解释性。豆瓣电影平台包含电影、影人、类型、地区、语言、评分与用户偏好等多维信息，为构建垂直领域知识图谱及开展推荐研究提供了良好的数据基础。针对上述问题，本文设计并实现了一个基于知识图谱的电影推荐系统，围绕数据采集、知识图谱构建、推荐算法实现、系统集成和实验验证形成完整链路。

在系统实现方面，本文采用 Playwright 对豆瓣电影和影人页面进行采集，使用 MySQL 存储原始数据与用户行为，使用 Neo4j 存储电影知识图谱，并基于 FastAPI 与 Vue 构建前后端分离系统。在推荐方法方面，本文实现了基于内容的推荐、基于物品的协同过滤、基于知识图谱路径的推荐、基于知识图谱嵌入的推荐以及融合式 `CFKG` 主链路推荐，并为推荐结果设计了图谱化解释机制。在数据与图谱规模方面，当前系统已整理电影记录 `192,032` 条、影人记录 `236,882` 条，预估图谱关系总量为 `1,890,704` 条。

实验部分采用 `sampled leave-one-out` 协议，对公开豆瓣用户数据进行离线评估。实验结果表明，在当前评估设置下，`ItemCF` 在准确率指标上整体表现最优，`KG-Embed` 是当前表现最好的图谱方法，而 `CFKG` 作为系统主链路，在准确性、多样性和可解释性之间取得了较好的平衡。在线路验收方面，系统在 `2026-04-12` 的线上链路验收中共覆盖 `46` 个场景，全部通过。

综上，本文所设计的系统验证了知识图谱在电影推荐场景中的可行性与应用价值，为电影内容组织、推荐解释和前后端集成提供了可落地的实现方案。后续工作可从图谱质量提升、主链路性能优化和在线用户反馈闭环等方向进一步完善系统能力。

**关键词：** 知识图谱；电影推荐；协同过滤；知识图谱嵌入；推荐解释

## 第一章 绪论

### 1.1 研究背景与意义

随着在线影视平台和内容社区的快速发展，用户面对的候选电影数量持续增长，如何从海量内容中高效筛选出符合用户兴趣的影片，已经成为推荐系统的重要应用场景之一。传统推荐方法主要包括基于内容的推荐与协同过滤推荐，其中基于内容的方法依赖电影自身属性进行相似度计算，协同过滤则通过用户或物品之间的交互关系发现潜在偏好。这类方法在工业界得到广泛应用，但在冷启动、数据稀疏和推荐理由解释方面仍存在明显局限 `[R01][R02][R13][R16][C03]`。

知识图谱作为一种以实体、关系和属性为核心的结构化知识组织方式，能够以图结构形式表达电影、影人、类型、地区、语言等多维信息。与传统扁平化元数据相比，知识图谱能够更自然地建模电影领域中的复杂关系，为推荐系统提供结构化语义支撑 `[R06][R08][R09][R12][C01][C04]`。在电影推荐场景下，如果能够将用户兴趣种子与图谱中的导演、演员、类型等实体建立关联，不仅有助于拓展候选召回范围，还能为推荐结果提供更加直观的证据路径。

豆瓣电影具有数据规模大、用户互动活跃和元信息维度丰富等特点，是研究垂直领域电影知识图谱与推荐应用的合适数据源。围绕豆瓣电影构建知识图谱并实现推荐系统，既具有一定的工程实践价值，也能为知识图谱增强推荐的研究提供案例支持。因此，本文以豆瓣电影为数据基础，设计并实现一个集数据采集、图谱构建、推荐算法、离线评估和前端展示于一体的端到端系统。

### 1.2 国内外研究现状

推荐系统研究已经形成较为成熟的理论体系。协同过滤与 ItemCF 等方法从用户交互和物品相似度角度验证了推荐的有效性 `[R01]`。随后，矩阵分解和排序学习方法进一步提升了推荐效果，使隐反馈推荐和个性化排序成为重要研究方向 `[R02][R03]`。随着深度学习和图学习的发展，LightGCN 和面向社交推荐的图神经网络综述等研究表明，图结构建模在复杂推荐场景中具有更强的表达能力 `[R05][R10][R13][C03]`。

在知识图谱领域，近年来综述研究系统梳理了知识图谱在表示学习、知识获取与应用落地方面的发展脉络 `[R06][R08][C04]`。围绕知识图谱表示学习与知识推理，研究者提出了 TransE 等经典嵌入模型，并不断将其扩展到推荐场景 `[R04][R11][R21]`。最新研究表明，知识图谱正从通用知识组织逐步走向垂直场景增强、规则约束和多任务协同，这为将其引入推荐系统提供了理论基础 `[R09][R12][C01]`。

基于知识图谱的推荐方法逐渐成为研究热点。综述研究指出，该方向主要围绕知识图谱嵌入、图卷积传播、路径推理和解释生成等路线展开 `[R07][R09][C01][C02]`。其中，一类工作通过知识图谱嵌入增强推荐模型表示能力，例如神经符号知识图谱嵌入、时序图卷积电影推荐和基于评论解释的电影推荐方法，分别从规则约束、时序建模和内容解释等角度扩展了推荐能力 `[R21][R23][R24]`。另一类工作结合图卷积、注意力传播和显式推理路径，将知识图谱中的结构关系用于候选生成与解释生成，例如 KGCN、KGAT 以及面向可解释推荐的图方法 `[R18][R28][R29][R30][C06][C08]`。这些研究说明知识图谱不仅能提升推荐效果，还能增强模型解释性。

总体来看，现有研究已证明知识图谱对推荐系统具有增益作用，但在具体应用场景中仍然面临若干问题。首先，很多研究聚焦于模型本身，缺少与完整业务系统的结合；其次，垂直场景数据采集、图谱构建、离线评估与前端展示往往由不同模块分别处理，缺少端到端案例；再次，推荐效果、系统复杂度和解释能力之间存在明显权衡。基于此，本文选择以系统设计实现为主线，在真实项目背景下完成从数据采集到推荐展示的完整闭环。

### 1.3 研究目标与主要工作

本文的目标是设计并实现一个面向豆瓣电影场景的知识图谱推荐系统，在保证系统完整性的基础上，验证知识图谱在电影推荐和推荐解释中的应用价值。围绕这一目标，本文开展了以下四方面工作：

1. 完成豆瓣电影数据采集、清洗和标准化处理，形成电影与影人基础数据集。
2. 基于 Neo4j 构建电影知识图谱，设计节点、关系、索引与 ETL 流程，实现 MySQL 与 Neo4j 的双库协同。
3. 实现 `content`、`item_cf`、`kg_path`、`kg_embed` 和 `cfkg` 五种推荐算法，并构建推荐解释机制。
4. 搭建基于 FastAPI 和 Vue 的完整系统，完成离线评测、消融分析和线上链路验收。

## 第二章 相关技术与理论基础

### 2.1 知识图谱与属性图模型

知识图谱是一种通过实体、关系和属性组织知识的结构化表示方式，可用于表达现实世界中对象之间的语义关联 `[R06][R08][C04]`。在电影场景中，电影、导演、演员、类型、地区与语言之间天然存在复杂的多对多关系，适合通过图结构进行建模。与传统关系型表结构相比，图结构在表达多跳关系、路径关联和局部邻域时更加直观，也便于后续做图谱查询与可视化展示。

本文采用属性图模型作为电影知识图谱的表达形式。在该模型中，节点代表 `Movie`、`Person`、`Genre`、`Region`、`Language`、`ContentType` 和 `YearBucket` 等实体类型；边用于表达 `DIRECTED`、`ACTED_IN`、`HAS_GENRE`、`IN_REGION`、`IN_LANGUAGE`、`HAS_CONTENT_TYPE` 和 `IN_YEAR_BUCKET` 等关系。属性图模型不仅保留了节点和边的拓扑信息，还允许在节点和关系上携带补充属性，能够满足电影知识图谱的建模需要。

### 2.2 推荐系统基本方法

推荐系统的核心任务是在候选物品集合中为用户排序并返回最有可能感兴趣的内容。基于内容的推荐方法通过比较电影属性与用户历史偏好之间的相似性完成推荐，优点是实现简单、解释直接，但对内容特征依赖较强，在冷启动和长尾场景下存在局限 `[R13][R16][C03]`。协同过滤方法则利用用户与物品交互模式发现隐含偏好，其中 ItemCF 通过物品之间的相似性构建推荐结果，在很多实际场景中具有较好的稳健性 `[R01][R05]`。

在隐反馈场景中，排序学习被广泛用于提升推荐质量。BPR 等方法通过构造用户对正负样本的偏序关系来优化排序结果，为推荐评价中的命中率和排序质量提供了理论支持 `[R03]`。随着深度学习和图学习的发展，图神经网络方法进一步将非线性特征学习和高阶关系建模引入推荐系统，在复杂交互场景中表现出更强的表达能力 `[R05][R10][C03]`。

### 2.3 基于知识图谱的推荐方法

知识图谱增强推荐通常可以分为三类思路。第一类是基于嵌入的方法，通过将实体和关系映射到低维向量空间中，利用结构相似性和语义相似性开展推荐。TransE 是该类方法的典型代表，其核心思想是让头实体向量与关系向量之和接近尾实体向量 `[R04]`。在推荐任务中，这类思想可用于构造电影和实体的向量表示，并将用户偏好映射到统一向量空间中。

第二类是基于图卷积或注意力传播的方法，例如 KGCN、KGAT 等，它们通过邻居聚合和关系加权机制引入高阶语义信息 `[R29][R30][C06][C07]`。这类方法在复杂图结构建模上具有优势，但通常训练与部署复杂度较高。第三类是基于路径推理和显式解释的方法，通过查找用户兴趣电影与候选电影之间的共享导演、演员、类型或多跳路径来形成推荐证据 `[R17][R18][R28][C08][C09]`。该类方法的准确率未必始终优于黑盒模型，但其解释性在实际系统中具有重要价值。

本文实现的 `kg_path` 与 `kg_embed` 分别对应显式路径方法和图谱嵌入方法，而 `cfkg` 则进一步将协同过滤信号与知识图谱信号进行融合，以平衡推荐精度、效率和解释性。

### 2.4 系统开发相关技术

本文系统采用前后端分离架构。后端使用 FastAPI 组织接口与业务逻辑，具备类型标注清晰、接口文档友好和开发效率高等特点；前端使用 Vue 3 与 Vite 进行页面构建，适合完成图谱探索、推荐展示和交互页面开发。数据层面使用 MySQL 与 Neo4j 双库协同，其中 MySQL 负责原始电影数据、用户行为与业务表，Neo4j 负责图谱结构存储、图查询和推荐解释所需的图关系访问。

采集端采用 Playwright 而不是纯请求式爬虫，主要是考虑到目标页面存在动态渲染和反爬策略。通过浏览器自动化方式，可以更稳定地完成电影与影人数据采集，并配合任务锁与代理策略完成多 worker 抓取。这种设计为后续图谱构建和系统扩展提供了稳定的数据来源。

### 2.5 本章小结

本章介绍了知识图谱、推荐系统、图谱增强推荐以及系统实现所依赖的关键技术。相关理论为后续系统设计和实验分析提供了方法基础。下一章将在此基础上进一步说明系统的总体架构、数据处理流程和推荐模块实现方案。

## 第三章 系统设计与实现

### 3.1 需求分析与设计目标

本文系统的设计目标可以概括为三个方面。第一，完成从豆瓣电影数据采集到图谱构建的完整数据链路，为后续推荐实验提供稳定的数据基础。第二，实现多种推荐算法并构建统一的推荐入口，使系统既能开展效果对比，也能支撑实际页面展示。第三，在系统层面提供搜索、详情、图谱探索、个性化推荐和推荐解释等功能，使研究结果能够以可交互的形式展示。

围绕上述目标，系统功能需求包括数据采集、数据清洗、图谱构建、电影与影人查询、图谱关系浏览、推荐结果展示、推荐解释和统计分析等模块。非功能需求则主要包括模块化架构、数据可扩展性、推荐结果可解释性以及基本的系统稳定性。

### 3.2 系统总体架构

系统整体由采集层、数据层、业务层和展示层构成。采集层使用 Playwright 抓取豆瓣电影和影人页面信息；数据层由 MySQL 和 Neo4j 共同组成；业务层基于 FastAPI 封装电影查询、影人查询、图谱探索、推荐与解释接口；展示层基于 Vue 构建电影检索、图谱浏览、推荐页和统计页等功能。

从数据流角度看，完整链路可概括为：

`Playwright 爬虫采集 -> MySQL 原始数据与用户行为 -> 清洗标准化 -> ETL 导入 Neo4j -> FastAPI 提供查询/推荐/解释接口 -> Vue 前端展示`

这种双库协同架构的原因在于，关系型数据库更适合存储原始数据、用户行为和权限相关业务表，而图数据库更适合多跳关系访问、路径推理和图可视化。二者分工明确，避免单一数据库同时承担所有职责而导致结构混乱。

### 3.3 数据采集与预处理

在数据采集阶段，系统首先基于排行榜、标签页与详情页发现电影条目，再进一步扩展至影人详情页。初期方案曾尝试使用 Scrapy 进行请求式采集，但在处理动态渲染与反爬限制时稳定性不足，因此后续重构为基于 Playwright 的浏览器自动化方案。重构后的采集模块支持多 worker 并发、任务锁、失败重试和代理切换，更适合中大规模抓取任务。

在数据预处理阶段，系统主要对 `genres`、`regions`、`languages` 等多值字段进行标准化处理，对空字符串、异常评分和无效字段进行清洗，并检查 `actor_ids` 和 `director_ids` 的格式完整性。根据当前数据质量报告，Movies 表中已整理 `192,032` 条记录，Persons 表中已整理 `236,882` 条记录。评分、剧情简介、别名和片长等字段存在较高缺失率，但电影名称、年份、地区、语言等核心结构字段质量较好，能够满足图谱构建与推荐实验的基本要求。

### 3.4 知识图谱构建与双库协同

图谱构建阶段采用属性图模型，将电影、影人和类型作为核心实体，同时引入地区、语言、内容类型和年代桶等扩展结构节点。节点之间通过 `DIRECTED`、`ACTED_IN`、`HAS_GENRE` 等关系连接，从而形成面向电影推荐场景的知识图谱。当前预估图谱关系规模约为 `1,890,704` 条，其中 `ACTED_IN` 关系数量最多，其次为 `DIRECTED` 和 `HAS_GENRE`。

在实现上，系统先从 MySQL 中读取清洗后的电影基础字段，再通过 ETL 过程写入 Neo4j，并在图数据库中建立节点唯一约束和辅助索引，以保证结构一致性与查询性能。MySQL 与 Neo4j 的协同方式如下：

- MySQL 负责电影、影人原始字段、用户表和行为表；
- Neo4j 负责图关系存储、图谱探索和推荐解释相关查询；
- 推荐模块通过共享图谱缓存访问电影画像、导演演员关系和类型信息。

这种设计既保留了关系型数据库对业务数据的管理优势，又发挥了图数据库在路径访问和邻域查询方面的优势。

### 3.5 推荐算法设计与解释机制

为了验证知识图谱在推荐场景中的作用，系统实现了五种推荐方法：基于内容的推荐、基于物品的协同过滤、基于知识图谱路径的推荐、基于知识图谱嵌入的推荐以及融合式 `CFKG` 主链路推荐。

`content` 方法根据电影元数据与用户历史偏好构建相似性评分，适合作为轻量级基线和冷启动兜底方案；`item_cf` 方法通过用户历史正反馈电影与候选电影之间的相似性完成推荐，在当前实验中表现最稳定；`kg_path` 方法利用共享导演、演员、类型及扩展结构路径对候选电影进行打分，并能直接输出路径证据；`kg_embed` 方法基于 TransE 思想将图谱实体与关系映射到向量空间中，再通过用户兴趣电影嵌入聚合得到偏好向量，用以计算候选分数。

`cfkg` 方法作为系统主链路，并不试图让单一模型包揽全部任务，而是将 `item_cf`、`kg_embed` 和 `kg_path` 三路信号进行融合。其中，`item_cf` 负责协同信号，`kg_embed` 负责潜在语义关联，`kg_path` 负责显式解释和图结构证据。当用户行为较少时，系统还会使用内容方法进行兜底。此设计的出发点不是追求单一指标绝对最优，而是在准确率、多样性、解释性和可部署性之间寻找平衡。

在推荐解释方面，系统会优先查找目标推荐电影与用户历史兴趣种子之间的共享导演、演员、类型等证据。如果存在明确重叠关系，则构建解释图；如果没有足够重叠证据，则退化为展示目标电影自身的上下文信息。该机制为系统页面展示提供了可读性较强的推荐理由。

### 3.6 前后端系统实现

后端以 FastAPI 为核心，按路由层与服务层进行组织。路由层负责参数接收与响应输出，服务层负责具体业务逻辑。系统提供电影查询、影人查询、图谱探索、统计分析、用户行为与推荐相关接口，项目整理文档显示当前后端接口总数为 `58`。前端基于 Vue 3 构建电影搜索、详情展示、图谱浏览、个性化推荐、统计图表等页面，当前项目文档显示页面总数为 `13`。

前后端联动后，用户可在系统中完成电影搜索、影人查看、图谱探索和推荐结果浏览等操作。对于推荐结果，用户不仅能看到推荐列表，还能打开解释抽屉查看共享导演、演员和类型等图谱化证据。这一设计增强了系统展示性，也使论文中的“可解释推荐”不再停留于算法描述层面。

### 3.7 本章小结

本章从需求分析、系统架构、数据采集、知识图谱构建、推荐算法设计和前后端实现六个方面介绍了本文系统的设计与实现过程。通过 MySQL 与 Neo4j 双库协同、五种推荐算法对照和图谱化解释机制，系统完成了从数据到应用的完整链路，为后续实验与结果分析奠定了基础。

## 第四章 实验与结果分析

### 4.1 实验环境

本文实验在本地开发环境完成，主要软件栈包括 Python 3.11、FastAPI、Vue 3、MySQL 8.x、Neo4j 5.x 以及相关算法依赖。数据采集阶段使用 Playwright 进行浏览器自动化抓取，图谱构建阶段通过 ETL 将清洗后的关系型数据导入 Neo4j。推荐算法模块基于统一的推荐接口实现，离线评测通过评估脚本统一生成报告。

硬件层面，本文实验采用个人开发主机与 Docker 化图数据库环境。考虑到本文研究重点在系统设计与实现，实验目标主要是验证推荐链路的可用性与对比效果，而非追求分布式训练规模。

### 4.2 数据集与评估协议

实验数据来源于豆瓣电影与影人页面。根据当前数据质量报告，Movies 表共包含 `192,032` 条记录，Persons 表共包含 `236,882` 条记录，不重复影人 ID 数为 `236,886`，电影类型数为 `32`。在图谱结构层面，当前预估关系总数为 `1,890,704`，其中 `ACTED_IN`、`DIRECTED` 和 `HAS_GENRE` 是最主要的关系类型。

推荐算法离线评估采用 `validation_and_test_sampled_leave_one_out (1 positive + 499 negatives)` 协议，对公开豆瓣用户数据进行验证与测试。具体设置如下：

- 用户来源：公开豆瓣用户
- 负采样随机种子：`[42, 52, 62, 72, 82]`
- 验证集用户数：`100`
- 测试集用户数：`400`
- 主要评价指标：`HR@K`、`NDCG@K`
- 补充指标：`Coverage@20`、`Diversity@10`、平均测试耗时

由于每个用户仅保留一个测试正样本，因此本实验中 `Recall@K` 与 `HR@K` 数值恒等。正文中可重点展示更常见的命中率与排序质量指标，并辅以覆盖率、多样性与耗时进行综合讨论。关于推荐系统评价指标的选择与解释，也可参考中文综述文献中的常见划分方式 `[C05]`。

### 4.3 离线评测结果

在五种推荐方法的横向比较中，系统当前结果如表 4-2 所示。以 `K=10` 为例，`CFKG` 的 `HR@10` 为 `0.7910`，`NDCG@10` 为 `0.5240`；`ItemCF` 的 `HR@10` 为 `0.6935`，`NDCG@10` 为 `0.4522`；`KG-Embed` 的 `HR@10` 为 `0.7405`，`NDCG@10` 为 `0.4800`；`KG-Path` 的 `HR@10` 为 `0.5200`，`NDCG@10` 为 `0.3972`；基于内容的方法表现最低。

从更完整的结果看，`CFKG` 在 `K=5`、`K=10` 和 `K=20` 下均表现出较高的命中率和排序质量，但如果只看单一方法的主观工程价值，当前项目文档的统一口径仍是：`ItemCF` 整体表现最稳，`KG-Embed` 是当前效果最好的图谱方法，`CFKG` 是系统主链路。这样表述更适合论文答辩和正文分析，因为它同时体现了工程取舍与实验事实。

在覆盖率、多样性和耗时方面，不同方法体现出明显差异。`ItemCF` 和 `KG-Path` 在多样性指标上更高，但 `KG-Path` 的平均测试耗时显著高于其他方法；`KG-Embed` 在图谱方法中兼顾了准确率与效率；`CFKG` 的平均主测试耗时约为 `9.3891s`，虽然高于传统方法，但仍能作为系统主链路进行展示。

### 4.4 消融实验与结果分析

为进一步分析各模块贡献，本文对 `CFKG`、`KG-Path` 和 `KG-Embed` 进行了关键消融实验。

对于 `CFKG`，当只使用 `KG-Embed` 直排时，`HR@10` 为 `0.7415`；加入 `ItemCF` 线性辅助后，`HR@10` 提升至 `0.7910`，说明协同过滤信号对主链路准确率提升具有显著作用。加入 `KG-Path` 解释机制后，主指标未进一步提升，但解释能力增强，说明路径模块在当前系统中的主要价值是解释而非主排序加分。

对于 `KG-Path`，只保留结构路径时，`HR@10` 为 `0.2165`；加入协同用户路径后，`HR@10` 提升至 `0.5185`；开启度惩罚、扩展关系和用户行为路径后，指标进一步提升至 `0.5200`。这说明显式路径方法在结合协同信号与路径筛选后，能够获得更稳定的效果。

对于 `KG-Embed`，仅使用结构嵌入时，`HR@10` 为 `0.2760`；加入用户三元组后，`HR@10` 提升至 `0.7320`；进一步启用融合排序后，指标提升至 `0.7405`。这表明用户行为与扩展关系对嵌入方法十分关键，单纯依赖图结构本身难以获得较好推荐效果。

### 4.5 定性分析与线上链路验收

除了离线指标外，本文还从系统展示和线上链路稳定性两个方面进行定性分析。系统前端提供推荐结果抽屉、图谱探索与详情页展示，推荐解释模块能够在共享导演、演员或类型存在时构造图谱化解释证据。对于实际展示型系统而言，这类可解释输出有助于增强结果可读性，也便于答辩演示时直观说明推荐逻辑。

在线路验收方面，`2026-04-12` 的 `KG` 线上链路验收共覆盖 `46` 个场景，全部通过，未出现失败场景。该结果说明当前系统主链路在接口、算法和页面联动方面已经具备较好的可用性，能够支撑论文中的系统实现与验证章节。

### 4.6 结果讨论

综合实验结果可以得到以下几点认识。第一，协同过滤在当前数据条件下仍然具有强竞争力，说明用户行为信号对电影推荐任务具有较高价值。第二，知识图谱方法在推荐任务中的主要优势不只是“是否超过传统方法”，还体现在显式语义建模、结构化扩展和解释能力增强。第三，融合式主链路虽然未必在所有场景都显著优于单一方法，但在系统层面更符合实际需求，因为它兼顾了多种信息源。

同时，当前系统仍然存在一些局限。其一，数据采集阶段存在部分字段缺失，对基于内容和排序解释存在影响；其二，`KG-Path` 的耗时偏高，难以直接用于高频主排序；其三，论文当前实验主要基于离线评估与链路验收，尚缺少更长周期的在线用户反馈验证。这些问题也是后续改进的重点方向。

### 4.7 本章小结

本章从实验环境、数据规模、评估协议、离线结果、消融实验、定性分析和线上验收等方面对系统进行了验证。结果表明，本文所实现的电影知识图谱推荐系统能够有效支撑电影推荐场景，知识图谱在推荐效果增强和解释生成方面均具有实际价值。

## 第五章 总结与展望

### 5.1 工作总结

本文围绕豆瓣电影场景，完成了一个基于知识图谱的电影推荐系统设计与实现。全文工作主要包括以下四个方面：完成数据采集、清洗和标准化处理；构建电影知识图谱并设计双库协同架构；实现五种推荐算法与推荐解释机制；完成前后端系统搭建、离线评估、消融分析与线上验收。通过这些工作，本文验证了知识图谱在电影推荐系统中的工程可行性和应用价值。

### 5.2 研究不足

尽管系统已经形成较完整的功能闭环，但仍存在一些不足。首先，豆瓣原始数据中的评分、剧情、别名和片长等字段缺失较多，限制了内容特征的进一步利用。其次，当前系统主链路仍偏向离线效果与展示效果验证，在响应时间和在线闭环上还有优化空间。再次，参考文献、图表格式和模板化排版仍需继续规范，以满足学校最终提交要求。

### 5.3 后续展望

后续工作可从三个方向展开。第一，进一步提升图谱质量与用户行为建模能力，例如补充更多结构实体、优化清洗规则并增强用户偏好分层。第二，围绕 `CFKG` 主链路继续优化召回与重排效率，降低图谱方法耗时，提高系统实用性。第三，完善论文模板化排版、文献格式和图表表达，并在条件允许时补充更多在线反馈或用户侧验证结果，使系统评估更全面。
