# 🎥 电影库排序算法优化总结

## 📌 问题背景

原系统的电影库默认按评价**平均分(`rating`)**直接进行降序排列。
这导致部分评价人数极少但打分极高的冷门作品（甚至是非真实影视的条目，如小众演唱会录影）占据了排行榜最前列，而几百万人打分、被奉为经典的《肖申克的救赎》《霸王别姬》等反而被挤到了后面。这与用户预期的“高分必看榜单”体验相去甚远。

## 🔬 解决方案：贝叶斯加权平滑 (Weighted Rating, WR)

为了解决这个问题，我们调研并引入了 IMDb 和 豆瓣 Top250 所使用的经典**贝叶斯加权算分公式**：

> **`WS = (v × R + m × C) / (v + m)`**

### 参数释义与确定

我们在项目的 MySQL 全库数据中进行了统计，最终确立了极其“硬核”的平滑参数，使得榜单前列与真实的豆瓣榜单（含高分剧集）表现一致：

- **`v`**：该电影的评分人数 (`votes`)。
- **`R`**：该电影的纯均分 (`rating`)。
- **`C`** (全库平均分先验)：**`7.0`**（系统全库统计出的均分约为 7.06）。
- **`m`** (最小票数门槛)：**`50000`**。
  _注：由于系统中的头部影视票数最高达到了 325万（肖申克），且前 5% 的影视票数线为 4.6万。若将 `m` 设为 1000 仍会使得 5000票的 `9.8` 分视频高于 300万票的 `9.7` 分经典。最终实验得出，将 `m` 设为 `50000` 能完美制衡分数偏差，让大众公认的经典排在最前面。_

## 🛠️ 实施细节

### 1. 数据库支持 (`etl_to_neo4j.py`)

由于爬虫原本的 MySQL 表 `movies` 中存在 `douban_votes`，但没有被导入 Neo4j 图库，导致图查询无法用人数加权。

- 修改了 `etl_to_neo4j.py` 脚本，在读取时包含了 `douban_votes` 字段。
- 在 `MERGE` 的 Cypher 语句中新增：`SET movie.votes = m.votes`。
- **全量重跑了 ETL** 脚本同步数据，使超过 9.4 万部电影在图谱中获得了 `votes` 属性。

### 2. 后端核心逻辑 (`movie_service.py` & `routers/movies.py`)

- 在获取最高分榜单和电影筛选库请求中，增加了额外的控制参数 `sort_by`。
- Cypher 查询在 `sort_by=weighted` 模式时，使用公式直接在库端进行计算并排序：

```cypher
MATCH (m:Movie) WHERE m.rating IS NOT NULL
WITH m, (coalesce(m.votes, 0) * m.rating + 50000 * 7.0) / (coalesce(m.votes, 0) + 50000) AS score
ORDER BY score DESC
```

- 为了防止部分全冷门内容返回出现 Null，利用了 `coalesce` 将空 `votes` 视为 `0`（退化为先验平均分 7.0）。

### 3. 前端界面交互 (`MovieFilterView.vue`)

- 在影片列表顶部加入了**排序选项卡**（`el-select` 下拉选择器）。
- 提供三种排序选项：
    1. 🏆 加权推荐（默认 `weighted`）
    2. ⭐ 评分最高 (`rating`)
    3. 💬 最多人评 (`votes`)
- 绑定了 API 调用动态切换，提升了用户体验，让用户可以在探索“绝对经典”和“小众高分”中自由切换。

## 🎬 扩展：内容形式 (content_type) 筛选

针对系统中不仅有电影，还有电视剧、综艺、纪录片等情况，我们在电影库中加入了按**内容形式**筛选的功能。

- **后端改动**：在 `movie_service.py` 和 `routers/movies.py` 的 `/filter` 接口新增 `content_type` 参数，并在 Cypher 语句中拼接 `m.content_type = $content_type` 条件。
- **前端改动**：在 `MovieFilterView.vue` 加入 `el-radio-group`，提供 `全部`、`电影 (Movie)`、`剧集 (TV)` 三个单选框。用户点击即可筛选出纯电影榜单或纯电视剧榜单。
