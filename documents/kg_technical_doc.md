## 1. 总体技术架构

本系统采用分层架构，将整个知识图谱的生命周期划分为"数据采集 → 数据处理 → 知识建模 → 图谱存储 → 应用服务"五个阶段。

```
┌────────────────────────────────────────────────────────┐
│                    应用层（待实现）                       │
│      Vue3 前端  +  FastAPI 后端  +  推荐算法引擎         │
├────────────────────────────────────────────────────────┤
│                    知识存储层                            │
│         Neo4j 图数据库（属性图模型，Cypher 查询）         │
├────────────────────────────────────────────────────────┤
│                    知识建模层                            │
│   本体设计（Movie / Person / Genre）+ 三元组抽取         │
├────────────────────────────────────────────────────────┤
│                    数据处理层                            │
│    数据质量分析  →  数据清洗 Pipeline  →  ETL 导入        │
├────────────────────────────────────────────────────────┤
│                    数据采集层                            │
│   Playwright 并发爬虫 + 代理IP池 + 分布式任务调度        │
├────────────────────────────────────────────────────────┤
│                    数据源                               │
│             豆瓣电影网站（movie.douban.com）              │
└────────────────────────────────────────────────────────┘
```

### 1.1 技术栈选型

| 层次        | 技术选型                                  | 选型理由                                                                               |
| ----------- | ----------------------------------------- | -------------------------------------------------------------------------------------- |
| 数据采集    | Python + Playwright                       | 豆瓣使用大量动态渲染，Playwright 可以模拟真实浏览器行为，绕过 JS 渲染限制              |
| 反爬对抗    | 代理 IP 池 + User-Agent 轮换 + BID Cookie | 应对豆瓣的 IP 频率限制、浏览器指纹检测和 Cookie 鉴权                                   |
| 关系型存储  | MySQL 5.7（utf8mb4）                      | 作为数据采集的中间存储，支持结构化管理和 SQL 查询                                      |
| 数据处理    | Pandas + PyMySQL                          | 成熟的数据分析与清洗工具链                                                             |
| 图数据库    | Neo4j 5（Docker 部署）                    | 业界最成熟的属性图数据库，支持 Cypher 声明式查询语言，免索引邻接保证常数级关系遍历性能 |
| Python 驱动 | neo4j（官方驱动）                         | 支持 Bolt 协议，提供事务管理和批量操作能力                                             |

## 2. 数据采集层设计

### 2.1 采集目标与范围

本系统从豆瓣电影网站采集两大类实体数据：

| 实体类型       | 目标网页                           | 采集字段                                                                  | 最终数据量     |
| -------------- | ---------------------------------- | ------------------------------------------------------------------------- | -------------- |
| 电影（Movie）  | `movie.douban.com/subject/{id}/`   | 名称、评分、类型、导演、演员、地区、语言、时长、上映日期、简介等 20+ 字段 | **135,990 条** |
| 影人（Person） | `movie.douban.com/celebrity/{id}/` | 姓名、性别、出生日期、出生地、职业、个人简介等 10 个字段                  | **198,067 条** |

### 2.2 采集架构

爬虫系统采用**两阶段 BFS（广度优先搜索）**策略进行数据发现与采集：

**阶段一：种子发现（Seed Discovery）**

1. 从豆瓣电影分类页获取初始电影 ID 集合，存入 `subjects` 表。
2. 在采集每部电影详情页时，同时提取页面底部的"相关推荐"电影 ID，作为新的种子回写到 `subjects` 表。
3. 通过 BFS 不断扩展待采集队列，最终积累了 **13.5 万+** 部电影的 ID。

**阶段二：详情采集**

对 `subjects` 表中的每个 `douban_id`，分别采集电影详情页和影人详情页。

### 2.3 反爬虫对抗策略

豆瓣网站部署了多层反爬虫机制，本系统采取了以下应对措施：

| 反爬措施        | 豆瓣行为                                  | 应对方案                                                     |
| --------------- | ----------------------------------------- | ------------------------------------------------------------ |
| IP 频率限制     | 单 IP 短时间内大量请求会触发 HTTP 403/429 | 构建代理 IP 池，按顺序轮换使用，单个 IP 冷却周期 ≥5 分钟     |
| 浏览器指纹检测  | 检测 User-Agent 是否为真实浏览器          | 从 9 种主流浏览器 UA 中随机选取，每个 Worker 独立 UA         |
| Cookie 鉴权     | 需要有效的 `bid` Cookie 才能访问详情页    | 程序自动生成随机 11 位字母数字组成的 BID                     |
| JavaScript 渲染 | 部分页面内容依赖 JS 动态加载              | 使用 Playwright（非 requests）进行全浏览器渲染               |
| 流量异常检测    | 请求模式过于规律会触发验证码              | 请求间隔采用随机延迟（2-5 秒），每爬取 200 部电影休息 3 分钟 |

### 2.4 分布式并发设计

为了提升采集效率，系统支持多 Worker 并发和跨机器分布式部署：

```
┌─────────────────────────────────────────┐
│              MySQL subjects 表           │
│   （全局任务队列，标记 crawl_status）     │
├─────────────────┬───────────────────────┤
│  Machine A      │     Machine B         │
│  ┌──────────┐   │   ┌──────────┐        │
│  │ Worker 1 │   │   │ Worker 1 │        │
│  │ Worker 2 │   │   │ Worker 2 │        │
│  │ Worker 3 │   │   │ Worker 3 │        │
│  └──────────┘   │   └──────────┘        │
└─────────────────┴───────────────────────┘
```

**任务锁定机制**：

- `subjects` 表增设 `crawl_status` 字段（0=待爬, 1=爬取中, 2=已完成, 3=失败）
- Worker 在领取任务时使用 `SELECT ... FOR UPDATE SKIP LOCKED` 实现悲观锁，避免多个 Worker 重复爬取同一部电影
- 超时释放机制：任务锁定超过 30 分钟自动重置为 0（应对 Worker 崩溃场景）

## 3. 数据清洗与预处理

原始爬取的网页数据不可避免地存在格式不统一、缺失值、异常值等问题。在将数据导入知识图谱之前，需要经过系统性的清洗和标准化处理。

### 3.1 清洗策略设计

本系统采用**原地清洗（In-Place Cleaning）**策略，直接在 MySQL 中通过 SQL UPDATE 语句执行清洗操作，避免了数据导出导入的开销。

清洗 Pipeline 分为以下 7 个步骤：

| 步骤 | 操作                            | 目标                         | 清洗记录数           |
| ---- | ------------------------------- | ---------------------------- | -------------------- |
| 1    | 名称去空格                      | 去除 `name` 字段首尾多余空格 | movies: 0, person: 0 |
| 2    | genres 分隔符统一               | `" / "` → `"/"` 格式规范化   | 0（原数据已规范）    |
| 3    | regions 分隔符统一              | `" / "` → `"/"` 格式规范化   | **11,852 条**        |
| 4    | languages 分隔符统一            | `" / "` → `"/"` 格式规范化   | **11,824 条**        |
| 5    | 评分异常处理                    | 评分为 0 的记录设为 NULL     | 0（无异常）          |
| 6    | 空字符串统一                    | `""` → `NULL`，消除歧义      | **2,633 条**         |
| 7    | actor_ids/director_ids 格式校验 | 验证 `姓名:ID` 格式正确性    | 0 条异常             |

### 3.2 分隔符标准化

爬虫从网页中提取的类型、地区、语言等"多值合一"字段，在不同页面中使用了不一致的分隔符。例如：

- 有些页面使用 `"剧情 / 喜剧"`（两端有空格）
- 有些使用 `"剧情/喜剧"`（无空格）

本系统将所有多值字段统一为 **`/` 无空格** 的分隔格式，确保后续切分不会产生含空格的脏值。

### 3.3 缺失值处理策略

对不同字段采用了差异化的缺失值处理策略：

| 字段类别 | 字段                         | 处理方式             | 理由                                       |
| -------- | ---------------------------- | -------------------- | ------------------------------------------ |
| 核心标识 | `name`, `douban_id`          | 不允许为空（无缺失） | 图谱节点的唯一标识和显示名称               |
| 图谱属性 | `douban_score`               | 保留 NULL            | 30.5% 缺失，这些电影确实无评分（冷门作品） |
| 图谱关系 | `actor_ids`, `director_ids`  | 保留 NULL            | 部分短片、纪录片确实没有标注演员/导演      |
| 辅助信息 | `storyline`, `alias`, `mins` | 保留 NULL            | 缺失率较高但不影响图谱核心关系             |
| 无效字段 | `imdb_id`, `official_site`   | 忽略                 | 缺失率 > 99%，爬虫未有效提取               |

## 4. 知识图谱本体设计

本体设计（Ontology Design）是知识图谱构建的核心环节，它定义了图谱中的实体类型、属性和关系，决定了知识的组织结构和表达能力。

### 4.1 设计原则

本系统在本体设计过程中遵循以下原则：

1. **语义完整性**：确保图谱能够完整表达电影领域的核心知识。
2. **适度抽象**：只将对推荐算法和语义查询有价值的信息建模为独立节点或关系，避免图谱过度膨胀。
3. **消歧优先**：在建模设计阶段就解决实体消歧问题，而非事后补救。
4. **可扩展性**：预留扩展接口（如 Person 节点的 `birth`、`death` 属性），支持未来功能增长。

### 4.2 节点类型设计

本图谱定义了三种节点类型（Label）：

#### 4.2.1 Movie 节点（电影）

**设计依据**：电影是本图谱的核心实体，承载了所有的属性信息和关系连接。

| 属性           | Neo4j 类型 | 说明                 | 设计理由                       |
| -------------- | ---------- | -------------------- | ------------------------------ |
| `mid` (**PK**) | String     | 豆瓣 ID              | 全局唯一标识符，保证实体不重复 |
| `title`        | String     | 电影名称（含中英文） | 用户检索的主要入口             |
| `rating`       | Float      | 豆瓣评分 (0-10)      | 推荐算法的核心权重因子         |
| `year`         | Integer    | 上映年份             | 时间维度筛选                   |
| `content_type` | String     | `movie` / `tv`       | 区分电影与电视剧               |
| `release_date` | String     | 具体上映日期         | 可用于时间序列分析             |
| `cover`        | String     | 封面图 URL           | 前端展示用                     |
| `url`          | String     | 豆瓣详情页链接       | 数据溯源                       |
| `regions`      | String     | 制片国家/地区        | 地域维度属性                   |
| `languages`    | String     | 语言                 | 语言维度属性                   |
| `runtime`      | Integer    | 片长（分钟）         | 筛选条件                       |
| `storyline`    | String     | 剧情简介             | 基于内容推荐的文本特征         |
| `alias`        | String     | 别名                 | 搜索别名匹配                   |

> **设计决策**：将 `regions`（地区）和 `languages`（语言）作为 Movie 节点的 **内嵌属性** 而非独立节点。理由如下：
>
> - 如果将"中国大陆"、"美国"等建模为独立的 `Country` 节点，图谱将增加约 100 个国家节点和 **14 万+** 条 `PRODUCED_IN` 关系。但这些关系对推荐算法的贡献极低——用户很少因为"这也是中国大陆拍的"产生兴趣关联。
> - 相比之下，将其作为属性存储，可以通过 `WHERE m.regions CONTAINS '美国'` 轻松实现筛选，同时大幅减少了图谱的关系数量，提升查询性能。

#### 4.2.2 Person 节点（影人）

**设计依据**：影人（导演、演员）是连接不同电影的核心桥梁，是基于图结构推荐算法（如 Personalized PageRank）发挥作用的关键。

| 属性           | Neo4j 类型 | 说明             | 设计理由                       |
| -------------- | ---------- | ---------------- | ------------------------------ |
| `pid` (**PK**) | String     | 豆瓣影人 ID      | 唯一标识符，解决实体消歧的核心 |
| `name`         | String     | 姓名（含中英文） | 显示和检索                     |
| `sex`          | String     | 性别             | 人物属性                       |
| `birth`        | String     | 出生日期         | 人物属性                       |
| `death`        | String     | 去世日期         | 人物属性                       |
| `birthplace`   | String     | 出生地           | 人物属性                       |
| `profession`   | String     | 职业             | 区分演员/导演/编剧等多重身份   |
| `biography`    | String     | 个人简介         | 基于内容推荐的文本特征         |

> **核心设计决策：统一的 Person 节点，而非拆分为 Actor 和 Director**
>
> 这是本系统本体设计中最重要的决策之一。在影视领域，同一人常兼任多种角色。例如：
>
> - 姜文既是演员也是导演（参演并执导《让子弹飞》）
> - 周星驰既是演员、导演又是编剧
> - 赵薇同时是演员和导演
>
> 如果将 Actor 和 Director 设计为两种独立节点，那么姜文在图谱中将有两个不同的节点，导致：
>
> 1. **知识割裂（Knowledge Fragmentation）**：两个"姜文"节点之间没有直接关联，无法反映"同一个人"的语义。
> 2. **推荐精度下降**：基于图遍历的推荐算法无法发现"姜文导演的电影"和"姜文主演的电影"之间的隐性关联。
>
> **本系统的解决方案**：使用统一的 `Person` 节点，通过**关系类型**（`DIRECTED` vs `ACTED_IN`）来区分角色，而非通过节点类型区分。这样，姜文在图谱中只有一个节点，同时拥有指向不同电影的 `DIRECTED` 和 `ACTED_IN` 两种关系，完整保留了语义。

> **实体消歧策略**
>
> 实体消歧（Entity Disambiguation）是知识图谱构建中的经典难题——如何区分同名的不同实体？
>
> 本系统在数据采集阶段就从根源解决了这个问题：爬虫在提取演员和导演信息时，不仅提取了姓名，还同时提取了豆瓣官方分配的 **`person_id`**（影人唯一 ID）。这个 ID 存储在 `actor_ids` 和 `director_ids` 字段中，格式为 `"姓名:ID"`。
>
> 因此，即使有多个叫"张伟"的演员，他们在图谱中也会因为不同的 `pid` 而被正确区分为独立的 Person 节点。这种方案在学术上称为**基于权威标识符的实体对齐（Authority-ID-based Entity Alignment）**，准确率理论上为 100%。
>
> ETL 阶段的抽样验证也证实了这一点：从 movies 表提取的 198,071 个唯一影人 ID，在 person 表中的**匹配率为 100%**。

#### 4.2.3 Genre 节点（类型）

**设计依据**：电影类型是推荐系统中最直观的内容标签，也是基于内容推荐算法的基础特征维度。

| 属性            | Neo4j 类型 | 说明     |
| --------------- | ---------- | -------- |
| `name` (**PK**) | String     | 类型名称 |

本图谱共提取出 **32 种** 电影类型，涵盖了豆瓣电影的全部分类体系：

```
剧情、喜剧、爱情、纪录片、动画、短片、动作、惊悚、犯罪、
悬疑、奇幻、恐怖、真人秀、冒险、科幻、家庭、古装、历史、
战争、同性、音乐、传记、武侠、情色、灾难、西部、运动、
歌舞、脱口秀、鬼怪、儿童、黑色电影
```

### 4.3 关系类型设计

本图谱定义了三种关系类型（Relationship Type）：

#### 4.3.1 DIRECTED（执导）

```
(:Person)-[:DIRECTED]->(:Movie)
```

- **语义**：影人 P 执导了电影 M
- **属性**：无附加属性
- **数据来源**：从 `movies.director_ids` 字段解析 `"姓名:ID"` 对，建立 (Person)-[:DIRECTED]->(Movie) 关系

#### 4.3.2 ACTED_IN（参演）

```
(:Person)-[:ACTED_IN {order: Integer}]->(:Movie)
```

- **语义**：影人 P 参演了电影 M
- **附加属性**：`order`（番位/演员表排序，1 = 主演）
- **数据来源**：从 `movies.actor_ids` 字段解析，`|` 分隔的顺序即为 `order` 值

> **边属性设计理由**：`order`（番位）属性为后续推荐算法提供了**权重依据**。在基于图遍历的推荐中，主演（`order=1`）对电影风格的"代表性"远高于配角（`order=10`）。可以据此为 ACTED_IN 关系赋予不同的权重，提升推荐精度。

#### 4.3.3 HAS_GENRE（属于类型）

```
(:Movie)-[:HAS_GENRE]->(:Genre)
```

- **语义**：电影 M 属于类型 G
- **属性**：无附加属性
- **数据来源**：从 `movies.genres` 字段按 `/` 切分

## 5. 知识抽取与三元组构建

知识抽取（Knowledge Extraction）是将 MySQL 中的半结构化数据转化为知识图谱的三元组（Triple）表示。

### 5.1 实体抽取

由于本系统的数据来源为结构化的 MySQL 表，实体抽取采用**基于规则的确定性抽取**（而非 NLP 的概率性抽取），准确率可达 100%。

| 实体类型 | 抽取来源                     | 抽取方式                        | 数量    |
| -------- | ---------------------------- | ------------------------------- | ------- |
| Movie    | `movies` 表每条记录          | 1 条记录 → 1 个 Movie 实体      | 135,990 |
| Person   | `person` 表每条记录          | 1 条记录 → 1 个 Person 实体     | 198,067 |
| Genre    | `movies.genres` 字段全局去重 | 按 `/` 切分 → 去重 → Genre 实体 | 32      |

### 5.2 关系抽取

关系抽取的核心挑战在于解析 `actor_ids` 和 `director_ids` 字段的嵌套格式。

**字段格式示例**：

```
actor_ids = "蒂姆·罗宾斯 Tim Robbins:1054521|摩根·弗里曼 Morgan Freeman:1054534|..."
director_ids = "弗兰克·德拉邦特 Frank Darabont:1047973"
```

**解析算法**：

```python
# 以 actor_ids 为例
for movie in movies:
    actors = movie['actor_ids'].split('|')       # 按 | 切分为多个演员
    for order, item in enumerate(actors, 1):      # order 为番位
        name, pid = item.rsplit(':', 1)           # 按最后一个 : 切分出姓名和 ID
        # 构建三元组: (Person:pid) -[ACTED_IN {order}]-> (Movie:mid)
```

### 5.3 三元组统计

生成的三元组分布如下：

| 三元组模式                    | 数量          | 平均每部电影            |
| ----------------------------- | ------------- | ----------------------- |

| (Person)-[:ACTED_IN]->(Movie) | 839,923       | ~6.17                   |

## 6. 知识存储与图数据库选型

### 6.1 为什么选择图数据库而非关系型数据库？

知识图谱的核心价值在于**关系的表达和遍历**。以一个典型的推荐查询为例："查找与《肖申克的救赎》有共同演员的电影"。

**MySQL 实现**（需要复杂的多表 JOIN）：

```sql
SELECT DISTINCT m2.name
FROM movies m1
JOIN (/* 需要复杂的字符串解析 actor_ids */) a1 ON ...
JOIN (/* 再次解析 */) a2 ON a1.person_id = a2.person_id
JOIN movies m2 ON a2.douban_id = m2.douban_id
WHERE m1.name = '肖申克的救赎';
```

**Neo4j 实现**（声明式图遍历）：

```cypher
MATCH (m1:Movie {title: '肖申克的救赎'})<-[:ACTED_IN]-(p:Person)-[:ACTED_IN]->(m2:Movie)
RETURN DISTINCT m2.title;
```

| 对比维度     | MySQL（关系型）                     | Neo4j（图数据库）              |
| ------------ | ----------------------------------- | ------------------------------ |
| 多跳关系查询 | 需要多层 JOIN，复杂度随跳数指数增长 | 图遍历，复杂度仅与路径长度相关 |
| 关系表达     | 通过外键隐式表达                    | 关系是一等公民，物理存储       |
| 查询语言     | SQL（面向表）                       | Cypher（面向图模式匹配）       |
| 性能         | 2-3 跳查询可能需要数秒              | 同样查询毫秒级                 |

### 6.2 Neo4j 的免索引邻接（Index-Free Adjacency）

Neo4j 采用**免索引邻接**存储架构。每个节点物理上都保存着其相邻节点的直接指针，而不是像关系型数据库那样通过索引中间表来查找。这意味着：

- 遍历一条关系的时间复杂度为 **O(1)**，与图谱总规模无关。
- 对于包含 ~33 万节点和 ~120 万关系的图谱，多跳查询依然可以在毫秒级完成。

## 7. ETL 导入实现

ETL（Extract-Transform-Load）脚本负责将 MySQL 中清洗后的数据转化为图结构并加载到 Neo4j。

### 7.1 导入流程

整个 ETL 过程分为 7 个有序阶段：

```
Phase 1: 创建约束和索引
   ↓
Phase 2: 导入 Genre 节点 （32 个）
   ↓
Phase 3: 导入 Movie 节点 （135,990 个）
   ↓
Phase 4: 导入 Person 节点 （198,067 个）
   ↓
Phase 5: 创建 DIRECTED 关系 （105,757 条）
   ↓
Phase 6: 创建 ACTED_IN 关系 （839,923 条）
   ↓
Phase 7: 创建 HAS_GENRE 关系 （253,252 条）
   ↓
验证阶段: 统计节点/关系数量，执行示例查询
```

**设计要点**：必须先导入所有节点（Phase 2-4），再创建关系（Phase 5-7）。因为关系的创建依赖于两端节点的存在（通过 `MATCH` 匹配）。

### 7.2 批量导入优化

面对 13 万+节点和 120 万+关系的导入量，逐条 `CREATE` 的性能极差。本系统采用 Neo4j 的 **`UNWIND` 批量操作**进行优化：

```cypher
-- 传统逐条插入（慢）
CREATE (m:Movie {mid: '1292052', title: '肖申克的救赎', ...})

-- UNWIND 批量插入（快，每批 2000 条）
UNWIND $movies AS m
MERGE (movie:Movie {mid: m.mid})
SET movie.title = m.title,
    movie.rating = m.rating,
    ...
```

`UNWIND` 将一个 JSON 数组展开为行，在一次事务中处理多条记录，大幅减少了网络往返和事务开销。

### 7.3 导入性能实测

| 阶段           | 数据量        | 耗时       | 吞吐量            |
| -------------- | ------------- | ---------- | ----------------- |
| Movie 节点     | 135,990       | ~17 秒     | ~7,958 条/秒      |
| Person 节点    | 198,067       | ~14 秒     | ~13,590 条/秒     |
| DIRECTED 关系  | 105,757       | ~4 秒      | ~26,007 条/秒     |
| ACTED_IN 关系  | 839,923       | ~34 秒     | ~3,528 部电影/秒  |
| HAS_GENRE 关系 | 253,252       | ~8 秒      | ~14,656 部电影/秒 |
| **总计**       | **1,533,021** | **~77 秒** | —                 |

整个 ETL 过程在约 **77 秒** 内完成，总导入量超过 153 万条数据。
